🌸 Bloom Filters

A probabilistic set that answers "have I seen x?" in O(k) with tiny memory. It can say "definitely not" or "possibly yes" — never a false negative, but sometimes a false positive.
probabilistic membership m bits · k hashes no false negatives O(k) add / query

A Bloom filter is a bit array of size m (all 0) plus k independent hash functions. Add(x) hashes x with all k functions and sets those k bits to 1. Query(x) checks the same k bits: if any is 0, x was definitely never added; if all are 1, x is possibly in the set (those bits could have been set by other elements — a false positive).

The gotcha: a Bloom filter has no false negatives, only false positives — flip that in an interview and it's an instant tell. You cannot delete (clearing a bit could break other members → use a counting Bloom filter) and cannot enumerate members. Rule of thumb: ~10 bits/element ≈ 1% false-positive rate. Cousins: Count-Min Sketch (frequencies) and HyperLogLog (cardinality). Systems use: a Bloom filter sits in front of each SSTable in Cassandra / RocksDB / HBase / Bigtable to skip disk reads for keys that are definitely absent.

Add(x)O(k) — set k bits
Query(x)O(k) — check k bits
SpaceO(m) bits — independent of element size
FP rate≈ (1 − e^(−kn/m))^k for n inserted

What you're seeing

  1. Add(x): k hash functions map x to k bit positions; each is set to 1 (already-1 bits just stay 1).
  2. Query(x): check the same k positions. Any 0 → definitely NOT in the set.
  3. All k positions 1 → possibly in the set: x may have been added, or its bits were set by other items.
  4. As you add more items the fill ratio climbs and the estimated false-positive rate rises.
  5. The demo finds an unseen item whose k bits were all set by others → a true false positive.
