🦓 ZooKeeper — znodes, watches & leader election

A hierarchical namespace of znodes coordinates a distributed system. Persistent nodes hold config; ephemeral nodes vanish when their client's session dies — giving you automatic failure detection and the canonical leader-election pattern.
distributed coordination hierarchical znodes ephemeral = session-bound one-shot watches leader election

ZooKeeper exposes a filesystem-like tree of znodes. Persistent znodes (e.g. /config) survive until explicitly deleted and can hold data. Ephemeral znodes are tied to a client session and disappear the instant that client dies — that is the failure-detection primitive. Clients place one-shot watches on a znode to be notified once when its data changes or it is deleted.

Strong consistency: a ZooKeeper ensemble keeps every replica in agreement through the ZAB (ZooKeeper Atomic Broadcast) consensus protocol — all writes go through a single leader and are totally ordered, so every client sees the same tree.
The gotcha: watches are one-shot. After a watch fires it is consumed; if you still care about that znode you must re-register the watch — and a change happening in the gap between fire and re-register can be missed.

Run a command

Create persistent znodes for config, set their data to fire watches, or have clients Join election to create ephemeral-sequential nodes. Kill the leader to watch a re-election ripple through.

Event log

What you're seeing

  1. Create a znode under the tree — persistent (holds config) or ephemeral (owned by a client session).
  2. Set data on a watched znode fires its one-shot watch → the watching client is notified, then must re-register.
  3. Join election creates an ephemeral sequential znode /election/n_…; the lowest sequence is the leader.
  4. Each follower watches the next-lower znode (not the leader) to avoid a herd of wake-ups.
  5. Kill a client → its ephemeral znodes vanish. If the leader dies, the watcher of its znode wakes and the next-lowest becomes leader.

📖 Reference — infrastructure, sharding, API & recipes

Infrastructure & deployment

  • Ensemble of 3 or 5 nodes (odd count so a strict majority can form a quorum; 5 nodes tolerate 2 failures).
  • ZAB (ZooKeeper Atomic Broadcast) consensus — one leader + followers; every write is proposed by the leader and totally ordered.
  • Writes go through the leader and need a quorum ack → strongly consistent, but not high-write-throughput (every write fsyncs + round-trips a majority).
  • Clients hold a session kept alive by heartbeats; lose the heartbeat and the session expires.
  • Ephemeral znodes vanish the instant their session dies (the failure detector); watches are one-shot notifications — re-register after each fire.

Sharding & scaling

  • It is a coordination service, not a data store — do not shard application data into it. Store pointers/metadata, not your dataset.
  • Reads scale with followers (any replica serves them), but a read may be slightly stale — issue sync first if you need the latest.
  • All writes serialize through the leader, so keep write volume low; the ensemble does not scale writes by adding nodes.
  • Data per znode is small (~1MB cap) — it's for config/locks/flags, not blobs.
  • Modern alternative: etcd (Raft consensus) — same role, common in Kubernetes.

Use-case recipes

👑 Leader election
# each candidate creates an ephemeral SEQUENTIAL znode
create -e -s /election/n_        # → /election/n_0000000007
# lowest sequence number == leader; others watch the NEXT-LOWER znode
ls /election                     # sort children; am I the smallest?
get -w /election/n_0000000006    # follower watches only its predecessor
When the leader's session dies its ephemeral znode vanishes, waking just the one follower watching it (no thundering herd) → that follower is now lowest → new leader.
🔒 Distributed lock
# same ephemeral-sequential queue under a lock node
create -e -s /lock/req_          # take a ticket
ls /lock                         # you HOLD the lock iff you are lowest
get -w /lock/req_<prev>          # else watch the request just ahead of you
Ephemeral means a crashed holder auto-releases (no stuck locks). Watching only the predecessor turns it into a fair FIFO queue.
⚙️ Config management
set /config/db "host=primary"   # operator updates config
get -w /config/db                # every client watches it
# on the watch firing → re-read and hot-reload, then re-register the watch
One source of truth; clients learn of changes via the one-shot watch and reload without a restart or polling.
🧭 Service discovery
# each instance registers itself as an ephemeral znode
create -e /services/payments/inst_a "10.0.0.7:8080"
ls /services/payments            # consumers list live instances
# a dead instance's session expires → its znode auto-deregisters
No stale endpoints: the registry self-cleans because membership is tied to live sessions.
👥 Group membership / barriers
# each member announces presence with an ephemeral znode
create -e /group/members/node_3
ls -w /group/members             # watch the set; reacts as members join/leave
# barrier: proceed only once child count reaches N
Live membership comes for free from ephemeral znodes; barriers gate a group until everyone has arrived (or departed).
