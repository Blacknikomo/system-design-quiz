🌍 CDN — the cache that lives next door

PoPs, cache keys, purges and the brutal arithmetic of hit ratios — how a fleet of reverse proxies makes a Virginia origin feel local in Sydney. ☕ ~15 min read
reverse-proxy cache anycast / DNS steering cache key TTL & purge hit-ratio math edge compute

A CDN is a fleet of reverse-proxy caches parked in hundreds of cities. Users are routed to the nearest one; it serves what it has, fetches what it doesn't, and keeps a copy for the next person. That's the whole trick. The craft is in the cache key, the TTLs and the invalidation story — and the payoff is measured by one number, the hit ratio, whose economics are brutally non-linear.

In the interview: a CDN is your first move whenever reads are global and the same bytes can be served to many users — static assets, images, video, public API GETs. The interviewer then expects you to handle, unprompted: what exactly the cache key is, how invalidation works (hint: you design so you almost never invalidate), what a hit-ratio dip does to the origin, and how private content stays private on a shared cache.

1 · The mental model: corner shops for a Virginia warehouse

Your origin is a warehouse in Virginia. A customer in Sydney is ~200 ms away as the packet flies — and a fresh HTTPS fetch isn't one round trip, it's several: TCP handshake, TLS handshake, then the request itself. Stack three or four 200 ms round trips and the user has stared at a blank screen for the best part of a second before the first byte of your 2 MB of JavaScript even starts moving. No amount of server tuning fixes the speed of light.

The CDN answer is the corner shop. Instead of shipping every carton from Virginia, you park small shops — PoPs (points of presence, also "edge locations") — in hundreds of cities, each one a rack of reverse proxies with fast disks and fat peering links. The shop stocks whatever locals have asked for recently. First customer for an item waits for the warehouse run; everyone after gets it across the counter in ~20 ms. Nobody in Sydney talks to Virginia any more; they talk to a machine a few suburbs away that pretends to be you.

Two mechanisms get the user to the nearest shop. Anycast: every PoP announces the same IP address, and BGP — the internet's routing fabric — naturally delivers each packet to the topologically closest announcer (Cloudflare's model). Or DNS-based steering: the CDN's DNS server looks at where the query comes from and hands back the address of a nearby PoP (the Akamai/CloudFront lineage). Either way, the crucial consequence is that TLS terminates at the edge: the handshake round trips happen over a 15 ms path instead of a 250 ms one, and the edge keeps warm, long-lived, congestion-tuned connections back to your origin for whatever it can't serve itself.

On a miss, the edge behaves like any reverse proxy: it forwards the request to the origin (often via a mid-tier — more in §4), stores the response according to its headers, and serves it. This fetch-on-demand style is a pull CDN, and it is how almost everything works today. Push (pre-warming) still exists for special occasions — game-patch day, a product launch video — when you'd rather pre-place the bytes than let ten million first-customers pile up. But the default mental model is: the CDN stocks itself, driven by demand, bounded by TTLs.

And the number everyone stares at is the hit ratio. The origin only ever sees misses, so origin load is proportional to (1 − hit ratio) — which makes small dips enormous. Going from 98% to 97% hits sounds like a rounding error; it means misses went from 2% to 3%, i.e. your origin traffic just grew 50%. This one piece of arithmetic explains most CDN engineering, most CDN incidents, and is the centerpiece of the simulator in §5.

CDN vs "a cache" — the one-sentence test. Redis caches computed data inside your datacenter, for your servers. A CDN caches whole HTTP responses next to your users. The litmus test for the CDN layer: could this exact URL, byte for byte, be served to two different users? If yes, it belongs on the CDN. If every response is personal, the CDN can still terminate TLS and carry the bytes — but it's a fast pipe, not a cache.

2 · Syntax & core API — the API is HTTP itself

There's no query language here: a CDN is programmed with HTTP response headers (what the origin says) plus a slab of CDN configuration (what the edge is allowed to override). Fluency in these headers is the hands-on signal in an interview — they are the knobs.

Cache-Control: public | privateWho may store this. public = shared caches OK; private = browser only — the CDN must pass it through. Forgetting private on a per-user page is the classic data-leak incident.
max-age=60Freshness lifetime in seconds, for all caches including the browser.
s-maxage=300Freshness for shared caches only (the CDN); overrides max-age there. Idiom: short max-age for browsers (you can't purge browsers), long s-maxage for the edge (you can purge that).
no-cache / no-storeTwo very different things: no-cache = store, but revalidate before every use; no-store = never write to disk at all (auth pages, PII).
immutable"This URL's content will never change" — don't even revalidate on reload. Only safe with content-hashed filenames (§ deep dive).
stale-while-revalidate=60May serve a stale copy for up to 60 s past expiry while refreshing in the background — users never wait on a revalidation.
stale-if-error=600If the origin is down or 5xx-ing, keep serving the stale copy for up to 10 min. Free resilience — set it everywhere.
ETag / If-None-MatchRevalidation: origin returns a version tag; on expiry the edge asks "still "9f3a2c"?" and gets a body-less 304 Not Modified instead of a re-download.
Vary: Accept-EncodingExtends the cache key by a request header. Accept-Encoding is fine (one copy per compression). Vary: Cookie is death — cookies are near-unique per user, so hit ratio → ~0 (§ deep dive).
Age: 41Added by the cache: seconds since this copy was fetched. With X-Cache: Hit / CF-Cache-Status, your main debugging tool — "is the edge actually serving hits?"
Surrogate-Key: prod-42 htmlTag responses with keys (Fastly; Cloudflare calls them cache tags); later purge by tag in one API call instead of enumerating URLs.
signed URL / signed cookiePrivate content on a public CDN: the URL carries an expiry + signature; the edge verifies it cryptographically without calling your origin.
# anatomy of a cacheable response — origin proposes, edge disposes
HTTP/1.1 200 OK
Cache-Control: public, s-maxage=300, max-age=60, stale-while-revalidate=60, stale-if-error=600
ETag: "9f3a2c"
Vary: Accept-Encoding            # key extension: one copy per encoding — harmless
Age: 41                          # edge adds this: cached 41 s ago

# five minutes later the edge revalidates instead of re-downloading:
GET /article/42 HTTP/1.1
If-None-Match: "9f3a2c"
  → HTTP/1.1 304 Not Modified    # headers only — no body crosses the ocean

The second thing to know cold is the two-tier asset strategy — it dissolves 90% of invalidation questions before they're asked:

# tier 1 — content-hashed assets: cache FOREVER, never purge
GET /static/app.3f9c1b.js
Cache-Control: public, max-age=31536000, immutable

# tier 2 — the small HTML that points at them: short TTL + revalidation
GET /index.html
Cache-Control: public, s-maxage=60, stale-while-revalidate=300
ETag: "release-2041"

# deploy = upload NEW hashed assets first, then flip the HTML.
# if you must purge at all, you purge only the HTML — by tag:
curl -X POST https://api.cdn.example/purge -d '{"surrogate_keys":["html"]}'

New release → app.7d21e0.js is a different URL, so there is nothing stale to purge; old HTML keeps working against old assets until its 60-second TTL rolls over. The asset that never changes can be cached for a year; the pointer that changes is cheap to refresh. This pattern is why serious shops treat "we purge assets on every deploy" as a smell.

3 · Use cases — when the CDN is the answer

🧱 Static assets with content-hashed names
The default, non-negotiable case: JS bundles, CSS, fonts, product images. Hashed filenames + max-age=1 year, immutable → hit ratios in the high-90s, zero purge traffic, deploys that can't serve half-old half-new bundles.
Why a CDN: these bytes are identical for every user on Earth — the definition of cacheable. Your origin becomes a factory that only ever builds each file once.
🎞 Video & large media via ranged requests
HLS/DASH streaming chops video into small segment files (a few seconds each) plus a tiny playlist — segments are immutable and perfectly cacheable, so a viral video is served almost entirely from the edge. Big downloads use Range: requests; CDNs cache in chunks so a million people resuming a 4 GB game patch don't re-pull it from origin.
Why a CDN: video is ~most of internet traffic by volume; egress from the edge is cheaper and closer. Live streaming works too — segments are cacheable the moment they're written, with 2–6 s TTLs.
⚡ API acceleration — even for uncacheable traffic
Point api.example.com at the CDN with caching mostly off. Users get edge TLS termination (handshakes over ~20 ms, not ~200), HTTP/3 at the edge even if your origin speaks HTTP/1.1, and warm persistent connections over the provider's tuned backbone to your origin. Then selectively cache the safe GETs — a product catalog with s-maxage=30 — for extra credit.
Why a CDN: global p95 drops materially without touching a line of backend code; DDoS and TLS churn are absorbed at the edge.
🛍 Flash-sale / launch-day front door
Ticket drop, sneaker release, exam-results page: 99% of the traffic is the same product page, requested a million times a minute. Cache it at the edge with s-maxage=5, stale-while-revalidate=30: the origin renders the page once every few seconds per shield, and the CDN absorbs everything else. Pair with an edge-served waiting room for the write path.
Why a CDN: this converts "we melted at launch" into "the origin served a few QPS while the edge did the screaming."

When a CDN is the WRONG answer

personalized everythingIf ~every response is per-user (dashboards, feeds with server-rendered personalization), there's nothing shareable to cache. The CDN still helps as a TLS/transport layer, but don't promise hit ratios — or restructure: cache the shell, fetch the personal bits as an API call.
strong read-after-write"User updates price, next GET must show it" on the cached path = fighting your own TTLs with purges. Either bypass the CDN for those reads or accept bounded staleness explicitly.
single-region, internal usersAn internal tool used by 200 employees in one office gains nothing from 300 PoPs — a local cache/proxy (or nothing) is simpler and cheaper.
as a system of recordA CDN is a cache: eviction is a feature, not an incident. Anything that must exist durably lives in object storage; the CDN sits in front of it, never instead of it.

4 · Distributed setup — PoPs, shields and multi-CDN

A real deployment is a cache hierarchy: hundreds of edge PoPs → a handful of regional shields (mid-tier caches; CloudFront calls the concept regional edge caches / Origin Shield) → your origin. The shield exists to collapse fan-in. Without it, a popular object expiring means up to one miss per PoP — hundreds of near-simultaneous requests hammering the origin for the same bytes. With a shield, edges miss to the shield, the shield misses once to the origin, and your origin's peak concurrency stops scaling with the provider's PoP count. In capacity terms: origin load ≈ one fetch per object per TTL per shield, not per PoP.

What happens when a PoP dies — or a new one comes online cold? Traffic shifts (BGP reconverges under anycast; DNS steering updates in seconds-to-minutes) to a neighboring PoP, which has its own cache — warm for the hot head of the catalog, cold for the long tail. The result is a hit-ratio dip and an origin bump, not an outage. The design consequence: your origin must survive the worst-case regional miss storm, or you must let the CDN shed load for you (serve-stale, request collapsing, rate limits at the edge). "Origin sized for steady state only" is a latent incident.

Multi-CDN is the grown-up resilience posture: run two or more providers, with a DNS-level traffic steerer choosing per-user based on real-user-measurement data (latency, availability), plus commercial leverage as a bonus. Big streamers and game launchers live like this. The costs are real: you can only rely on lowest-common-denominator features, cache keys/configs/purges must be maintained twice, TLS certs live in two places, and your effective global hit ratio drops because each provider warms its own copy.

Finally, remember what the CDN also is: an availability layer. stale-if-error turns an origin outage into "users see slightly old content"; an edge-hosted static fallback page turns it into a branded apology instead of a connection reset. Some of the best incident stories end with "…and the CDN quietly served stale for 40 minutes while we fixed it."

Gotchas that bite in production: (1) CDN config is a global deploy that lands in seconds — a bad cache-key or header rule is instantly worldwide; the industry's most famous CDN outages were config pushes tripping latent bugs (Fastly's 2021 global outage is the canonical example), so treat CDN config as code: versioned, reviewed, canaried where the provider allows. (2) Purges are eventually consistent — seconds to minutes on most providers; a client can re-fetch during propagation and re-cache stale from a not-yet-purged shield. (3) There is no read-after-write on a cached path, full stop — design the write path around the cache, not through it.

🔍 Deep dive · The cache key — where hit ratios are made and destroyed click to expand

Every cache lookup answers one question: have I seen this key before? The default key is roughly host + path + query string — and every element you add to it fragments the cache into more distinct copies. Cache-key design is therefore a knife's edge: too broad a key and you serve the wrong content to the wrong person; too narrow (too many dimensions) and your hit ratio quietly dies of a thousand variants.

The classic slow poison is the query string. Marketing links arrive as /product/42?utm_source=nl&utm_campaign=july — same page, thousands of "different" URLs, each cached separately, each a miss. The fix is key normalization at the edge: whitelist the query parameters that actually change the response (?size=, ?page=), drop or sort everything else. Same story for device buckets: don't key on the full User-Agent (thousands of strings — a fragmentation bomb); have the edge classify into mobile|desktop and key on that one bit.

Vary is the origin reaching into the key, and it deserves fear. Vary: Accept-Encoding is benign — two or three variants. Vary: Cookie means "one cache copy per unique Cookie header", and cookies carry session IDs — so effectively every user gets a private copy, hit ratio ≈ 0, and your CDN degrades into an expensive TCP terminator. Many CDNs simply refuse to cache such responses. If personalization is one widget on the page, don't vary the page: cache the shared shell and fetch the personal fragment client-side (or assemble at the edge — see the edge-compute dive).

The opposite failure is scarier: an under-specified key serves user A's content to user B. The incident archetype (it has happened to household-name sites): a config change makes the CDN ignore cookies while the origin still renders per-user pages, or an origin suddenly emits Set-Cookie (or a session-dependent body) on a cacheable response — and the edge dutifully serves one user's account page, or their session cookie, to the next thousand visitors. Defenses: never cache responses bearing Set-Cookie (most CDNs do this by default — do not disable it), mark private endpoints Cache-Control: private/no-store at the origin, and treat "make X cacheable" changes as security reviews, not perf tweaks.

🔍 Deep dive · Invalidation — purge APIs vs the immutable pattern click to expand

Two hard problems in computer science, and cache invalidation is both of them once your cache has 300 locations. Every CDN offers a purge API: by exact URL, by prefix/wildcard, or — the scalable form — by surrogate key / cache tag: the origin stamps responses with tags (Surrogate-Key: product-42 catalog), and one API call purges every URL carrying the tag, whether that's three objects or three million. Purge-by-tag is the difference between "invalidate the product page, its API JSON and its OG image" being one call or a scavenger hunt.

Speed varies wildly and is worth knowing: Fastly made near-instant purge its brand — ~150 ms globally is the advertised figure — because its market (news front pages, live sports, retail inventory) treats freshness as a product feature. Most other CDNs propagate purges in seconds to minutes; CloudFront invalidations are the slow, billed end (first ~1,000 paths/month free, then per-path fees — and minutes of propagation). Design rule: if your correctness depends on sub-second purge, you've either picked Fastly deliberately or you've designed a fragile system.

Which is the real lesson: the robust pattern is to make purging unnecessary. Content-hashed, immutable filenames (app.3f9c1b.js, cached for a year) mean assets never need invalidation — a new deploy mints new URLs. Only the thin HTML layer that references them stays short-TTL'd (or no-cache + ETag, so every request is a cheap 304 revalidation). Deploy choreography matters: upload new assets first, then flip HTML, and keep the previous release's assets around — a user holding 60-second-old HTML must still find app.OLD.js, or you've built a self-inflicted outage window into every deploy.

Purges also fail upward: a "purge everything" against a busy property converts your entire request rate into misses within seconds — a self-inflicted DDoS on the origin (the simulator's 🧹 button is exactly this). Mature setups restrict wildcard purges, rely on tags scoped to what actually changed, and lean on stale-while-revalidate so even a purge-then-refill cycle never makes users wait.

🔍 Deep dive · Request coalescing, stampedes & negative caching click to expand

A cache's worst moment is the synchronized miss. A hot object expires (or gets purged) and ten thousand concurrent requests arrive at edges that all say "not in cache". Naively, all ten thousand go to origin — the thundering herd / cache stampede. The origin, sized for a 2% miss rate, meets 100× its normal load for one URL and folds; its slowness makes more requests pile up; the herd grows. Classic metastable failure.

The fix is request coalescing (a.k.a. collapsed forwarding): the first miss for a key acquires a per-key lock and goes to origin; every concurrent request for the same key waits for that one fetch and is answered from it. Ten thousand requests, one origin fetch, per PoP. Layer the shield on top (all PoPs miss to it; it coalesces again) and a global celebrity-post spike costs your origin roughly one fetch per TTL. Every serious cache implements this — Varnish/Fastly natively, Nginx via proxy_cache_lock — and the §5 toggle shows the effect precisely.

Coalescing handles the burst; stale-while-revalidate removes the cliff entirely — expiry stops being an event because one background fetch refreshes while users keep getting the stale-but-fine copy. For self-inflicted synchronization (thousands of objects cached at deploy time, all expiring together), add TTL jitter — ±10% noise so expirations smear out.

Negative caching is the sharp knife in the drawer: caching 404s/5xxs briefly (CloudFront defaults to ~10 s for error responses) protects the origin from a herd of requests for something that doesn't exist — imagine a bad HTML deploy referencing app.WRONG.js, and every visitor's browser hammering that 404 through to origin. But the same mechanism masks recovery: cache a 503 for 5 minutes and you've extended every blip by 5 minutes, globally. Keep negative TTLs in seconds, never minutes, and never let an error response inherit the success TTL.

🔍 Deep dive · Beyond static — signed URLs & compute at the edge click to expand

Private content on a public cache sounds like a contradiction; signed URLs resolve it. The origin (which knows who's allowed) mints a URL carrying an expiry and a cryptographic signature (CloudFront signed URLs, GCS/S3 presigned variants); the edge validates the signature and expiry locally — no call to your origin — then serves the cached bytes. The object is cached once; authorization travels in the URL. Signed cookies do the same for whole sessions ("this user may fetch any segment of this video for 6 hours" — one cookie, thousands of segment URLs). Keep expiries short (minutes), scope them narrowly, and lock the origin bucket so the CDN is the only door (CloudFront's Origin Access Control): otherwise the "private" file is public at its S3 URL.

Edge compute moved the CDN from "cache" to "programmable first hop". Cloudflare Workers run V8 isolates in every PoP with ~ms startup; CloudFront pairs lightweight CloudFront Functions (header-level tweaks) with Lambda@Edge (heavier, runs at regional caches); Fastly runs WebAssembly. The sweet-spot uses are request-shaping before the cache: auth (verify a JWT at the edge, reject junk without waking the origin), A/B bucketing (assign a cohort, key the cache on it — two variants, not per-user copies), geo/compliance routing, cache-key normalization beyond what config allows, and assembling a cached shell with small dynamic fragments — the modern descendant of edge-side includes.

The honest caveats: edge functions run in hundreds of locations with no shared state (state = a global KV/durable-object service with its own consistency lattice), debugging is distributed-systems debugging, and CPU/memory budgets are tight. The interview-safe framing: "logic that shapes requests and caching belongs at the edge; logic that owns data doesn't."

⚖️ At scale — numbers, limits & failure modes

Order-of-magnitude figures for capacity talk — round aggressively out loud.

footprintMajor CDNs run PoPs in hundreds of cities (Cloudflare advertises 300+; CloudFront many hundreds of edge locations behind a dozen-ish regional caches; Akamai the largest/oldest footprint). Consequence: most users sit < ~30–50 ms from an edge, vs 100–300 ms cross-continent.
latency winA cold HTTPS fetch is ~3 round trips before the first body byte (TCP + TLS 1.3 + request). Over a 200 ms path ≈ ~600 ms; over a 20 ms edge path ≈ ~60 ms — and a hit never leaves the city.
hit-ratio economicsOrigin load = miss rate. 98% → 97% hits = 2% → 3% misses = +50% origin traffic; 99% → 98% doubles it. Alarm on miss rate, not hit rate — the same delta reads as "1 point" on one graph and "+50%" on the other.
hit-ratio ceilingEdge storage is finite and LRU-ish: the long tail re-evicts before re-request. Static assets hit 95–99%+; long-tail media plateaus lower no matter the TTL. Raising TTLs can't beat eviction — only a bigger/warmer tier (shield) can.
capacityBig providers advertise aggregate edge capacity in the hundreds of Tbps — absorbing volumetric DDoS at the edge is a core product. Egress list price is roughly $0.02–0.09/GB at entry tiers (CloudFront ~$0.085/GB NA/EU first tier; budget CDNs ~a cent or two), falling steeply with volume; origin→CDN fetch is often free in-provider (S3 → CloudFront).
purge speedFastly: ~150 ms global (advertised). Most others: seconds–minutes. CloudFront: minutes + per-path billing past the free ~1,000 paths/month. Tag/surrogate-key purge scales; wildcard purge storms don't.
defaults worth knowingOrigin sends no caching headers → CDN applies a default TTL (CloudFront: 24 h) — surprise staleness. Error responses get short negative TTLs (~10 s CloudFront default). Multi-GB objects are fine — fetched and cached in ranged chunks.
Failure modes that only appear at scale: purge storm (mass invalidation → your full request rate becomes misses → self-DDoS of the origin); cache stampede (hot-key expiry without coalescing/SWR → thundering herd); private-data caching (a key/header misconfig caches Set-Cookie or per-user pages — the classic "logged in as someone else" incident); Vary explosion (Vary: User-Agent/Cookie shreds the cache into per-client copies — hit ratio collapses silently); negative-cache masking recovery (a cached 503 keeps "failing" minutes after the origin healed); cold-PoP miss storm (failover/new region → tail re-warms through your origin — size it, or serve stale).

8 · In the interview

How it comes up: rarely as "design a CDN" (that's an infra-team special); constantly as a component. Any design with global users and read-heavy content — social feed images, video platform, news site, static hosting, "make p95 < 200 ms worldwide" — should have you drawing the CDN box in the first five minutes. The predictable follow-ups: what exactly do you cache and on what key? · how does an update reach users? (TTL vs purge vs hashed URLs) · what happens to your origin if the CDN goes cold? · how do private files stay private? (signed URLs) · what's your hit ratio and what does a 1-point dip mean? (+50% at 98→97 — say it before they ask).

The choice script: in an AWS design, default to CloudFront ("integrated with S3/ALB, origin access control, Lambda@Edge if I need edge logic"). Reach for Cloudflare when you want the whole front door — anycast DNS, WAF, DDoS, Workers — as one layer. Name Fastly when freshness is a product feature (news, inventory, sports scores) because of instant purge. Mention multi-CDN + steering when availability requirements are extreme (major streamers). And say the quiet part: the pattern (immutable assets, tagged purge, shielded origin) transfers across vendors — the vendor is a pricing decision.

9 · Check yourself

Your dashboard shows hit ratio slipping 99% → 98%. Why is ops being paged over "one percent"?

Because the origin doesn't see hit ratio — it sees misses, and misses went from 1% to 2% of total traffic: origin load doubled. The general rule: origin traffic ∝ (1 − hit ratio), so deltas near the top of the scale are enormous in relative terms (98→97% = +50%). This is also why capacity planning for the origin starts from "worst credible miss rate" (cold PoP, purge storm), not from yesterday's average.

An engineer adds Vary: Cookie to fix "users seeing each other's pages". What actually happens, and what's the right fix?

Cookies are near-unique (session IDs), so the cache key now includes an effectively per-user value: every user gets a private copy, hit ratio collapses toward zero, and most CDNs will simply stop caching the response. The symptom is "fixed" by disabling the cache. Right fixes: mark truly per-user responses Cache-Control: private/no-store; never cache anything bearing Set-Cookie; restructure the page — cache the shared shell, fetch personal fragments via an authenticated API call or assemble them in edge compute keyed on a low-cardinality cohort, not the raw cookie.

How do you serve private user photos through a CDN without making them public?

Signed URLs: the app (which knows the ACL) mints a URL with an expiry and signature; the edge verifies both cryptographically without contacting the origin, then serves the cached object. For session-wide access (video segments), signed cookies authorize a whole path prefix. Two details that separate senior answers: keep expiries short (minutes) and scoped narrowly, and lock the origin bucket so only the CDN can read it (CloudFront OAC) — otherwise the object is still public at its storage URL.

You deploy a new JS bundle. Walk through why you don't purge — and what you do instead.

Assets are content-hashed: the new bundle is app.7d21e0.js — a brand-new URL with no stale copies anywhere, cached max-age=1y, immutable. The HTML that references it carries a short s-maxage (or no-cache + ETag), so it rolls over within seconds-to-minutes — purge the HTML tag if you're impatient. Choreography: upload new assets first, keep the old release's assets available, then flip HTML — a user holding old HTML must still resolve old assets. Result: zero purge dependencies, no window of broken half-deployed pages.

A celebrity posts a link and one URL goes from 10 to 500k requests/min. What protects your origin, layer by layer?

(1) The object is cached: 99.9%+ served at the edges. (2) On expiry, request coalescing collapses each PoP's concurrent misses into one origin-bound fetch. (3) The origin shield collapses the PoPs' fetches again — the origin sees roughly one request per TTL. (4) stale-while-revalidate removes even that synchronized wait by refreshing in the background. Without (2)–(4), a single expiry moment becomes a thundering herd — the difference between "one fetch per TTL" and "ten thousand concurrent misses".
