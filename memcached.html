🧊 Memcached — the cache that does one thing

A multithreaded hash table in RAM — and the client-side engineering (ketama, leases, gutter pools) that turned it into planetary infrastructure at Facebook. ☕ ~15 min read
look-aside cache client-side sharding slab allocator leases ≤1 MB values · 250 B keys

Memcached is an in-memory hash table with LRU eviction behind a TCP port — get, set, delete, and deliberately nothing else. No persistence, no replication, no data structures, no cluster protocol: every hard problem is pushed to the client. That refusal to do more is the feature — a node stays predictable at sub-millisecond latency, scales with cores, and never surprises you — and it's why Meta still runs oceans of it while the rest of the world defaulted to Redis.

In the interview: Memcached appears the moment you say "add a cache in front of the database". The interviewer then expects, unprompted: the cache-aside pattern, invalidation by delete (not set), the thundering-herd problem and its fix (leases / single-flight), what a dead node means (a miss storm, not data loss), and a crisp Memcached-vs-Redis call.

1 · The mental model: a coat check, not a vault

Picture a coat check at a theater. You hand over a coat, get a number, and retrieval is instant. But the contract is thin: the attendant may quietly discard the least recently used coats when the racks fill, and if the theater loses power overnight, the racks are simply empty in the morning. Nobody is upset — everyone's real coat (the database row) still exists at home. A cache that loses everything and inconveniences no one: that is Memcached's entire ambition.

It was born in 2003 at LiveJournal (Brad Fitzpatrick), and its design philosophy has not moved since: the server is dumb; the client is smart. A Memcached server knows about its own RAM and its own connections — full stop. It doesn't know other servers exist, doesn't replicate, doesn't persist, doesn't fail over. Sharding? The client does it. Redundancy? The client (or your architecture) does it. This is the exact opposite of Redis, which packs data structures, persistence, replication and scripting into the server — the Swiss-army knife versus the scalpel.

What the server does do, it does exceptionally well. It's multithreaded — one process saturates a modern multi-core box, where a single Redis instance is fundamentally one command-processing thread. Memory is managed by a slab allocator that never fragments itself to death under churn (deep dive 1). And the operational surface is so small that fleets of thousands of nodes are boring to run — which, at Meta's scale, is the killer feature.

One consequence to internalize: because eviction, restarts, and node deaths are all "normal", every get may miss. Correct Memcached code treats the cache as an optimization layered over a source of truth, never as the source itself. The moment you catch yourself wanting the data back after a restart, you have left Memcached's use case.

The one-sentence test. If you would be sad that the data vanished — sessions you can't recreate, a leaderboard, a queue — it does not belong in Memcached. If a vanished entry merely costs you one database read, you've found its habitat.

2 · Syntax & core API

The whole protocol fits in your head, and the text version can literally be typed over telnet — worth doing once in your life. The vocabulary below is what you name in an interview.

get k / get k1 k2 k3Fetch one key — or many in one round trip (multiget). Batching is not a nicety: a page render can need hundreds of keys, and round trips are the cost that matters (deep dive 4).
set k flags ttl bytesUnconditional store. TTL in seconds — with the classic footgun: a value ≤ 30 days is relative, anything larger is parsed as an absolute unix timestamp. 0 = no expiry (but still evictable — this is a cache).
add / replaceStore only-if-absent / only-if-present. add doubles as a crude best-effort mutex (add lock:job42) — fine for dedup hints, not for real locks (no fencing, node = SPOF).
delete kThe invalidation primitive. After a DB write, you delete the key rather than set it — see deep dive 3 for the race that rule prevents.
incr / decrAtomic unsigned-64 arithmetic on an existing numeric value. Missing key → NOT_FOUND, not zero — seed with add first.
gets → casOptimistic concurrency: gets returns a CAS token; cas writes only if nobody changed the value since. The building block for read-modify-write without a lock.
touch k ttlRefresh a TTL without resending the value.
stats / stats slabsHit rate, evictions, and per-slab-class occupancy — where you diagnose calcification (deep dive 1).
limitsValue ≤ 1 MB by default (-I raises it; think twice), key ≤ 250 bytes. Item metadata overhead is on the order of ~50 B — tiny values are overhead-dominated.

3 · Use cases — when Memcached is the answer

🗄 Database query-result cache
The canonical deployment: get user:42 → miss → SELECT → set with a TTL. Hot rows, expensive aggregates, rendered JSON responses. A ~95%+ hit rate turns a drowning database into a bored one — the cache absorbs the read multiple, the DB keeps the truth.
Why Memcached: values are opaque blobs, loss is harmless, and raw ops/node × operational simplicity is the whole requirement.
🧩 Rendered-fragment cache
The original LiveJournal use: cache computed HTML fragments or serialized view models keyed by fragment:profile:42:v3. Skips not just the DB but the templating/serialization CPU. Version the key (:v3) and "invalidation" on deploy becomes letting the old keys age out.
Why Memcached: big-ish opaque strings, high churn, zero need for structure — and key versioning sidesteps invalidation entirely.
🎫 Ephemeral session-ish data
Login sessions, CSRF tokens, per-user flags — if a lost entry only means a re-login. State the caveat in the interview: a node death logs out ~1/N of users at once. If that's unacceptable (paid checkout flows), sessions belong in Redis with replication, or in the DB with a cache in front.
Why Memcached (sometimes): sessions are small, hot, and TTL-friendly; the loss-tolerance question is the design decision.
🚦 Best-effort rate-limit counters
incr ratelimit:ip:1.2.3.4 with a 60 s TTL is a serviceable fixed-window limiter at the edge. Caveats you must say: counters vanish on restart (fail-open), windows are fixed not sliding, and the counter lives on one node — fine for abuse damping, wrong for billing-grade quotas.
Why Memcached: incr is atomic per node and costs microseconds; "approximately enforced" is often exactly the requirement.

When Memcached is the WRONG answer

data you must get backAnything that must survive a restart — carts, queues, counters of record. No persistence, no replication. Use Redis (AOF/replicas) or a database.
pub/sub, queuesThere is no push, no blocking pop, no streams. Redis/Kafka/SQS territory.
sorted sets / leaderboardsValues are opaque bytes; there are no data structures to update server-side. This is the single most common "actually, Redis" trigger.
distributed locksadd has no fencing tokens, no consensus, and the node is a SPOF. Locks want ZooKeeper/etcd (or Redis with eyes open).
big blobs>1 MB values need config surgery and wreck slab efficiency; put blobs in S3/object storage and cache the pointer.

4 · Distributed setup — the cluster that lives in the client

Here is Memcached's signature move: distribution is 100% client-side. A "Memcached cluster" is just N independent servers that have never heard of each other, plus a client library holding the server list. The client consistent-hashes each key onto the ring (the classic implementation is ketama) and talks straight to the owning node. Add or remove a server and only ~1/N of keys remap — the mechanics are exactly the consistent-hashing story on the scaling page, running inside your application process.

What you give up is failover — deliberately. A dead node isn't "promoted around"; its share of keys simply starts missing, and the misses fall through to the database. A dead cache node is a miss storm, not data loss. Whether that's a shrug or an outage depends on whether your DB can absorb 1/N of cache traffic — which is why Facebook built the gutter (deep dive 2, and the widget below). At fleet scale the "smart client" often migrates into a sidecar/proxy — Meta's mcrouter — which centralizes routing, pooling, replication of hot keys, and shadow traffic without touching application code.

The config-drift gotcha: the server list is the cluster, so two app tiers with different lists silently split your cache into two half-warm caches with mixed staleness — no error is ever raised. Pool config must come from one source of truth (service discovery, config service), and rolling it out is a real deployment event: every ring change cools ~1/N of your cache.

  4. Toggle 🎫 Leases → ⏰ Expire → 🌊 → one lease-holder refills (1 DB query); 7 saved. Watch client-1's purple badge.
  5. 💀 Kill mc-1 (gutter off) → 🌊 twice → the ring re-hashes to mc-2: one cold wave, then warm — but imagine every hot key on the dead box doing this simultaneously.
  6. ↺ Reset → 💀 Kill → toggle 🛟 Gutter → 🌊 twice → the gutter absorbs the dead node's traffic with short TTLs; mc-2 never inherits the load (no cascading hot spot).

Model honesty: 8 clients stand in for tens of thousands — in production the difference between 1 and N cache-fill queries is the difference between a blip and an outage. Real leases are issued per key at a limited rate (on the order of once per ~10 s), and waiting clients typically retry within tens–hundreds of ms or are served slightly-stale data; the gutter is a small spare pool (single-digit % of the fleet), not a full replica.

🔍 Deep dive · The slab allocator — no fragmentation, but calcification click to expand

A cache that mallocs and frees millions of variably-sized values per second would shred its heap into unusable confetti. Memcached refuses to play: at startup it divides memory into slab classes, each serving one fixed chunk size, with sizes growing geometrically (default factor 1.25 — roughly 96 B, 120 B, 152 B, … up to the 1 MB page size). Memory is claimed in 1 MB pages, each page assigned to a class and carved into its chunks. An incoming item is stored in the smallest chunk that fits. Freed chunks return to their class's free list — same size, instantly reusable, fragmentation structurally impossible. The rent: internal waste (a 130 B item occupies a 152 B chunk), bounded by the growth factor.

The famous failure mode is slab calcification. Pages get assigned to classes on demand — and once the memory is fully claimed, the assignment is sticky. Ship a redesign that grows your objects from ~150 B to ~600 B, and the fleet's memory sits pinned in the 152 B class while the 616 B class thrashes with evictions at 40% total occupancy. Modern versions run an automover that rebalances pages between classes in the background, which largely defuses it — but "our hit rate cratered after a deploy that changed object sizes" is still a real incident pattern, diagnosed with stats slabs, and naming it is a senior tell.

Eviction is LRU per slab class, not global — each class evicts its own tail, so one hot size class can be evicting furiously while another sits idle (calcification's visible symptom). Modern Memcached upgraded the plain list to a segmented LRU — hot / warm / cold queues with a background crawler reaping expired items — so one mass scan can't flush the genuinely hot set. Phrase it lightly: "LRU per slab class, segmented in modern versions."

🔍 Deep dive · Scaling Memcache at Facebook (NSDI '13) — how a cache becomes a planetary system click to expand

The must-know paper for this page. Facebook ran (and Meta still runs) Memcached as a look-aside cache for sharded MySQL, at a scale the paper describes as billions of requests per second over trillions of items — and every mechanism they added lives around the unmodified-in-spirit dumb server. Four ideas earn interview points:

Leases — one mechanism, two bugs killed. On a miss, the server hands the client a short-lived lease token; only the token-holder may fill the key, and token issuance is rate-limited per key (on the order of once per 10 s). Everyone else briefly waits (or takes slightly-stale data). That kills the thundering herd — one miss = one DB query, not thousands. And because a delete invalidates outstanding tokens, a stale set from a reader who fetched before a concurrent DB write gets rejected — killing the stale-set race that plain cache-aside can't close (deep dive 3).

The gutter — a small spare pool (order of 1% of the fleet) that absorbs traffic for dead servers. Clients failing to reach a node retry against the gutter with short TTLs, so a node death costs a brief warm-up instead of hammering MySQL with the dead node's entire read load. Notably, they chose the gutter over re-hashing to surviving nodes: re-hashing migrates hot keys onto already-busy servers and risks a cascade — the widget above lets you feel that choice.

mcsqueal — invalidation as a pipeline, not a convention. Daemons tail the MySQL commit logs on the master region and fan out deletes (batched through mcrouter) to every frontend cluster. Invalidation driven by the authoritative write log can't be forgotten by an app codepath and gives an upper bound on staleness. Cross-region, writes from a replica region set a remote marker that detours reads to the master region until replication catches up — eventual consistency with the sharp edge filed down. Plus cold-cluster warmup: a freshly-built cluster serves its misses from a warm cluster's cache instead of the DB until its hit rate recovers.

🔍 Deep dive · Cache-aside correctly — delete-not-set, the stale-set race, TTL jitter click to expand

The pattern everyone writes: read = get → on miss SELECT → set. The half everyone gets wrong is the write path. After updating the DB, delete the key — don't set the new value. Two writers who both compute-then-set can interleave so the older value lands last and is now cached as if fresh, wrong until TTL. Delete is safe under any interleaving: worst case you forced an extra read. Say "invalidate by delete, repopulate on read" and you've passed this checkpoint.

But delete alone still leaves a narrow hole — the stale-set race: reader misses and SELECTs (old value) → writer updates DB and deletes the key → reader's belated set writes the old value back, now unexpiring-ly stale. This is precisely what Facebook's leases close: the delete invalidates the reader's lease token, so the poisonous late set is refused. Without leases, the mitigations are short TTLs (bound the damage), CAS-based fills, or a single-flight layer in your client. The general rule worth quoting: a TTL is not a consistency mechanism — it's a staleness upper bound; explicit invalidation plus TTL-as-backstop is the grown-up combination.

Three smaller patterns that mark production experience: TTL jitter — a deploy that sets a million keys with ttl=3600 schedules a synchronized miss-storm for one hour later; add ±10% randomness. Negative caching — cache "this doesn't exist" (short TTL) or nonexistent IDs become a hole punched straight through to the DB, queryable by any attacker in a loop. Probabilistic early refresh — have some readers treat a nearly-expired entry as a miss (probability rising as expiry nears), so hot keys refresh before the cliff instead of stampeding at it.

🔍 Deep dive · Multigets, hot keys & incast — the network is the failure domain click to expand

Loading one page of a social product fans out to hundreds of cache keys (the Facebook paper reports a median in the ~500-key range for popular pages). The client groups keys by owning server and issues one multiget per server — turning ~500 round trips into ~Nservers. This is why multiget exists, and why the sharding function being in the client is a performance feature: the batching happens where the key→server map lives.

Fan-out has a physics bill: incast congestion. Blast requests at 100 servers and their responses converge on your one NIC within microseconds of each other, overflowing shallow switch buffers → drops → TCP retransmits → a latency spike caused by your own burst. The classic fix (straight from the paper) is a sliding window of outstanding requests per client — small enough to avoid incast, large enough to avoid serializing; the same congestion-control idea TCP uses, one layer up.

Then there's the hot key: a celebrity profile or a global config blob that every request touches. Consistent hashing puts each key on exactly one node — so one key's traffic cannot be sharded, and a single node melts while nineteen idle. Fixes, in escalating order: a tiny in-process L1 (even a 1-second local TTL absorbs enormous read multiples), client-side key replication (write cfg#0…cfg#9, read a deterministic-per-client replica — trading N× invalidation cost for N× read capacity), or mcrouter-style proxy replication of designated hot pools. The interview phrase: "you can't shard a single key — you replicate it."

⚖️ At scale — numbers, limits & failure modes

Ballparks for capacity talk — order of magnitude is the point.

per-node throughputMultithreaded: a big box reaches the order of 10⁵–10⁶ ops/s on small values — commonly the NIC or packet rate saturates before the CPU. This raw ceiling per node is Memcached's classic edge over single-threaded designs.
latencyServer work is tens of µs; in-DC RTT dominates → ~0.1–0.5 ms typical gets. Budget the round trip, not the hash lookup.
item limitsValue ≤ 1 MB default (-I raises it; slab waste grows), key ≤ 250 B, per-item overhead ~50 B — a fleet of 100 B values is mostly overhead; batch tiny things into one value.
TTL semantics≤ 30 days = relative seconds; > 30 days = absolute unix timestamp (the classic "why did every key expire instantly" incident). 0 = no expiry, still evictable.
pool arithmeticRing change (add/remove/replace a node) cools ~1/N of the cache. Know your DB's headroom for a 1/N miss storm — or size a gutter (small, single-digit % of fleet).
reference scaleFacebook's memcache tier (NSDI '13): ~billions of reads/s, trillions of items, look-aside over sharded MySQL — the existence proof that the dumb-server model scales arbitrarily if the client layer is smart.
modern extensionsextstore spills values to NVMe/SSD keeping keys+metadata in RAM (10× capacity per node for warm data); recent versions can survive a clean restart with cache intact (not a crash). Persistence is still explicitly a non-goal.
Failure modes to name: node death → miss storm (1/N of reads suddenly hit the DB; the DB's spare capacity — or a gutter — decides if it's a blip or an outage); thundering herd on a hot key (one expiry → thousands of concurrent refills — leases/single-flight); slab calcification (object-size shift strands memory in old classes; hit rate craters post-deploy); synchronized TTL expiry (mass warmup with equal TTLs = scheduled stampede — jitter); hot key (unshardable by definition — replicate client-side or L1-cache); multiget incast (huge fan-out bursts overflow switch buffers — cap the request window).

8 · In the interview

How it comes up: you introduce it yourself — "reads are heavy, I'll cache-aside in front of Postgres" — and the follow-ups write themselves: how do you invalidate? (delete on write, TTL as backstop), what if a hot key expires under load? (leases/single-flight — walk them through the widget scenario), what if a cache node dies? (miss storm, not data loss; gutter or DB headroom), why Memcached and not Redis? (next paragraph, verbatim).

The choice script: "If the workload is pure look-aside blob caching and I want maximum throughput per node with the simplest possible operations — Memcached: multithreaded, scales with cores, nothing to operate. The moment I need data structures (sorted sets, counters with windows), persistence or replication, pub/sub, or atomic Lua, I take Redis — or its open-source continuation Valkey — and accept a bit more operational surface. The modern default skews Redis; Memcached persists where fleets are huge and the job is exactly one thing — Meta being the standing example." Then the senior close: at scale the interesting engineering is client-side either way — consistent hashing, leases, hot-key replication — and that logic is portable between the two.

Alternatives — one line each

Redis / ValkeyThe default competitor: data structures, persistence, replication/failover, pub/sub, Lua — a server that participates in your logic. Single-threaded core; shard to scale writes.
Hazelcast / IgniteJVM data grids: partitioned, replicated in-memory storage with compute colocated — heavier, for when the cache is the application platform.
CDN / edge cachesFor HTTP-shaped, shareable responses, cache at the edge before caching in the datacenter — different layer, often the bigger win.
Caffeine / GuavaIn-process L1 in front of Memcached's L2: zero network hop, absorbs hot keys; costs per-instance memory and per-instance staleness.
DAXDynamoDB's managed read-through/write-through cache — the "add a cache without writing cache code" move for Dynamo-backed designs.
Phrases that score: "cache-aside; on writes I delete the key, not set it" · "TTL is a staleness bound, not consistency — explicit invalidation plus TTL backstop" · "one lease per miss: refills are single-flight, so an expiry costs one DB query" · "a dead node is a miss storm, not data loss — the gutter absorbs it" · "TTL jitter so a deploy doesn't schedule a stampede" · "hot keys can't be sharded — replicate them client-side".

9 · Check yourself

After updating a row, why DELETE the cache key instead of SETting the fresh value?

Two concurrent writers can compute-then-set out of order: the older value lands last and is now cached as fresh — wrong until TTL. Delete is safe under any interleaving; the next reader repopulates from the DB, costing at most one extra read. The remaining hole is the stale-set race (a reader's in-flight fill from before the write lands after the delete) — closed by leases/CAS, since the delete invalidates the outstanding token.

A hot key expires and 5,000 requests hit it in the same 100 ms. Walk me through it.

Without protection: 5,000 misses → 5,000 identical DB queries — a thundering herd that can take the DB down, which then takes everything down (the cache can't refill from a dead DB). Fixes: leases/single-flight (one request refills; the rest wait briefly or serve stale), probabilistic early refresh (hot keys renew before the cliff), and TTL jitter so many keys can't reach the cliff together. The one-liner: "an expiry should cost one DB query, not one per waiting client."

A node dies in your 20-node pool. What actually happens, and is any data lost?

No data is lost — it's a cache; the truth lives in the DB. What happens is ~5% of reads become misses at once and land on the database. If the DB has headroom, it's a blip while the ring re-hashes those keys onto survivors (~1/N remap with consistent hashing) and they re-warm. If not, you want a gutter pool: a small spare set absorbing the dead node's traffic with short TTLs — Facebook chose this over re-hashing precisely because migrating hot keys onto already-loaded survivors risks a cascade.

Why can a single Memcached node out-throughput a single Redis node on plain GET/SET?

Memcached is multithreaded — worker threads across all cores serve requests concurrently from one process — while Redis executes commands on a single main thread (newer I/O threads help with network handling, but command execution stays serial). Add that Memcached does nothing per operation beyond hash-table + slab work, and one big box goes further. Redis's answer is running multiple shards per machine — valid, but now you're operating a cluster to match one process. This matters at the "maximum ops per node" margin; both are far beyond most workloads' needs.

Interviewer: "Sessions are small and hot — just keep them only in Memcached, it's fast."

Push back on durability, not speed: Memcached evicts under memory pressure (LRU, per slab class), loses everything on restart, and a node death drops ~1/N of sessions instantly — so "sessions only in Memcached" means "we're fine mass-logging-out users at random times." Either declare that acceptable explicitly (some products can), or keep sessions in Redis with replication/persistence, or make the cache a look-aside over a session table. It's the one-sentence test: if losing it makes anyone sad, it doesn't live in Memcached.
