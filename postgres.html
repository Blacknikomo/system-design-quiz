🐘 PostgreSQL — indexes, transactions & MVCC

Relational system of record. Watch an Index Scan jump straight to a row vs a Seq Scan reading every row, see a transaction stay invisible until COMMIT, and watch every tuple's xmin/xmax drive MVCC: UPDATE writes a new version, DELETE just stamps xmax, the observer session flips between Read Committed and Repeatable Read, and VACUUM reclaims dead tuples — unless a frozen snapshot pins them.
ACID B-tree index (default) MVCC xmin / xmax RC vs RR isolation

Postgres is a system of record: writes are Atomic, Consistent, Isolated, Durable. Reads can use a B-tree index to seek a row in O(log n) instead of scanning the whole table in O(n). Concurrency is handled by MVCC — each row update creates a new version, so readers never block writers.

Index types: B-tree is the default (equality & range). Also GIN for full-text search & JSONB containment, GiST/PostGIS for geo/spatial, and BRIN for huge naturally-ordered tables. An index trades write cost & disk for fast reads.
Trade-off: indexes speed up reads but slow writes (every INSERT/UPDATE/DELETE must also maintain the index) and consume disk. Index the columns you filter/join on — not everything.

Run SQL

Try a SELECT with the index ON (jumps straight to the row), then toggle the index OFF and SELECT again (scans every row). Then BEGIN and INSERT/UPDATE/DELETE — watch each tuple's xmin/xmax change. Flip the other session to REPEATABLE READ, COMMIT a change, and see its frozen snapshot ignore it. Finish with VACUUM — and see the RR snapshot pin dead tuples (the xmin-horizon gotcha).

What you're seeing

  1. Index Scan: the B-tree (sorted emails → row pointers) is binary-searched, jumping straight to the row. Seq Scan (index OFF) reads every tuple — including dead ones it must visibility-check and discard (why bloat hurts).
  2. xmin/xmax stamps: INSERT writes a tuple with xmin = my xid. UPDATE stamps the old version's xmax and writes a new tuple. DELETE only stamps xmax — no new tuple, nothing moves.
  3. COMMIT vs ROLLBACK: COMMIT marks the xid committed — all its stamps become real at once. ROLLBACK just marks the xid aborted in pg_xact: nothing is physically undone; aborted tuples (☠) wait for VACUUM.
  4. Isolation = snapshot timing: the other session at Read Committed re-snapshots per statement (sees each commit immediately); at Repeatable Read it keeps one frozen snapshot — later commits stay invisible to it, though the tuples' xmin/xmax are identical.
  5. VACUUM: reclaims dead tuples (heap + index entries) — but only those no live snapshot can still see. A held RR snapshot pins them: the xmin-horizon problem behind "idle in transaction" outages.

🔬 xmin / xmax — how MVCC actually decides visibility

Every heap tuple carries hidden system columns. The two that drive MVCC are xmin and xmax — a pair of transaction IDs (XIDs) that stamp a row version's birth and death. There is no separate "current value"; a live row is simply the tuple version that is visible to your snapshot. Query them directly: SELECT xmin, xmax, ctid, * FROM users;

xminXID of the transaction that created this tuple version (INSERT, or the new version an UPDATE produced). Never changes after write.
xmax0 = live/current. Non-zero = the XID that deleted or superseded this version (DELETE, or the old version of an UPDATE). Also set — with a lock flag in infomask — by SELECT … FOR UPDATE/SHARE.
cmin/cmaxcommand IDs — order visibility within the same transaction (statement N doesn't see rows a later statement N+1 writes).
ctidphysical location (page, offset). An UPDATE points the old tuple's t_ctid at the new version, forming the update chain Postgres walks to find the latest row.
The visibility rule (simplified). A tuple version is visible to a snapshot when its xmin is committed and in the past, and its xmax is 0, or points to a transaction that is aborted / still in progress / in the future. In pseudocode: 
visible(tuple, snapshot) =
     committed(tuple.xmin) AND NOT in_progress(tuple.xmin, snapshot)   -- born & already committed
  AND ( tuple.xmax == 0                                                -- never deleted, OR…
        OR NOT committed(tuple.xmax)                                   -- …deleter aborted / not yet committed
        OR in_progress(tuple.xmax, snapshot) )                         -- …deleter still running vs my snapshot
That's the whole engine: readers never block writers because a reader just evaluates this predicate against tuple versions that already exist — it never waits on a lock.

A snapshot is what "now" means to you

When a statement (or transaction) begins, Postgres takes a snapshot: {xmin, xmax, xip_list} — the lowest still-running XID, the next XID to be assigned, and the list of XIDs in-progress at that instant. The visibility rule above is evaluated against this snapshot. Isolation level is entirely about when the snapshot is taken and how long it's held.

Read Committed (default)A fresh snapshot per statement. Each new SELECT sees everything committed before it started → non-repeatable reads & phantoms are possible. On a write conflict it doesn't abort: it walks the update chain (via ctid) to the latest committed version and re-evaluates the WHERE (EvalPlanQual).
Repeatable Read (snapshot isolation)One snapshot taken at the first statement, held for the whole transaction. You never see XIDs that committed after you started, so reads are stable. A concurrent UPDATE/DELETE to a row you also write raises could not serialize access → you retry.
Serializable (SSI)Repeatable Read's snapshot plus Serializable Snapshot Isolation: tracks read/write dependencies with predicate locks (SIReadLock) and aborts one transaction in a dangerous cycle. Guarantees the result equals some serial order. Same xmin/xmax machinery underneath — SSI only adds conflict detection on top.
Key point for the interview: the tuple's xmin/xmax are identical regardless of isolation level. What changes is which snapshot the visibility check runs against. Read Committed re-snapshots every statement; Repeatable Read and Serializable freeze one snapshot at transaction start.

What each operation does to xmin/xmax

INSERTWrites one new tuple: xmin = my_xid, xmax = 0. Invisible to others until my_xid commits.
SELECTReads nothing, writes nothing. Just evaluates the visibility rule against each tuple version and returns the ones that pass. This is why it never blocks.
UPDATETwo-step, never in place: set the old version's xmax = my_xid, then INSERT a new tuple with xmin = my_xid, and chain old→new via ctid. Old readers keep seeing the old version; the old tuple becomes dead once no snapshot needs it → reclaimed by VACUUM.
DELETESets xmax = my_xid on the current version. No new tuple. Others still see the row until you commit; once committed and past all snapshots, VACUUM frees the space.
SELECT … FOR UPDATESets xmax = my_xid too, but flags it as a lock, not a delete, in the tuple's infomask. The row stays live & visible; other writers of that row block. This is how a row-level lock and a delete share the same field.
ROLLBACKNothing is physically undone. The aborted xid is simply marked aborted in pg_xact (commit log); the visibility rule then treats any tuple with that xmin as never-born and any xmax as never-deleted.
The cost of never overwriting. Because UPDATE and DELETE only stamp XIDs and leave old versions behind, tables accumulate dead tuples → bloat. VACUUM reclaims them, but only versions older than the oldest snapshot still running (the xmin horizon) — so one long-running or idle in transaction session pins the horizon and blocks cleanup fleet-wide. Separately, XIDs are 32-bit, so VACUUM FREEZE must stamp very old, still-live tuples as "frozen / always visible" before the counter wraps — miss it and you hit transaction-ID wraparound, the canonical Postgres outage.

⚖️ At scale — numbers, limits & failure modes

Concrete figures to quote in a capacity discussion. Round aggressively — orders of magnitude are the point, not precision.

vertical headrooma single primary is comfortable into the low tens of TB and ~tens of thousands of TPS on good hardware. Shard only when you must.
connectionseach connection = an OS process (a few MB + work_mem). Practical ceiling ~a few hundred direct connections → front with PgBouncer for thousands of clients.
read scalingstreaming replicas serve reads; async replicas lag, sync replicas add write latency.
partition / shardpartition (RANGE/HASH/LIST) when one table exceeds ~100s GB so the planner can prune; Citus for true horizontal sharding.
write costevery index slows writes; a HOT update avoids index bloat when no indexed column changes.
Scale failure modes: (1) Transaction-ID wraparound — autovacuum must freeze old rows or the 32-bit XID space wraps and the DB stops accepting writes (the canonical Postgres-at-scale outage). (2) Connection storms — thousands of direct connections thrash; pool. (3) A long-running / idle in transaction session holds the xmin horizon, blocking VACUUM → bloat. (4) Lock contention on hot rows — use SELECT … FOR UPDATE SKIP LOCKED for queue tables.

📖 Reference — infrastructure, sharding, SQL & recipes

Infrastructure & deployment

  • Primary + replicas: one primary takes writes; streaming replication ships the WAL (write-ahead log) to replicas, which replay it. Replicas serve reads.
  • Sync vs async: synchronous_commit + synchronous_standby_names makes the primary wait for a replica to fsync the WAL (no data loss, higher latency). Async (default) acks first → replicas lag.
  • Durability & PITR: changes hit the WAL before the heap; checkpoints flush dirty pages. A base backup + archived WAL gives point-in-time recovery (replay WAL to any moment).
  • Pooling: a backend is a heavyweight OS process — put PgBouncer (transaction-pooling) in front so thousands of clients share a small pool of backends.
  • HA / failover: Patroni (+ etcd/Consul) does leader election & auto-promotes a replica when the primary dies.
  • Backups: pg_dump (logical, per-DB) or a physical base backup (pg_basebackup) + WAL archive for PITR & fast restore.

Sharding & scaling

  • Scale reads with read replicas — route SELECTs to them, but watch replication lag (a replica read can be stale).
  • Partitioning: declarative RANGE / LIST / HASH partitions split one huge table into child tables; the planner prunes to the relevant partition.
  • Horizontal sharding: the Citus extension distributes a table across worker nodes by a shard key and fans queries out.
  • Vertical scaling goes far: a single beefy Postgres is comfortable into the low TBs — shard only when you must.
  • MVCC: each UPDATE writes a new tuple version, so readers don't block writers (and vice-versa); dead tuples are reclaimed by VACUUM.
  • Transaction-ID wraparound: autovacuum must freeze old rows or the 32-bit XID space wraps and the DB stops accepting writes — the canonical Postgres-at-scale outage.
  • Connections are processes: each connection ≈ a backend process with its own work_mem, so connection count is capped — front it with PgBouncer/pooling rather than thousands of direct connections.

Use-case recipes

🔑 Indexing strategy
-- composite: serves WHERE a=? AND b=? (leftmost-prefix rule)
CREATE INDEX ix_ab ON t (a, b);
-- partial: only index the rows you query
CREATE INDEX ix_active ON users (email) WHERE active;
-- covering: answer from the index alone (index-only scan)
CREATE INDEX ix_cov ON orders (user_id) INCLUDE (status, total);
A composite (a,b) index also serves WHERE a=? alone, but not WHERE b=?. Partial indexes shrink the index; INCLUDE lets Postgres skip the heap.
🔁 Optimistic concurrency (version column)
UPDATE accounts SET balance = :new, version = version + 1
WHERE id = :id AND version = :expected;   -- 0 rows ⇒ someone else won
-- app sees rowcount 0 → re-read and retry
No locks held between read and write. The version guard makes the UPDATE a no-op if another txn changed the row first — cheap when conflicts are rare.
🔒 Pessimistic lock (inventory)
BEGIN;
SELECT stock FROM items WHERE id = 42 FOR UPDATE;  -- locks the row
UPDATE items SET stock = stock - 1 WHERE id = 42;
COMMIT;                                            -- lock released
FOR UPDATE blocks other writers of that row until COMMIT — use when conflicts are likely (hot inventory) and you can't afford a lost update.
⬆️ Upsert
INSERT INTO counters (k, n) VALUES ('hits', 1)
ON CONFLICT (k) DO UPDATE SET n = counters.n + 1;
One atomic statement: insert if new, otherwise update. Needs a unique/PK constraint on the conflict target. DO NOTHING for insert-if-absent.
🔎 Full-text search
ALTER TABLE docs ADD COLUMN ts tsvector
  GENERATED ALWAYS AS (to_tsvector('english', body)) STORED;
CREATE INDEX ix_ts ON docs USING gin (ts);
SELECT * FROM docs WHERE ts @@ to_tsquery('english', 'postgres & index');
A tsvector column + GIN index gives stemmed, ranked search without a separate engine for moderate volumes.
🧩 JSONB (flexible schema)
CREATE TABLE events (id bigserial, payload jsonb);
CREATE INDEX ix_payload ON events USING gin (payload);   -- containment/key lookups
SELECT * FROM events WHERE payload @> '{"type":"click"}';
SELECT payload->>'userId' FROM events;
Store semi-structured data in one column; a GIN index on the JSONB makes @> containment queries fast. Promote hot fields to real columns when they stabilize.
