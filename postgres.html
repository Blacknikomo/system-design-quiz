🐘 PostgreSQL — indexes, transactions & MVCC

Relational system of record. Watch an Index Scan jump straight to a row vs a Seq Scan reading every row, see a transaction stay invisible until COMMIT, and watch an UPDATE write a new row version instead of overwriting in place.
ACID B-tree index (default) MVCC Read Committed

Postgres is a system of record: writes are Atomic, Consistent, Isolated, Durable. Reads can use a B-tree index to seek a row in O(log n) instead of scanning the whole table in O(n). Concurrency is handled by MVCC — each row update creates a new version, so readers never block writers.

Index types: B-tree is the default (equality & range). Also GIN for full-text search & JSONB containment, GiST/PostGIS for geo/spatial, and BRIN for huge naturally-ordered tables. An index trades write cost & disk for fast reads.
Trade-off: indexes speed up reads but slow writes (every INSERT/UPDATE/DELETE must also maintain the index) and consume disk. Index the columns you filter/join on — not everything.

Run SQL

Try a SELECT with the index ON (jumps straight to the row), then toggle the index OFF and SELECT again (scans every row). Then BEGIN a transaction and INSERT/UPDATE — changes stay invisible to other sessions until COMMIT.

What you're seeing

  1. Index Scan: the B-tree (sorted emails → row pointers) is binary-searched, jumping straight to the row.
  2. Seq Scan: with no index, Postgres reads every row top-to-bottom until it matches.
  3. BEGIN: INSERT/UPDATE become pending (dashed) — durable only after COMMIT, discarded by ROLLBACK.
  4. Another session uses Read Committed: it does not see your uncommitted changes.
  5. MVCC: an UPDATE writes a new tuple and marks the old one superseded (xmax) — readers can still read the old version.

📖 Reference — infrastructure, sharding, SQL & recipes

Infrastructure & deployment

  • Primary + replicas: one primary takes writes; streaming replication ships the WAL (write-ahead log) to replicas, which replay it. Replicas serve reads.
  • Sync vs async: synchronous_commit + synchronous_standby_names makes the primary wait for a replica to fsync the WAL (no data loss, higher latency). Async (default) acks first → replicas lag.
  • Durability & PITR: changes hit the WAL before the heap; checkpoints flush dirty pages. A base backup + archived WAL gives point-in-time recovery (replay WAL to any moment).
  • Pooling: a backend is a heavyweight OS process — put PgBouncer (transaction-pooling) in front so thousands of clients share a small pool of backends.
  • HA / failover: Patroni (+ etcd/Consul) does leader election & auto-promotes a replica when the primary dies.
  • Backups: pg_dump (logical, per-DB) or a physical base backup (pg_basebackup) + WAL archive for PITR & fast restore.

Sharding & scaling

  • Scale reads with read replicas — route SELECTs to them, but watch replication lag (a replica read can be stale).
  • Partitioning: declarative RANGE / LIST / HASH partitions split one huge table into child tables; the planner prunes to the relevant partition.
  • Horizontal sharding: the Citus extension distributes a table across worker nodes by a shard key and fans queries out.
  • Vertical scaling goes far: a single beefy Postgres is comfortable into the low TBs — shard only when you must.
  • MVCC: each UPDATE writes a new tuple version, so readers don't block writers (and vice-versa); dead tuples are reclaimed by VACUUM.

Use-case recipes

🔑 Indexing strategy
-- composite: serves WHERE a=? AND b=? (leftmost-prefix rule)
CREATE INDEX ix_ab ON t (a, b);
-- partial: only index the rows you query
CREATE INDEX ix_active ON users (email) WHERE active;
-- covering: answer from the index alone (index-only scan)
CREATE INDEX ix_cov ON orders (user_id) INCLUDE (status, total);
A composite (a,b) index also serves WHERE a=? alone, but not WHERE b=?. Partial indexes shrink the index; INCLUDE lets Postgres skip the heap.
🔁 Optimistic concurrency (version column)
UPDATE accounts SET balance = :new, version = version + 1
WHERE id = :id AND version = :expected;   -- 0 rows ⇒ someone else won
-- app sees rowcount 0 → re-read and retry
No locks held between read and write. The version guard makes the UPDATE a no-op if another txn changed the row first — cheap when conflicts are rare.
🔒 Pessimistic lock (inventory)
BEGIN;
SELECT stock FROM items WHERE id = 42 FOR UPDATE;  -- locks the row
UPDATE items SET stock = stock - 1 WHERE id = 42;
COMMIT;                                            -- lock released
FOR UPDATE blocks other writers of that row until COMMIT — use when conflicts are likely (hot inventory) and you can't afford a lost update.
⬆️ Upsert
INSERT INTO counters (k, n) VALUES ('hits', 1)
ON CONFLICT (k) DO UPDATE SET n = counters.n + 1;
One atomic statement: insert if new, otherwise update. Needs a unique/PK constraint on the conflict target. DO NOTHING for insert-if-absent.
🔎 Full-text search
ALTER TABLE docs ADD COLUMN ts tsvector
  GENERATED ALWAYS AS (to_tsvector('english', body)) STORED;
CREATE INDEX ix_ts ON docs USING gin (ts);
SELECT * FROM docs WHERE ts @@ to_tsquery('english', 'postgres & index');
A tsvector column + GIN index gives stemmed, ranked search without a separate engine for moderate volumes.
🧩 JSONB (flexible schema)
CREATE TABLE events (id bigserial, payload jsonb);
CREATE INDEX ix_payload ON events USING gin (payload);   -- containment/key lookups
SELECT * FROM events WHERE payload @> '{"type":"click"}';
SELECT payload->>'userId' FROM events;
Store semi-structured data in one column; a GIN index on the JSONB makes @> containment queries fast. Promote hot fields to real columns when they stabilize.
