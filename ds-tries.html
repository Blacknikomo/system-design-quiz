🌿 Tries (Prefix Trees)

A tree keyed by characters: each edge is one letter, each path spells a prefix, and marked nodes end words. Lookups cost O(L) in the key length — completely independent of how many words you've stored.
prefix tree O(L) lookup prefix queries autocomplete radix compresses it

A trie stores strings by sharing common prefixes. Walking from the root, each edge consumes one character; a node flagged end-of-word (green dot) means the path to it spells a stored word. Insert/Search walk one node per character. StartsWith just checks a prefix path exists. Autocomplete walks to the prefix node, then collects every word in the subtree below it.

The gotcha: a lookup costs O(L) where L = the length of the key — it is independent of n, the number of words stored. That's the whole point: 10 words or 10 million, finding "cat" still takes 3 hops. The trade-off is memory (a pointer array per node); radix / Patricia tries compress single-child chains to fix that. A hash set is usually faster by a constant factor for pure exact-membership (one hash + bucket probe vs L pointer hops — both are O(L) in key length) — but a hash set cannot do prefix queries at all. For real autocomplete you also augment each node with a cached top-K list of completions so you don't walk the whole subtree on every keystroke.

What you're seeing

  1. The root is empty; every edge below it is one character.
  2. Insert walks the word char-by-char, creating nodes as needed, then marks the last node end-of-word (green dot).
  3. Search exact walks the chars and checks the final node is an end mark.
  4. StartsWith only checks the prefix path exists — no end mark required.
  5. Autocomplete walks to the prefix node, then collects every end-marked word in the subtree below (purple).
