🟧 Kafka — partitions, offsets & consumer groups

A topic is a partitioned, append-only log. Produce a record, watch it get hashed to a partition, then see two consumer groups read the same retained data independently — and replay it from the start.
append-only log partitioned ordered within a partition retained & replayable offset per group

A topic is split into partitions; each is an ordered, immutable, append-only log. A record's key is hashed (partition = murmur2(key) % numPartitions) so the same key always lands in the same partition — that's how Kafka guarantees per-key ordering. Records are retained (not deleted on read), and each consumer group tracks its own committed offset per partition, so groups read the same data independently.

The gotcha: a low-cardinality or single "hot" key sends most traffic to one partition → a hot partition. Since consumer parallelism within a group is capped at the partition count (one consumer per partition per group), one skewed key bottlenecks the whole group no matter how many consumers you add.

What you're seeing

  1. Client PRODUCE(key,value) → broker hashes the key.
  2. partition = hash(key) % 3 picks the target column (same key → same partition, ordered).
  3. Record is appended at the partition's next offset — never overwritten or deleted on read.
  4. Each consumer group consumes the next record per partition and advances its own committed offset — both groups see all data independently.
  5. Replay resets a group's offsets to 0 so it re-reads everything — possible because Kafka retains records (unlike a queue that deletes on ack).
