🟧 Kafka — partitions, offsets & consumer groups

A topic is a partitioned, append-only log. Produce a record, watch it get hashed to a partition, then see two consumer groups read the same retained data independently — and replay it from the start. Records without a key are spread by the producer's default sticky partitioner (it fills a batch on one partition, then moves on) — even spread without the per-record cost of old round-robin.
append-only log partitioned ordered within a partition retained & replayable offset per group

A topic is split into partitions; each is an ordered, immutable, append-only log. A record's key is hashed (partition = murmur2(key) % numPartitions) so the same key always lands in the same partition — that's how Kafka guarantees per-key ordering. Records are retained (not deleted on read), and each consumer group tracks its own committed offset per partition, so groups read the same data independently.

The gotcha: a low-cardinality or single "hot" key sends most traffic to one partition → a hot partition. Since consumer parallelism within a group is capped at the partition count (one consumer per partition per group), one skewed key bottlenecks the whole group no matter how many consumers you add.

Run a command

Click PRODUCE to append a record. With a key it's hashed to one of 3 partitions; leave the key blank for the sticky partitioner (even spread, no per-key order). Then pick a group and CONSUME to read the next record on each partition.

Command log

What you're seeing

  1. Client PRODUCE(key,value) → broker hashes the key.
  2. partition = hash(key) % 3 picks the target column (same key → same partition, ordered).
  3. Record is appended at the partition's next offset — never overwritten or deleted on read.
  4. Each consumer group consumes the next record per partition and advances its own committed offset — both groups see all data independently.
  5. Replay resets a group's offsets to 0 so it re-reads everything — possible because Kafka retains records (unlike a queue that deletes on ack).

⚖️ At scale — numbers, limits & failure modes

Concrete figures to quote in a capacity discussion. Round aggressively — orders of magnitude are the point, not precision.

throughput / broker~100s MB/s to ~1 GB/s (sequential disk + NIC bound). Cluster scales to many GB/s by adding brokers.
partition = parallelismone partition is read by at most one consumer in a group and tops out at ~tens of MB/s. Add partitions to add throughput — but you can't reduce them later.
partitions / clusterZooKeeper era: ~4000/broker, ~200k/cluster (controller reloaded all state on failover — ~14 s at 200k). KRaft removes that ceiling → millions/cluster (single broker tested to ~600k).
retentionbounded by disk (time- or size-based). Tiered storage offloads cold segments to object storage so retention isn't capped by local disk.
durabilityRF=3 typical; acks=all + min.insync.replicas=2. Replication divides usable throughput by RF.
Scale failure modes: (1) Rebalance storms — adding/removing a consumer (or a slow heartbeat) reshuffles all partitions and pauses consumption; use the cooperative-sticky assignor + static membership. (2) Hot partition from a skewed key (one big customer) saturates one consumer — raise key cardinality or custom-partition. (3) Too many partitions → more open files, slower leader elections, higher end-to-end latency. (4) Consumer lag is the health metric — alert on it.

📖 Reference — infrastructure, sharding, commands & recipes

Infrastructure & deployment

  • Cluster: multiple brokers; a controller manages metadata + leader election — KRaft (Raft quorum of controllers) today, ZooKeeper historically.
  • Topic → partitions: each partition is an ordered append-only log; partitions spread across brokers for parallelism.
  • Replication: each partition has one leader + follower replicas (replication.factor=3); only in-sync followers — the ISR — are eligible for promotion on leader failure.
  • Durability knobs: acks=0|1|all (producer waits for none / leader / all ISR) paired with min.insync.replicas=2 — together they bound how many copies hold a write before it's acked.
  • Producers batch + compress (linger.ms, compression.type) for throughput.
  • Retention: time/size (retention.ms) or log compaction (cleanup.policy=compact keeps the latest value per key).
  • Rule of thumb: throughput scales with partition count; a broker handles high-hundreds-of-MB/s to GB/s depending on message size and replication. Storage per broker ranges from a few TB to tens of TB (more with tiered storage) — scale out when partition count, disk, or replication bandwidth saturates, not at a fixed message rate.

Sharding & scaling

  • Partition = unit of parallelism: within a consumer group exactly one consumer reads a partition → max parallelism = partition count.
  • Placement: partition = murmur2(key) % numPartitions — same key → same partition → per-key ordering. No key → the producer's sticky partitioner (fills one batch, then moves on) for even spread without per-key order.
  • Adding partitions scales throughput but breaks the existing key→partition mapping (rehash) — over-provision instead of resharding live.
  • Consumer groups + rebalancing: partitions are redistributed when consumers join/leave; offsets are committed per group. Rebalances are painful (stop-the-world) — cooperative/incremental rebalancing (KIP-429) and static membership (group.instance.id) reduce the churn.
  • Exactly-once scope: Kafka's exactly-once is per read-process-write within Kafka (transactions + idempotent producer), not end-to-end to arbitrary external sinks.
  • Tiered storage: offload old segments to object storage for cheap long retention without bloating brokers.
  • Hot partition: a low-cardinality/hot key skews one partition — no extra consumers help past one-per-partition.

Use-case recipes

🧱 Durable event log
# topic with 3 replicas
kafka-topics --create --topic orders --partitions 12 --replication-factor 3
# broker/topic: require 2 in-sync copies before ack
min.insync.replicas=2
# producer
acks=all                 # wait for all ISR
enable.idempotence=true  # no dup on retry
acks=all + min.insync.replicas=2 means a write survives one broker loss; idempotence stops producer retries from duplicating. The classic "don't lose data" config.
🔢 Ordered processing
# produce WITH a key so related events share a partition
kafka-console-producer --topic orders \
  --property parse.key=true --property key.separator=:
> order-42:created
> order-42:paid
> order-42:shipped     # all → murmur2("order-42")%N → same partition
Ordering is per-partition only. Keying by entity id (order-42) lands its events on one partition, preserving order — there is no global order across partitions.
👥 Multiple independent consumers
# each team = its own group.id, its own committed offsets
kafka-console-consumer --topic orders --group analytics
kafka-console-consumer --topic orders --group billing
# replay one team without touching others
kafka-consumer-groups --group analytics \
  --reset-offsets --to-earliest --topic orders --execute
A separate group.id per team reads the same retained data independently and tracks its own offset — so one team can replay from the start while others keep their position.
♻️ Retry + DLQ
# Kafka has NO native retry/DLQ — build topics for it
orders            # main
orders.retry.5s   # consumer re-publishes failures here, with delay
orders.dlq        # after N attempts → dead-letter topic
# consumer: try → on fail, produce to retry; exhausted → produce to dlq
Unlike SQS, Kafka won't retry or dead-letter for you. The pattern is explicit topics: a delayed retry topic and a terminal DLQ you can inspect/replay. Keep consumers idempotent.
🎯 Exactly-once (read-process-write)
# producer config
enable.idempotence=true
transactional.id=payments-tx-1
# in code
producer.initTransactions();
producer.beginTransaction();
producer.send(outRecord);
producer.sendOffsetsToTransaction(offsets, "consumer-grp");
producer.commitTransaction();   # output + consumed offset commit atomically
Idempotent producer kills in-flight dup; transactions make the output write and the input-offset commit atomic, giving exactly-once for read-process-write pipelines (consumers read with isolation.level=read_committed).
