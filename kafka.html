🟧 Kafka — partitions, replication, consumer groups & offsets

A topic is a partitioned, append-only log. Five drill-downs: produce & replay across groups, how a partition leader is elected from the ISR, how a real cluster lays out topics × partitions × replicas across brokers, how a consumer group assigns partitions, and how an offset is committed and reused.
append-only log partitioned ordered within a partition replicated (leader + ISR) retained & replayable offset per group

A topic is split into partitions; each is an ordered, immutable, append-only log living on a broker. Each partition is replicated (one leader + follower replicas; only the in-sync ones — the ISR — can be promoted). A record's key is hashed (partition = murmur2(key) % numPartitions) so the same key always lands in the same partition — that's per-key ordering. Records are retained (not deleted on read), and each consumer group tracks its own committed offset per partition, so groups read the same data independently.

The gotcha: a low-cardinality or single "hot" key sends most traffic to one partition → a hot partition. Since consumer parallelism within a group is capped at the partition count (one partition is read by at most one consumer per group), one skewed key bottlenecks the whole group no matter how many consumers you add.

Command log

What you're seeing

  1. Client PRODUCE(key,value) → broker hashes the key.
  2. partition = hash(key) % 3 picks the target column (same key → same partition, ordered).
  3. Record is appended at the partition's next offset — never overwritten or deleted on read.
  4. Each consumer group consumes the next record per partition and advances its own committed offset — both groups see all data independently.
  5. Replay resets a group's offsets to 0 so it re-reads everything — possible because Kafka retains records (unlike a queue that deletes on ack).
👑 Interactive · partition leader election & the ISR (in-sync replicas) click to collapse
1 leader + N followers followers FETCH from leader ISR = caught-up replicas High Watermark = committed elect leader from ISR

One partition, RF=3: replica on 3 brokers, one is the leader (all reads/writes go through it), the others are followers that pull (fetch) from the leader to replicate. The ISR is the set of replicas fully caught up with the leader (within replica.lag.time.max.ms, default 30 s). A record is committed — visible to consumers — only once every ISR member has it; that boundary is the High Watermark (HW). acks=all means the producer's ack waits for the full ISR.

The gotcha: on leader failure the controller elects a new leader from the ISR — guaranteed to hold every committed record, so no data loss. If the ISR is empty (all followers lagging) you're stuck: either wait, or enable unclean leader election (promote an out-of-sync replica) which trades durability for availability — committed records past its log are lost. Default is unclean.leader.election.enable=false.

Command log

What you're seeing

  1. Produce appends to the leader's log (LEO advances) — but it's beyond the HW, not yet committed/visible.
  2. Followers fetch from the leader and catch up; once all ISR members hold a record the HW advances → record is committed.
  3. A follower that stops fetching falls out of the ISR after the lag timeout — it can no longer be elected.
  4. Kill the leader: the controller elects a new leader from the ISR — it has every committed record, so no loss.
  5. Unclean election (ISR empty): an out-of-sync replica is promoted → committed records past its log are lost.
🏗️ Interactive · cluster layout — 3 topics × 6 partitions × RF 2 across brokers click to collapse
AutoMart dealership 3 topics 6 partitions each RF = 2 3 brokers

A car dealer's event backbone — three topics, each 6 partitions, RF=2: inventory (cars added / sold / re-priced, keyed by VIN), test-drives (bookings, keyed by showroom), orders (purchases, keyed by order-id). That's 3 × 6 × 2 = 36 replicas. Why 3 brokers? RF=2 needs ≥2 brokers; 3 lets the 36 replicas spread evenly (12 per broker, of which 6 are leaders) and gives the KRaft controller a real quorum. Each partition's 2 replicas always sit on different brokers, so every broker holds a replica of exactly 12 of the 18 partitions and is absent from 6.

The RF=2 gotcha (why prod prefers RF=3): with min.insync.replicas=2 + acks=all, losing one broker drops the ISR to 1 on every partition that had a replica there → those writes are rejected (NotEnoughReplicas) until it returns. RF=3 keeps 2 in-sync copies through a single broker loss, so writes continue. Kill a broker below to watch 12 partitions go under-replicated.

Each broker box lists the partitions it holds per topic — L# = it leads that partition, F# = it follows. Kill a broker to watch its 6 led partitions fail over to their followers (now promoted leaders) and all 12 of its partitions become under-replicated.

Command log

What you're seeing

  1. Each of the 18 partitions (3 topics × 6) has 2 replicas on distinct brokers — leader + 1 follower.
  2. Leadership is balanced: 18 leaders ÷ 3 brokers = 6 leaders per broker, 12 replicas total each.
  3. Kill a broker → its 6 led partitions re-elect: the surviving follower becomes leader (promoted).
  4. All 12 partitions that had a replica there are now under-replicated (RF effectively 1).
  5. With min.insync.replicas=2 those 12 reject acks=all writes until the broker returns — the RF=2 trade-off.
👥 Interactive · consumer-group assignment — one partition ↔ one consumer click to collapse
1 group 6 partitions partition → exactly one consumer a consumer can hold many extras sit idle

Within a single consumer group, the group coordinator hands each partition to exactly one consumer. The reverse is not symmetric: a consumer can own several partitions. So the real rule is "one partition is read by at most one consumer in the group", not "one consumer reads only one partition". Add/remove consumers and watch the assignor (round-robin here) redistribute on each rebalance.

The gotcha: max useful consumers = partition count. With 6 partitions, a 7th or 8th consumer gets nothing — it sits idle. You can't out-scale the partition count; to add parallelism you must add partitions (and accept the key→partition remap). Different groups each get the full set of partitions independently — that's fan-out, not extra parallelism within one group.

  5. Parallelism within the group is capped at the partition count.
📌 Interactive · offsets — position vs committed, lag & resume-after-crash click to collapse
position ≠ committed commit → __consumer_offsets lag = LEO − committed resume from committed at-least-once

A consumer tracks two cursors. Position is the next offset it will fetch — it moves as you poll. The committed offset is what's durably saved (to the internal __consumer_offsets topic) and only moves when you commit. The committed offset is the resume point: after a crash or rebalance, the consumer restarts from it — not from its in-memory position. Consumer lag = log-end-offset (LEO) − committed, the headline health metric.

The gotcha — delivery semantics live in the commit timing. Commit after processing → a crash before the commit re-delivers those records → at-least-once (duplicates; make handlers idempotent). Commit before processing → a crash mid-process loses them → at-most-once. Auto-commit (enable.auto.commit=true, every auto.commit.interval.ms≈5 s) commits your position on a timer — convenient but it can do either, which is why critical pipelines commit manually (or use transactions for exactly-once).

Command log

What you're seeing

  1. POLL advances position (next offset to fetch) and processes the record — but doesn't save anything.
  2. COMMIT writes position to __consumer_offsets; committed catches up; lag shrinks.
  3. Lag = LEO − committed — how far behind the group is on this partition.
  4. Crash → position snaps back to committed; the polled-but-uncommitted range is re-delivered.
  5. Re-processing those = at-least-once duplicates → idempotent handlers, or transactions for exactly-once.

⚖️ At scale — numbers, limits & failure modes

Concrete figures to quote in a capacity discussion. Round aggressively — orders of magnitude are the point, not precision.

throughput / broker~100s MB/s to ~1 GB/s (sequential disk + NIC bound). Cluster scales to many GB/s by adding brokers.
partition = parallelismone partition is read by at most one consumer in a group and tops out at ~tens of MB/s. Add partitions to add throughput — but you can't reduce them later.
partitions / clusterZooKeeper era: ~4000/broker, ~200k/cluster (controller reloaded all state on failover — ~14 s at 200k). KRaft removes that ceiling → millions/cluster (single broker tested to ~600k).
retentionbounded by disk (time- or size-based). Tiered storage offloads cold segments to object storage so retention isn't capped by local disk.
durabilityRF=3 typical; acks=all + min.insync.replicas=2. Replication divides usable throughput by RF.
Scale failure modes: (1) Rebalance storms — adding/removing a consumer (or a slow heartbeat) reshuffles all partitions and pauses consumption; use the cooperative-sticky assignor + static membership. (2) Hot partition from a skewed key (one big customer) saturates one consumer — raise key cardinality or custom-partition. (3) Too many partitions → more open files, slower leader elections, higher end-to-end latency. (4) Consumer lag is the health metric — alert on it.

📖 Reference — infrastructure, sharding, commands & recipes

Infrastructure & deployment

  • Cluster: multiple brokers; a controller manages metadata + leader election — KRaft (Raft quorum of controllers) today, ZooKeeper historically.
  • Topic → partitions: each partition is an ordered append-only log; partitions spread across brokers for parallelism.
  • Replication: each partition has one leader + follower replicas (replication.factor=3); only in-sync followers — the ISR — are eligible for promotion on leader failure.
  • Durability knobs: acks=0|1|all (producer waits for none / leader / all ISR) paired with min.insync.replicas=2 — together they bound how many copies hold a write before it's acked.
  • Producers batch + compress (linger.ms, compression.type) for throughput.
  • Retention: time/size (retention.ms) or log compaction (cleanup.policy=compact keeps the latest value per key).
  • Rule of thumb: throughput scales with partition count; a broker handles high-hundreds-of-MB/s to GB/s depending on message size and replication. Storage per broker ranges from a few TB to tens of TB (more with tiered storage) — scale out when partition count, disk, or replication bandwidth saturates, not at a fixed message rate.

Sharding & scaling

  • Partition = unit of parallelism: within a consumer group exactly one consumer reads a partition → max parallelism = partition count.
  • Placement: partition = murmur2(key) % numPartitions — same key → same partition → per-key ordering. No key → the producer's sticky partitioner (fills one batch, then moves on) for even spread without per-key order.
  • Adding partitions scales throughput but breaks the existing key→partition mapping (rehash) — over-provision instead of resharding live.
  • Consumer groups + rebalancing: partitions are redistributed when consumers join/leave; offsets are committed per group. Rebalances are painful (stop-the-world) — cooperative/incremental rebalancing (KIP-429) and static membership (group.instance.id) reduce the churn.
  • Exactly-once scope: Kafka's exactly-once is per read-process-write within Kafka (transactions + idempotent producer), not end-to-end to arbitrary external sinks.
  • Tiered storage: offload old segments to object storage for cheap long retention without bloating brokers.
  • Hot partition: a low-cardinality/hot key skews one partition — no extra consumers help past one-per-partition.

Use-case recipes

🧱 Durable event log
# topic with 3 replicas
kafka-topics --create --topic orders --partitions 12 --replication-factor 3
# broker/topic: require 2 in-sync copies before ack
min.insync.replicas=2
# producer
acks=all                 # wait for all ISR
enable.idempotence=true  # no dup on retry
acks=all + min.insync.replicas=2 means a write survives one broker loss; idempotence stops producer retries from duplicating. The classic "don't lose data" config.
🔢 Ordered processing
# produce WITH a key so related events share a partition
kafka-console-producer --topic orders \
  --property parse.key=true --property key.separator=:
> order-42:created
> order-42:paid
> order-42:shipped     # all → murmur2("order-42")%N → same partition
Ordering is per-partition only. Keying by entity id (order-42) lands its events on one partition, preserving order — there is no global order across partitions.
👥 Multiple independent consumers
# each team = its own group.id, its own committed offsets
kafka-console-consumer --topic orders --group analytics
kafka-console-consumer --topic orders --group billing
# replay one team without touching others
kafka-consumer-groups --group analytics \
  --reset-offsets --to-earliest --topic orders --execute
A separate group.id per team reads the same retained data independently and tracks its own offset — so one team can replay from the start while others keep their position.
♻️ Retry + DLQ
# Kafka has NO native retry/DLQ — build topics for it
orders            # main
orders.retry.5s   # consumer re-publishes failures here, with delay
orders.dlq        # after N attempts → dead-letter topic
# consumer: try → on fail, produce to retry; exhausted → produce to dlq
Unlike SQS, Kafka won't retry or dead-letter for you. The pattern is explicit topics: a delayed retry topic and a terminal DLQ you can inspect/replay. Keep consumers idempotent.
🎯 Exactly-once (read-process-write)
# producer config
enable.idempotence=true
transactional.id=payments-tx-1
# in code
producer.initTransactions();
producer.beginTransaction();
producer.send(outRecord);
producer.sendOffsetsToTransaction(offsets, "consumer-grp");
producer.commitTransaction();   # output + consumed offset commit atomically
Idempotent producer kills in-flight dup; transactions make the output write and the input-offset commit atomic, giving exactly-once for read-process-write pipelines (consumers read with isolation.level=read_committed).
