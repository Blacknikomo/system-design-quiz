⛏️ Heaps & Priority Queues

A binary heap shown two ways at once: as a tree and as its flat backing array. Push bubbles up, pop sifts down — both O(log n). Toggle min vs max and try the top-K demo.
complete binary tree array-backed push/pop O(log n) peek O(1)

A binary heap is a complete tree stored in a plain array — no pointers. For node at index i: children are 2i+1 and 2i+2, parent is ⌊(i−1)/2⌋. The heap property says every parent beats its children (smaller in a min-heap, larger in a max-heap), so the extreme element is always at the root (index 0). Push appends at the end and sifts up; pop swaps root with the last element, removes it, and sifts down.

The gotcha: a heap is only partially ordered — the root is the min/max, but the rest is not sorted, so you can't iterate it in order for free (that's what heapsort's repeated pops are for). The classic trip-up: to keep the K largest elements, use a min-heap of size K (not a max-heap) — the smallest of your current top-K sits at the root, ready to be evicted when something bigger arrives. Getting that direction backwards is the bug. Also: building a heap from n items bottom-up is O(n), not O(n log n).

What you're seeing

  1. The heap lives in an array; the tree is just that array drawn by index math.
  2. Push: append at the end, then sift up — swap with parent while it beats the parent.
  3. Pop: take the root, move the last element to the root, then sift down — swap with the better child.
  4. Each sift touches at most log n levels → O(log n).
  5. Top-K: a min-heap of size K keeps the largest K; its root is the weakest survivor.
