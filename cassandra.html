🟦 Cassandra — ring, QUORUM & the write path

Masterless peer-to-peer ring. Hash a partition key to find its 3 replica owners, write/read at QUORUM, watch the LSM write path (commit log + memtable → SSTables), and see how tunable consistency survives a node failure.
masterless ring RF = 3 LSM-tree (append-only) tunable consistency gossip + seed nodes

Cassandra has no leader. Any node can be the coordinator; it hashes the partition key to a token and the next RF=3 nodes around the ring own that key. Writes are durable via the commit log (WAL) and fast because they only append to an in-memory memtable — an LSM-tree, not a B-tree. With W + R > RF (here 2 + 2 > 3) read and write quorums overlap, so a QUORUM read always sees the latest QUORUM write.

The signature gotcha: a poor partition key creates unbounded partitions. If Discord had partitioned messages by channel_id alone, a busy channel becomes one giant ever-growing partition on a few nodes (hotspot, slow reads, compaction pain). The fix: add a time bucket to the key → (channel_id, bucket) so each partition is bounded.

Run an operation

Press WRITE. The key is hashed to a token; the 3 consecutive nodes that own it light up. The coordinator forwards to all 3 replicas and needs 2 acks (QUORUM) to succeed. Each replica appends to its commit log + memtable.

What you're seeing

  1. Hash the partition key → a token on the ring; the next RF=3 nodes own it (replica set).
  2. A coordinator forwards the write to all 3 replicas; QUORUM = ⌊RF/2⌋+1 = 2 acks needed.
  3. On each replica the write hits the commit log (WAL) for durability, then the memtable (in-memory).
  4. Flush seals the memtable into an immutable SSTable on disk; the memtable clears.
  5. A QUORUM read contacts 2 replicas; since W(2)+R(2) > RF(3) the sets overlap → latest timestamp wins (read repair).
  6. Delete writes a tombstone (a marker), never an in-place delete. Compaction merges SSTables and purges tombstones + superseded values.
