🔎 Elasticsearch — inverted index & search

Watch documents get tokenized into an inverted index (term → list of doc IDs), then see why a search hits only the relevant terms — while a SQL LIKE '%term%' has to scan every row.
inverted index tokenize + stem fuzzy / edit-distance built on Lucene sharded

When you index a document, Elasticsearch analyzes the text: lowercases it, splits on non-letters into tokens, drops stopwords (the, a, of…), and stems words (running → run). Each resulting term points to a posting list of the docs that contain it. A search analyzes the query the same way, looks up just those terms, and intersects their posting lists — no scan of the corpus.

Sharding: an index is split into primary shards, and each shard is a complete, self-contained Lucene index with its own inverted index. A query fans out to every shard and the results are merged. (Fuzzy matching uses edit-distance on terms to catch typos like concrt → concert.)
The gotcha: Elasticsearch is not your source of truth — keep a primary database and sync ES via CDC (change-data-capture). Also, the primary shard count is fixed at index creation; changing it requires a full reindex into a new index.

✍️ Interactive · how a WRITE (index) request is routed & stored click to collapse
coordinating node routing = hash(_id) % primaries primary → replica buffer + translog refresh → segment near-real-time

A write travels four hops: ① coordinating node (any node can receive it) → ② find the shard via shard = hash(_routing) % number_of_primaries → ③ find the node that holds that primary shard (cluster routing table) → ④ build the index on the primary: analyze → in-memory buffer + translog, then a refresh turns the buffer into a searchable Lucene segment. The primary also replicates to its replica before acking the client.

The gotcha — near-real-time: a doc is durable the instant it's in the translog, but not searchable until the next refresh (default refresh_interval: 1s) opens a segment. The in-memory buffer is invisible to search. _flush later fsyncs a Lucene commit and truncates the translog; background merges compact small segments into bigger ones.

What you're seeing

  1. Coordinating node — any node receives the write & coordinates it (owns routing + fan-out; needn't hold the shard).
  2. Find the shard — shard = hash(_routing) % number_of_primaries; _routing defaults to _id.
  3. Find the node — the cluster routing table says which data node holds that primary shard; forward there.
  4. Build the index — primary analyzes the doc → appends to in-memory buffer + translog (durable, not yet searchable).
  5. Replicate & ack — primary → replica shard; once wait_for_active_shards ack, client gets 201 created.
  6. Refresh (~1s) — buffer → immutable segment → now searchable. (flush fsyncs + clears translog; merge compacts segments.)
🔭 Interactive · how a search reads data — query then fetch (top-N) click to collapse
scatter / gather query phase: {id, score} coordinator merge fetch phase: _source

A search runs in two phases. Query phase: the coordinator scatters the query to one copy of every shard; each shard finds its local top-N and returns just {_id, _score} — cheap, no document body. Merge: the coordinator sorts all shards' hits and keeps the global top-N (e.g. top-20). Fetch phase: it asks only the shards holding those final IDs for the real _source, assembles, and returns.

The gotcha — deep pagination: for from:10000, size:20 every shard must ship from+size = 10020 hits to the coordinator, which sorts ~that×shards just to keep 20. Cost grows with the offset → use search_after (cursor) or scroll/PIT for big exports.

🗑️ Interactive · how a delete works — tombstone in a segment, purged at merge click to collapse
immutable segments delete = mark in liveDocs tombstone purged on merge

Lucene segments are immutable — you can't edit one in place. A delete just flips the doc's bit in the segment's liveDocs bitset (a .liv tombstone): the doc vanishes from results immediately but is still on disk. The space is reclaimed only when a background merge rewrites the live docs into a fresh segment and drops the old ones. (An update is just delete-then-reindex — the old version becomes a tombstone.)

Terminology check: this is the Lucene model — segments + a liveDocs tombstone, purged at merge. Same idea as an LSM tree's SSTable tombstone + compaction (Cassandra), but Elasticsearch is not built on SSTables — don't mix the names in an interview. Heavy delete/update churn ⇒ many tombstones ⇒ merge pressure & wasted disk until merges catch up.

⚖️ At scale — numbers, limits & failure modes

Concrete figures to quote in a capacity discussion. Round aggressively — orders of magnitude are the point, not precision.

shard sizeaim 10–50 GB/shard. Too small wastes per-shard overhead; too large makes recovery/rebalance slow.
heap / node≤ ~31 GB — above ~32 GB the JVM loses compressed object pointers and you get less usable heap. Leave the rest of RAM for the OS file cache (Lucene depends on it).
shards / node≤ ~20 shards per GB of heap → a 30 GB-heap node ~600 shards max. Oversharding is the #1 cluster killer.
shard countnumber_of_shards is fixed at index creation (reindex + alias swap to change). number_of_replicas is live-adjustable.
ingest_bulk (NDJSON) is the only way to index at volume; near-real-time — docs searchable after refresh (default 1 s).
Scale failure modes: (1) Oversharding — thousands of tiny shards exhaust heap with cluster-state/segment overhead and slow everything. (2) Deep pagination (from+size past ~10k) is O(n) across shards — use search_after. (3) Heavy aggregations / fielddata on text blow up heap → circuit breaker. (4) Mapping explosion (unbounded dynamic fields) can OOM the cluster. (5) Not a source of truth — rebuild from the primary DB via CDC.

📖 Reference — infrastructure, sharding, commands & recipes

Infrastructure & deployment

  • Cluster of nodes with roles: master-eligible (cluster state, shard allocation), data (hold shards, run queries), coordinating (fan-out + merge), ingest (run pipelines pre-index). Small clusters share roles.
  • Index → primary shards, each replicated to replica shards on other nodes for reads + HA. A query fans out to one copy of each shard and merges.
  • Each shard is a full Lucene index made of immutable segments; new docs go to a new segment, deletes are tombstones, segments are merged in the background.
  • Near-real-time: a doc isn't searchable until a refresh opens a new segment — default refresh_interval: 1s. Durability is the per-shard translog.
  • Not a source of truth. Keep a primary DB and sync ES from it (CDC) — treat the index as a rebuildable, derived view. Built on Apache Lucene.

Sharding & scaling

  • number_of_shards is fixed at index creation — to change it you must reindex into a new index and swap an alias (zero-downtime cutover).
  • number_of_replicas is live-adjustable (PUT /idx/_settings) — add replicas to scale read throughput + availability, drop to 0 for fast bulk loads.
  • Routing: shard = hash(_routing) % number_of_shards, _routing defaults to _id. Custom _routing co-locates related docs on one shard.
  • Time-based data (logs/metrics): roll over to time-based indices behind a write alias, managed by ILM (hot → warm → cold → delete).
  • Avoid oversharding: each shard has fixed overhead; aim for ~10–50 GB/shard and keep total shard count proportional to heap, not tiny shards everywhere.

Key API

PUT /idx {settings, mappings}create index — shard count, replicas, field types/analyzers
POST /idx/_doc {..}index a doc (auto _id); PUT /idx/_doc/ID to set the id
GET /idx/_doc/IDfetch a doc by id (real-time, bypasses refresh)
POST /idx/_search {query}run a query — match / bool / aggs / sort / paging
POST /_bulkbatch index/update/delete (NDJSON) — the way to ingest at volume
GET /idx/_analyze {analyzer, text}inspect how text is tokenized into terms
POST /_aliases {actions:[add/remove]}atomically repoint an alias — reindex/rollover swaps
PUT /idx/_settings {number_of_replicas}live-tune replicas, refresh_interval, etc.

Use-case recipes

🔍 Full-text search (match + fuzziness)
POST /articles/_search
{
  "query": {
    "match": {
      "title": {
        "query": "concrt tickets",
        "operator": "and",
        "fuzziness": "AUTO"   # tolerate typos (edit distance)
      }
    }
  }
}
# the field's analyzer (lowercase, stem, stopwords) runs on query + doc
Analyzed terms are looked up in the inverted index, not scanned. fuzziness:AUTO catches concrt→concert; tune relevance with boost / minimum_should_match.
🧮 Filter + aggregation (facets)
POST /products/_search
{
  "query": {
    "bool": {
      "must":   [ { "match": { "name": "running shoe" } } ],
      "filter": [ { "term":  { "brand": "acme" } },
                  { "range": { "price": { "lte": 100 } } } ]   # cacheable, no scoring
    }
  },
  "aggs": {
    "by_color":  { "terms": { "field": "color" } },          # facet counts
    "avg_price": { "avg":   { "field": "price" } }
  }
}
must scores relevance; filter is a yes/no bitset (cached, fast). aggs compute facets/metrics over the matched set in one round-trip.
⌨️ Autocomplete (edge_ngram / completion)
# option A: index-time edge_ngram analyzer → prefix terms "co","con","conc"...
PUT /idx { "settings": { "analysis": { "filter": {
  "edge": { "type": "edge_ngram", "min_gram": 2, "max_gram": 15 } } } } }
# option B: completion suggester (FST in memory, very fast)
POST /idx/_search
{ "suggest": { "s": { "prefix": "conc",
    "completion": { "field": "suggest" } } } }
edge_ngram = full search-as-you-type with filters; completion suggester = fastest prefix lookup but limited filtering. Pick by whether you need to combine with queries.
🔄 Sync from a primary DB (CDC)
# primary DB stays the source of truth; ES is a derived index
Postgres/MySQL  →  Debezium (CDC)  →  Kafka  →  Kafka Connect ES sink  →  Elasticsearch
# alternative: app dual-writes (DB then ES) — simpler, but risk of drift on failure
POST /_bulk
{ "index": { "_index": "products", "_id": "42" } }
{ "name": "Acme Runner", "price": 90, "brand": "acme" }
The index lags the DB slightly — fine for search. CDC gives ordered, replayable updates and lets you fully rebuild the index from the DB at any time.
📄 Deep pagination (search_after)
# BAD past a few pages: from/size re-collects everything up to the offset
POST /idx/_search { "from": 10000, "size": 10 }     # expensive, capped
# GOOD: search_after on a stable sort (tiebreak on _id)
POST /idx/_search
{ "size": 10,
  "sort": [ { "created": "desc" }, { "_id": "asc" } ],
  "search_after": [ 1718000000000, "doc#123" ] }   # cursor from last hit
# for full exports use the scroll API / point-in-time (PIT)
from/size is O(offset) per shard and capped by index.max_result_window. search_after is a stateless cursor; scroll/PIT snapshots the index for large exports.
