🪜 Skip Lists

A sorted linked list with randomized "express lanes" on top. Search starts at the top-left, skips far ahead, and drops down — giving O(log n) expected search without any tree rotations.
probabilistic O(log n) expected ~50% promotion Redis ZSET LSM memtable

A skip list is several sorted linked lists stacked up. Level 0 contains every key. Each higher level is a sparser "express lane": a node appears on level k+1 only if a coin flip promotes it. Search starts top-left and moves right while the next key ≤ target, otherwise drops down a level — so the express lanes skip over big chunks of the list. Insert finds the spot, then flips coins to decide how tall the new node is.

The gotcha: skip lists are probabilistic — O(log n) is the expected height/cost, not a worst-case guarantee (a pathological run of coin flips could make it tall and thin). The ~50% promotion probability gives expected height ≈ log₂ n and ~2× total node-copies across all levels (a geometric series: n + n/2 + n/4 + … ≈ 2n). This is how Redis sorted sets (ZSET) are implemented and how the in-memory memtable works in LSM-tree stores (LevelDB / RocksDB / Cassandra) — easier to make concurrent than a balanced tree.

What you're seeing

  1. Each node lives on level 0 and, by coin flips, on some higher express lanes.
  2. Search starts at the top-left head. Move right while next ≤ target; else drop down one level.
  3. Express lanes let you skip many nodes at once → O(log n) expected hops.
  4. Insert: find the position on each level, then flip a coin (~50%) repeatedly to set the new node's random height.
  5. Delete: unlink the node from every level it appears on.
