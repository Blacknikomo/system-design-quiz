🕸️ Graphs — BFS / DFS / Topo sort

An adjacency-list graph. Watch BFS fan out level-by-level with a queue (shortest paths in an unweighted graph), DFS plunge deep with a stack, and topological sort order a DAG.
adjacency list BFS = queue DFS = stack topo = DAG only

A graph is vertices + edges, stored as an adjacency list (each node → list of neighbors) — compact for sparse graphs. BFS uses a FIFO queue to visit in rings of increasing distance, so on an unweighted graph it yields the shortest path (fewest hops). DFS uses a LIFO stack (or recursion) to dive as deep as possible first. Topological sort linearizes a DAG so every edge points forward — the order you'd run tasks/build targets in.

The gotcha: BFS gives the shortest path only in unweighted graphs; once edges have weights you need Dijkstra (a min-heap / priority queue). A classic error is thinking DFS finds the shortest path — it finds a path, not the shortest. Topological sort exists only for a DAG: if there's a cycle there is no valid ordering — which is exactly how you detect a dependency cycle. And adjacency list (sparse) vs matrix (dense) is a real space/lookup trade-off, not a detail.

What you're seeing

  1. BFS: dequeue a node, mark visited, enqueue all unseen neighbors → the queue holds the current frontier.
  2. Because BFS expands by rings, each node's distance badge is its shortest hop-count from start.
  3. DFS: pop the top of the stack, push its neighbors → it plunges down one branch before backtracking.
  4. Topo sort: repeatedly take a node with in-degree 0 (Kahn's algorithm); the emitted order respects every edge.
  5. If no in-degree-0 node remains but vertices are left → a cycle; a DAG has no valid topo order, so this detects dependency cycles.
