🌊 Flink — windows, state & watermarks

Count events per key in tumbling 10-second windows. Send events (even out-of-order ones), advance the event-time clock, and watch the watermark trail behind to decide exactly when each window closes and emits — and what happens to events that arrive too late.
stateful streaming tumbling windows event-time watermarks checkpointed state → exactly-once

Flink processes an unbounded stream and keeps managed state — here, a running count per key inside each tumbling window [0,10), [10,20), …. Events carry an event timestamp; they route to the window that contains it, not the window that's "current". A watermark trails the event-time clock by a bounded "max out-of-orderness" lag; when the watermark passes a window's end, Flink decides no more in-order data is coming, closes the window and emits its aggregate. State survives crashes via checkpointing, giving exactly-once results.

First ask: do you actually need real-time? If a daily/hourly batch job is fine, reach for Spark / batch — it's simpler and cheaper. Flink earns its keep only when you need low-latency, continuous, stateful results over an unbounded stream.
The gotcha: "I'll just keep window counts in a plain in-memory map in my own service." That map is lost on crash, can't be redistributed when you scale out / rebalance keys, and has no principled way to handle late or out-of-order events — you'd either close windows too early or never. Flink's checkpointed keyed state + watermarks are exactly what solve all three.

What you're seeing

  1. An event (key, timestamp) routes to the window whose range contains its timestamp — by event-time, not arrival order.
  2. The window's per-key count is incremented in managed keyed state (this is the stateful aggregation; a stateless map/filter would keep nothing).
  3. +5s advances the event-time clock; the watermark follows at clock − 5s (bounded out-of-orderness).
  4. When the watermark passes a window's end, that window closes & emits its aggregate, then is greyed out.
  5. An event landing in an already-closed window is LATE → dropped by default (Flink could instead route to a side output or allow lateness).

⚖️ At scale — numbers, limits & failure modes

Concrete figures to quote in a capacity discussion. Round aggressively — orders of magnitude are the point, not precision.

throughputscales horizontally to millions of events/s across a cluster; per-subtask throughput × parallelism. Parallelism is set per operator.
state sizekeyed state in RocksDB spills to local disk → state can reach TBs; checkpointed to S3/HDFS for recovery.
checkpointsperiodic, async; interval trades recovery time vs overhead. Backpressure propagates upstream via network buffers (no data loss).
rescalingtake a savepoint, restart with new parallelism; Flink redistributes key-groups across slots.
latencytrue streaming → ms-level; event-time + watermarks handle out-of-order data with bounded lateness.
Scale failure modes: (1) Key skew — a hot key (one big tenant) sends all its events + state to one slot; the job is only as fast as the hottest slot. Pre-aggregate or salt the key. (2) State explosion — unbounded keyed state (no TTL) grows until checkpoints time out; set StateTtlConfig. (3) Checkpoint stalls under backpressure (alignment) → use unaligned checkpoints. (4) Watermark too tight drops late data; too loose fires windows late.

📖 Reference — infrastructure, scaling, API & recipes

Infrastructure & deployment

  • Job Manager (coordinator) — schedules tasks, triggers & coordinates checkpoints, handles recovery. Task Managers (workers) run the actual work in task slots (one slot = one parallel pipeline of subtasks).
  • Cluster modes: session cluster (long-lived, many jobs share it) vs application cluster (one job per cluster, isolated lifecycle). Deploy on Kubernetes or YARN.
  • State backends: heap (small state, fast) vs embedded RocksDB (state larger than RAM, spills to local disk) — pick RocksDB for big keyed state.
  • Checkpoints: periodic, automatic snapshots of all operator state to durable storage (S3/HDFS); on failure the job restores and replays from the last one → fault tolerance.
  • Savepoints: manual, versioned snapshots you trigger for upgrades, rescaling, or code changes — restart the job from one to migrate safely.

Sharding & scaling

  • Parallelism is set per operator (or job-wide default); each parallel instance is a subtask running in a slot.
  • keyBy partitions the stream by key (hash) so all events for a key — and that key's keyed state — live on one slot. The unit of state distribution.
  • Rescale: take a savepoint, restart with new parallelism; Flink redistributes keyed state across the new slots (key-groups).
  • Event-time + watermarks handle out-of-order / late data: a watermark of T asserts "no more events < T expected" and triggers event-time windows/timers.
  • Backpressure propagates naturally upstream through the network buffers — a slow sink throttles sources without dropping data.

Key API / concepts (DataStream)

env.addSource(...) / env.fromSource(...)ingest a stream (Kafka, files, sockets, …)

Use-case recipes

🪟 Windowed aggregation (clicks/min)
stream
  .keyBy(e -> e.userId)
  .window(TumblingEventTimeWindows.of(Time.minutes(1)))
  .aggregate(new CountAgg());      // one count per user per minute
The bread-and-butter streaming job: per-key counts/sums over fixed windows. Use aggregate/reduce (incremental) over process on a window to avoid buffering all events.
🧹 Stateful dedup (drop duplicate event ids)
ValueState<Boolean> seen;            // keyed by eventId
// in processElement:
if (seen.value() != null) return;    // already saw it → drop
seen.update(true);
out.collect(event);
// configure TTL so state doesn't grow forever
StateTtlConfig.newBuilder(Time.hours(24)).build();
Keyed ValueState per id remembers what's been seen. Always attach a TTL — unbounded keyspace (event ids) would otherwise blow up state size.
⏰ Late data (allowedLateness + side output)
OutputTag<Event> late = new OutputTag<>("late"){};
stream.keyBy(k).window(TumblingEventTimeWindows.of(Time.minutes(1)))
  .allowedLateness(Time.minutes(5))          // keep window state 5 min past watermark
  .sideOutputLateData(late)                  // truly-late events go here, not dropped
  .aggregate(new CountAgg());
By default events past the watermark are dropped. allowedLateness re-fires the window for slightly-late data; the side output captures the rest for repair/audit instead of silent loss.
🔗 Stream-stream join
clicks.keyBy(c -> c.adId)
  .intervalJoin(impressions.keyBy(i -> i.adId))
  .between(Time.seconds(-5), Time.minutes(10))   // click within window of impression
  .process(new JoinFn());
// or window join: .join(other).where(k).equalTo(k).window(...)
Join two keyed streams on a key within a bounded time interval/window. The interval bounds how much state is buffered — unbounded joins need careful TTL.
✅ Exactly-once sink
env.enableCheckpointing(60_000, EXACTLY_ONCE);
KafkaSink.builder()
  .setDeliveryGuarantee(DeliveryGuarantee.EXACTLY_ONCE)  // 2-phase commit
  .setTransactionalIdPrefix("clicks-")
  .build();
End-to-end exactly-once = checkpointing + a transactional sink that commits on checkpoint (Kafka two-phase commit). First ask: do I need real-time? Many problems are cheaper as a batch job (Spark) — only reach for Flink when low-latency stateful streaming truly matters.
