🪣 S3 & Object Storage — the internet's hard drive

Buckets, keys and immutable blobs: the storage layer every design with files stands on — plus the two patterns interviewers actually probe, presigned URLs and multipart upload. ☕ ~15 min read
flat namespace immutable objects multipart upload presigned URLs 11 nines

S3 is a giant key→blob map over HTTPS: PUT an immutable object under a key, GET it back from anywhere, never think about disks, RAID or filesystems again. You trade away filesystem semantics — no append, no rename, no real directories — and in exchange get effectively bottomless capacity, eleven nines of durability, and a price per GB that made every other way of storing files obsolete. Almost every system design that involves a file ends with the same sentence: "the bytes go to S3; the database keeps the key."

In the interview: any design with user files (photos, video, PDFs, ML datasets, backups) stores blobs in object storage, metadata in a database, and moves bytes with presigned URLs so your API never proxies payloads. Expect follow-ups on multipart upload (the 40 GB file on hotel Wi-Fi), per-prefix request limits, the upload → event → worker pipeline, and the egress bill you forgot about.

1 · The mental model: an infinite coat check

Think of S3 as a coat check with no cloakroom limit. You hand over a blob and a ticket string of your choosing — the key — and the attendant stores it somewhere you will never see. Present the ticket, get the exact bytes back. The attendant never opens the box, never edits its contents, never rearranges your tickets into folders. Everything else about object storage falls out of this picture.

First: the namespace is flat. A bucket is one giant sorted map from key strings to objects; photos/2026/07/cat.jpg is a single opaque key that merely contains slashes. The console draws folders because LIST supports a prefix and a delimiter, so it can pretend — but there is no directory to create, no directory to be empty, and crucially no atomic rename: "moving" an object is a server-side copy to a new key plus a delete of the old one, and "renaming a folder" is a copy-storm over every key sharing the prefix.

Second: objects are immutable. There is no append, no edit-in-place, no "just update bytes 100–200". A PUT to an existing key replaces the whole object; if versioning is on, the old version is retained rather than destroyed (and DELETE just adds a delete marker you can undo). Concurrent PUTs to the same key resolve as last-writer-wins — object storage has no locks. This immutability is not a limitation to work around; it's the property that makes caching, replication and eleven-nines durability tractable. Systems that need mutable, appendable data (logs, databases) are built on top of immutable objects, not out of them.

Third: everything is an HTTP request to a service — there's no connection state, no mount point, no file handle. One object can be up to 5 TB; each carries metadata (content type, cache headers, ~2 KB of your own key-values). Latency is that of an HTTPS round-trip to a regional service: first byte in the tens of milliseconds, not microseconds — S3 is a warehouse, not a cache.

The one-sentence test: if you'd describe the data as files that are written once and read whole (or in ranges) — object storage. If you need to append, lock, rename atomically, or read at memory speed — you're describing a log, a filesystem, or a cache, and S3 will punish the impersonation.

2 · Syntax & core API

The API surface that matters in an interview is small — a handful of verbs plus two mechanisms (presigning and multipart) that carry the whole design pattern.

PUT /bucket/keyWrite an object — whole thing, one shot, ≤ 5 GB per single PUT. Replaces silently if the key exists.
GET /bucket/keyRead it back. Add Range: bytes=0-1048575 for a byte slice — this is how video players seek and how download managers parallelize.
DELETE / COPYDelete (or add a delete marker under versioning); server-side copy for "rename" — above 5 GB even the copy must be multipart.
LIST (prefix, delimiter)Paginated, 1,000 keys per page, lexicographic order. Fine for occasional listing; a disaster as a query engine (more below).
Multipart uploadCreateMultipartUpload → UploadPart ×N → CompleteMultipartUpload. Parts 5 MB–5 GB, up to 10,000; parallel, individually retryable; required past 5 GB, recommended from ~100 MB.
Presigned URLAny request, signed with your credentials, handed to someone who has none. Expiry up to 7 days. THE pattern for direct client upload/download.
Event notifications3:ObjectCreated:* → SQS / SNS / Lambda / EventBridge, delivered at-least-once. The trigger of every ingestion pipeline.
Lifecycle ruleDeclarative aging: "after 30 days → Infrequent Access, after 365 → Glacier, after 7 years → delete; and always: abort incomplete multiparts after 7 days."
# the verbs, as the CLI speaks them
aws s3api put-object    --bucket media --key videos/v42.mp4 --body v42.mp4
aws s3api get-object    --bucket media --key videos/v42.mp4 --range bytes=0-1048575 first-mb.bin
aws s3api list-objects-v2 --bucket media --prefix videos/ --delimiter / --max-keys 1000

# presign: your API runs this (one HMAC computation, no I/O), client does the heavy lifting
aws s3 presign s3://media/videos/v42.mp4 --expires-in 900
# → https://media.s3.amazonaws.com/videos/v42.mp4?X-Amz-Expires=900&X-Amz-Signature=…
# the client PUTs/GETs that URL directly — your servers never see the bytes

Presigning is pure math — an HMAC over the canonical request using your secret key. No call to AWS happens when you sign; S3 verifies the signature when the client shows up. That's why it costs your API microseconds, and why expiry and clock skew live in the URL itself.

3 · Use cases — when object storage is the answer

🎬 Media originals, with a CDN in front
Users upload photos/video via presigned URLs; originals land in S3; workers derive thumbnails and transcodes back into S3; CloudFront serves the public reads with S3 as origin. The database stores keys and metadata only — never blobs.
Why S3: bottomless capacity and 11-nines durability for the one copy that must never be lost; the CDN absorbs read heat and the egress bill.
🧊 Data lake — storage and compute finally divorced
Events land as Parquet files partitioned by prefix (events/date=2026-07-05/…); Athena / Trino / Spark query them in place. Storage costs ~2 cents/GB-month while compute spins up only for the query.
Why S3: this killed HDFS — no cluster to keep alive just to hold bytes, and ten teams' engines can read the same files without ten copies.
💾 Backups & archives on a lifecycle conveyor
Database snapshots and compliance data PUT to Standard, then a lifecycle policy walks them down the price ladder: IA after 30 days, Glacier tiers after a year, delete after the retention period. Versioning + Object Lock for the ransomware/auditor conversation.
Why S3: durability is the product; retrieval latency (minutes–hours in Glacier tiers) is a price you volunteer to pay, not a surprise.
📦 The claim check — big payloads for small pipes
SQS caps messages at 256 KB; Kafka hates multi-MB blobs. So: PUT the payload to S3, enqueue {bucket, key}, and the consumer fetches the bytes itself. Same trick for email attachments, workflow engines, inter-service handoffs.
Why S3: queues move coordination, object storage moves bulk — each system does the thing it's priced and built for.

When it's the WRONG answer

filesystem semanticsAtomic rename, directories, appends, POSIX locks — the things build systems and legacy apps assume. That's EFS/NFS territory (or a redesign).
hot small reads1 KB values at p99 < 10 ms, millions of times a minute → that's a KV store / cache (DynamoDB, Redis). S3's tens-of-ms first byte and per-request pricing make it a terrible database.
append-heavy logsObjects don't append — "add a line" means rewrite the object. Streams of records belong in Kafka / a log service, which may tier to S3 later.
query by anything but key"Find all videos of user 7 longer than 2 min" via LIST = a full prefix scan, 1,000 keys a page. Metadata queries belong in a real database that stores the keys.

4 · Distributed setup — what AWS runs so you don't have to

Under the flat API sits a storage fleet that treats disk, host and even datacenter failure as weather. Each object is erasure-coded across at least three Availability Zones (for Standard): the object is cut into fragments plus parity, spread so that losing an AZ still leaves enough pieces to reconstruct every byte. That redundancy — not heroic hardware — is where "designed for 99.999999999% durability" comes from: the eleven nines are a math statement about independent failures, and the practical translation is "durability is no longer your problem; deleting the wrong thing yourself is" (hence versioning and MFA-delete). Availability is a separate, humbler number: Standard is designed for 99.99% and the SLA starts crediting you below ~99.9% — your design should tolerate a region having a bad afternoon.

The fact to state explicitly because half the internet's articles predate it: since December 2020, S3 is strongly consistent for all operations — read-after-write for new objects, overwrites, deletes, and LIST reflects completed writes immediately. The old "eventual consistency for overwrite PUTs" dance (and the read-your-stale-object bug reports it spawned) is history; if an interviewer's follow-up assumes the old model, correcting it politely is free senior signal. What remains asynchronous is everything between buckets and tiers: cross-region replication lags seconds to minutes (Replication Time Control adds a 15-minute SLA), and lifecycle transitions run as background jobs — strong consistency is per-key, per-bucket, not a global transaction.

Scale-wise, S3 partitions the keyspace by prefix and scales each partition to a ballpark of 3,500 writes/s and 5,500 reads/s, splitting partitions as sustained traffic grows. More prefixes → more parallel capacity, effectively without limit. Storage classes are the same durability math at different temperatures: Standard for hot, Standard-IA / One Zone-IA for monthly-touched (retrieval fee, ~30-day minimum), the Glacier tiers for archives (instant / minutes–hours / half-a-day-plus, ~90–180-day minimums), and Intelligent-Tiering to have a robot make the call per object.

The gotcha cluster: strong consistency did not add transactions — there's no multi-object atomicity, and two clients PUTting one key is still last-writer-wins (conditional writes with If-None-Match/If-Match, added in 2024, are the new escape hatch for create-if-absent and CAS). And replication is not a backup: it happily replicates your bad delete. Versioning is the backup.

5 · 🎛 Play with it — presigned flow + multipart upload

The full upload pattern: the client asks your API to sign, then ships bytes directly to S3 — watch the "bytes proxied" counter on your API stay at signature size. Pick a file size and mode: multipart cuts it into 100 MB parts uploaded in 8 parallel lanes with per-part retry; single PUT is one long request. Flip on the flaky network and compare how much each mode wastes when connections drop. (Time is compressed; failures are seeded — same settings replay the same run.)

What to try

  1. 2.5 GB, multipart, flaky OFF → Upload. Watch: sign request → presigned URLs → parts stream in 8 lanes → CompleteMultipartUpload assembles 25 ETags into one object.
  2. Flaky ON, Upload again → parts drop mid-flight, and only the dead part retries. Check the wasted-bytes counter: a failure costs at most ~100 MB.
  3. Switch to single PUT, flaky ON → one drop restarts from byte 0. Compare wasted bytes and total time with step 2.
  4. All the while: Your API — bytes proxied: ~KB. The payload never touched your servers; they only did HMAC math.
  5. Slide to 5 GB, single PUT: you're at the hard ceiling — past this size the object can only exist via multipart (on up to 5 TB).
🔍 Deep dive · Presigned URLs — why your servers never touch bytes click to expand

The naive upload design routes file bytes through your API: client → API → S3. Now every 2 GB upload occupies one of your app servers for minutes, your load balancer needs body-size limits raised, request timeouts stretched, and your fleet scales with bandwidth instead of with requests. The fix is to notice that the only thing your API adds is authorization — and a signature can carry authorization without carrying bytes.

A presigned URL is the request itself — method, bucket, key, expiry — HMAC-signed with your credentials. Generating one is local math: no AWS round-trip, microseconds of CPU. Your API authenticates the user, decides "yes, you may PUT to uploads/u123/v42.mp4 for the next 15 minutes", signs exactly that, and steps out of the data path. The client talks to S3 directly; S3 verifies the signature and enforces the expiry. Downloads mirror it: sign a GET for a private object so the mobile app can fetch it without your proxying. (For browser form uploads there's the POST-policy variant, which can also cap the content length — a presigned PUT does not enforce size, so quota checks happen at the "sign" step and on the ObjectCreated event.)

The operational fine print interviewers poke at: URLs expire (max 7 days with SigV4) — so clock skew between you and the client can make fresh URLs "already expired", and long multiparts need re-signing logic; possession is authorization — anyone holding the URL can use it until expiry, so keep expiries short and keys unguessable; and the URL inherits the signer's permissions, so sign with a role scoped to that one prefix, not your admin credentials. For public hot content, the same idea graduates to CDN signed URLs/cookies at the edge.

🔍 Deep dive · Multipart upload & ranged reads — parallelism as the throughput knob click to expand

A single HTTP connection to S3 tops out at some modest per-stream throughput — call it tens of MB/s on a good day — and a 40 GB upload over one connection is both slow and fragile: at 99% done, a dropped TCP session costs you 40 GB of retransmission. Multipart upload attacks both problems: CreateMultipartUpload opens a session, the client slices the file into parts (5 MB–5 GB each, up to 10,000, last part exempt from the minimum), uploads them in parallel over separate connections, collects an ETag per part, and finally CompleteMultipartUpload submits the ordered ETag list — at which point S3 stitches the object together server-side, atomically. The object either appears whole or not at all; readers never see half a file.

The properties that matter: per-part retry (a failed part re-uploads alone — the demo above makes the contrast visceral), parallel throughput (aggregate bandwidth scales with lanes until your NIC or S3's per-prefix limit pushes back), and pause/resume (parts already uploaded survive a client restart, since the session persists server-side). The dark corner: an abandoned session's parts sit invisibly and bill forever — they're not objects, LIST won't show them — until you AbortMultipartUpload or, better, set the lifecycle rule AbortIncompleteMultipartUpload after N days. Every real S3 estate has a story about finding terabytes of orphaned parts.

Downloads have the mirror-image trick: ranged GETs. Range: bytes=… lets a video player seek without fetching the file, lets a download manager run 8 parallel streams, and lets a data engine read just the Parquet footer + one column chunk out of a 1 GB file. Same lesson both directions: S3 throughput is a parallelism game, not a single-stream game.

🔍 Deep dive · Consistency — what changed in 2020, and what's still async click to expand

For its first 14 years, S3 was the canonical "eventually consistent" example: a new PUT was read-after-write (with caveats), but overwrites and deletes could serve stale data for a while, and LIST could lag reality. Whole design patterns existed just to cope — write-once keys, manifest files, "list-after-write" retry loops. In December 2020 AWS flipped the model: all S3 operations are now strongly consistent — what a completed PUT wrote is what any subsequent GET or LIST sees, overwrites and deletes included, at no extra cost or latency. When you see an article (or an interviewer) reciting the old model, date-stamp your correction: it's the cleanest way to show your knowledge isn't secondhand.

What strong consistency does not mean: no transactions (you cannot atomically PUT two objects, or PUT + delete), no isolation (two writers to one key race, last-writer-wins; versioning retains both, but "current" is whoever finished last), and no global convergence — cross-region replication is asynchronous (seconds to minutes typically; RTC sells a 15-minute SLA), lifecycle transitions and inventory reports are batch jobs, and event notifications are at-least-once and can arrive out of order, so consumers stay idempotent.

The newest chapter: in 2024 S3 added conditional writes — If-None-Match: * to create-only-if-absent, and ETag-based If-Match compare-and-swap on overwrite. That upgrades S3 from "no coordination whatsoever" to "single-key optimistic concurrency", which is exactly enough for patterns like leader-elected manifest files and table-format commit protocols (the Iceberg/Delta crowd celebrated loudly). Multi-object atomicity still belongs to databases layered on top.

🔍 Deep dive · Storage classes & the cost model — where the bill actually comes from click to expand

S3 pricing has three meters: GB-months stored (order of 2¢/GB-month in Standard), requests (order of $5 per million PUTs, $0.4 per million GETs), and egress to the internet (order of 5–9¢/GB, region-dependent). Intuition for which meter dominates: cold archives → storage; millions of tiny objects → requests; anything popular and public → egress, by a mile. A petabyte sitting still costs ~$25k/month; a petabyte downloaded costs multiples of that — which is why hot public content goes behind a CDN (cheaper per GB, and it absorbs the request storm), and why "we'll just serve it straight from S3" is a bill-shock story with a predictable ending.

The storage classes are one durability design sold at different temperatures. Standard: hot, no retrieval fee. Standard-IA: ~40% cheaper storage, but per-GB retrieval fees and a ~30-day minimum — for data touched monthly, not daily. One Zone-IA: same idea, one AZ, for re-derivable data only. The Glacier tiers: Instant (archive price, milliseconds access), Flexible (minutes-to-hours restores), Deep Archive (~half a day, cheapest — order of $1/TB-month), with ~90–180-day minimums. Intelligent-Tiering watches each object's access and moves it automatically for a small monitoring fee — the honest default when you can't predict access. At the opposite extreme, Express One Zone (2023) trades replication breadth for single-digit-millisecond access at a higher price — a niche for latency-sensitive analytics scratch space.

Lifecycle policies are how any of this stays managed: declarative rules per prefix/tag that transition objects down the ladder and eventually delete them. Two rules belong in every bucket you design out loud: abort incomplete multiparts after ~7 days, and expire old versions if versioning is on — otherwise "durable" quietly becomes "hoarding".

⚖️ At scale — numbers, limits & failure modes

The figures worth quoting — rounded, order-of-magnitude, as capacity talk should be.

object limitsObject ≤ 5 TB; single PUT ≤ 5 GB; multipart parts 5 MB–5 GB, ≤ 10,000 parts; key ≤ 1,024 bytes; user metadata ~2 KB.
request ratesPer prefix: ~3,500 writes/s, ~5,500 reads/s — and prefixes are unlimited, so aggregate scales with key design. Sudden ramps can briefly throw 503 SlowDown while partitions split: back off, retry, keep ramping.
latencyFirst byte ~tens of ms (p99 into the low hundreds); per-connection throughput modest → parallelize (multipart up, ranged GETs down). Express One Zone exists for single-digit ms.
durability / availabilityDesigned for 11 nines durability via erasure coding across ≥3 AZs; availability designed ~99.99%, SLA floor ~99.9%. Durability ≠ protection from your own DELETE — that's versioning / Object Lock.
LIST1,000 keys/page, lexicographic, strongly consistent — but O(keys under prefix) and priced per request. Never a query engine, never a queue.
consistencyStrong for all single-key ops since Dec 2020. Cross-region replication async (RTC: 15-min SLA); events at-least-once; conditional writes (2024) give single-key CAS.
cost anchors~2¢/GB-month Standard · egress ~5–9¢/GB · ~$5/M PUTs, ~$0.4/M GETs · Deep Archive ~$1/TB-month. Egress and per-request costs are where designs go to die.
Failure modes that only appear at scale: LIST-as-a-database (a prefix with 50 M keys, paged 1,000 at a time, on the request path — index keys in a real DB instead); the hot key (one viral object concentrates reads on one partition and can't be sharded — front it with a CDN/cache); tiny-object disease (a billion 4 KB objects: request costs and LIST pain dwarf storage — pack/batch them); orphaned multipart parts (invisible, billed, forever — lifecycle-abort them); presigned-URL expiry vs clock skew (mobile clients with drifted clocks "can't upload" — sign with margin); egress bill shock (serving hot public bytes region-direct instead of via CDN); replication ≠ backup (CRR ships your deletes too; versioning + lifecycle is the undo button).

8 · In the interview

How it comes up: never as "design S3" (that's a specialty question) — always as the storage layer you place in the first five minutes of any file-bearing design: Instagram, YouTube, Dropbox, a document service, ML training data. The canonical pipeline you should be able to draw half-asleep: client → (your API signs) → presigned PUT to S3 → ObjectCreated event → SQS → workers → derived artifacts back to S3 → CDN serves reads; the database stores keys + metadata, never blobs. Each arrow is a follow-up: why presigned (bytes bypass your fleet), why an event+queue (retryable async processing), why CDN (egress + latency), why keys-in-DB (queryability).

The choice script: object storage is the default for anything ≥ ~a few hundred KB that's written once and read whole or by range — say it, place it, move on. The interesting boundaries to name: metadata and small hot values go to a DB/cache, not S3 (latency + request cost); append/streaming workloads go to a log; if the requirement is POSIX (shared filesystem for legacy apps), that's EFS and it's fine to say so. And the pattern that upgrades your answer from correct to senior: DB stores the pointer, S3 stores the bytes, and the two are reconciled by events + idempotent workers — you've now answered the "what if the upload succeeds but your DB write fails" follow-up before it was asked.

9 · Check yourself

Why do uploads go client → S3 directly with a presigned URL, instead of through your API servers?

Because the API adds only authorization, and a signature carries that without the bytes. Proxying would make your fleet scale with upload bandwidth (long-held connections, big bodies, stretched timeouts, double egress) instead of with requests. The API authenticates, signs a narrowly-scoped URL (one method, one key, short expiry) in microseconds of local HMAC math, and steps out; S3 enforces the signature and expiry. Your servers see kilobytes of signing traffic while terabytes flow past them.

Design the upload of a 40 GB file from a flaky hotel connection. What mechanisms do you name?

Multipart upload — mandatory anyway above 5 GB: slice into parts (say 100 MB → 400 parts; limits are 5 MB–5 GB per part, ≤10,000 parts), upload 6–10 in parallel over separate connections, retry only failed parts, resume across client restarts (the session and completed parts persist server-side), then CompleteMultipartUpload assembles the object atomically. Presigned URLs per part (re-sign as they expire), and a lifecycle rule to abort incomplete multiparts so an abandoned attempt doesn't bill silently forever.

A teammate says "S3 is eventually consistent, so after upload we must poll until the object appears." Correct them precisely.

Outdated by half a decade: since December 2020 S3 is strongly consistent for all operations — a completed PUT is immediately visible to GET and LIST, overwrites and deletes included. What is still asynchronous: cross-region replication (seconds–minutes; 15-min SLA with RTC), lifecycle transitions, and event notification delivery (at-least-once, possibly reordered — keep the consumer idempotent). And strong consistency isn't atomicity: multi-object updates still race; single-key CAS exists via conditional writes (2024).

One meme image in your bucket goes viral — 500k requests/min. What breaks, and what's the fix?

A single key lives on one partition; per-prefix capacity is on the order of 5,500 GETs/s, and a hot single object can't be split across prefixes — key-sharding tricks don't help one key. You may see 503s, and even if S3 keeps up, you're paying internet egress per view. Fix: put a CDN in front (CloudFront) so the edge absorbs ~all reads with S3 as origin fetching once per edge location, add cache headers, and for private content use CDN signed URLs. In-region, an application cache does the same job.

Why is building a work queue on "workers LIST the bucket and grab new files" a design smell, and what's the idiomatic pipeline?

LIST is paginated (1,000/page), priced per request, O(keys under the prefix), and has no claim semantics — two workers list the same "new" object and both process it; nothing marks progress; polling burns money as the bucket grows to millions of keys. Idiom: S3 event notification → SQS — ObjectCreated events enqueue work, SQS provides visibility timeouts, retries and DLQ, workers scale on queue depth. Events are at-least-once, so processing stays idempotent (e.g., conditional write of the derived artifact).
