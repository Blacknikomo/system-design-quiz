🍃 MongoDB — documents, replica sets & the shard-key bet

The document database: store data the way the app reads it, tune consistency per query, and choose your shard key like you're choosing a spouse. ☕ ~15 min read
document model BSON ≤ 16 MB replica set tunable consistency shard key

MongoDB stores whole aggregates as single documents — the order with its line items, address and status packed together, the way the application will read it. That one bet buys you atomic single-document updates, no joins on the hot path, and easy horizontal sharding; the price is that your access patterns become your schema, and a badly chosen document boundary (or shard key) is the most expensive mistake in the system.

In the interview: MongoDB shows up when entities are heterogeneous and read as a whole — product catalogs, user profiles, CMS content, flexible event payloads. The interviewer expects you to defend it against "why not Postgres with JSONB?", to reason about embed vs reference, to name the w:1 rollback gotcha, and to explain why sharding on a monotonic key melts one shard.

1 · The mental model: assembled boxes, not parts bins

Open a relational database and you're standing in a parts warehouse: customers in one aisle, orders in another, order lines in a third — everything shelved by type, and every read is a walk through the aisles with a picking list (joins) to reassemble the thing you actually wanted. Open MongoDB and you're in a room of finished boxes: each document is a whole aggregate — the order with its items, shipping address and payment history inside — pre-packed exactly the way it will be read. The whole philosophy fits in one sentence: data that is accessed together is stored together.

Physically, a document is BSON — binary JSON with real types (ObjectId, dates, 64-bit ints, Decimal128, raw binary), capped at 16 MB per document. Documents live in collections, and collections don't enforce a schema: two documents in products can have completely different fields. Don't mistranslate that as "no schema" — the schema exists, it just lives in your application code (plus optional server-side validation rules) instead of in CREATE TABLE. Flexible schema is a loan: cheap to take out when iterating fast, repaid later in defensive code that handles three generations of document shapes.

The design decision that dominates everything else is embed vs reference. A user's shipping addresses (one-to-few, read with the user, bounded) — embed them. A user's orders (one-to-many, queried independently, growing forever) — reference them from their own collection, or group them with the bucket pattern. The classic self-inflicted wound is the unbounded array: comments embedded in a post, events embedded in a device. It demos beautifully, then the hot document grows toward 16 MB, every update rewrites megabytes, and the array's index bloats. If a list can grow without limit, it doesn't belong inside the document.

Why the obsession with getting the boundary right? Because in MongoDB the document is the unit of atomicity. Every single-document write — even one that sets three fields, increments a counter and pushes into an array at once — is atomic, always, no transaction required. Model the data so that each business action touches one document, and you get ACID-feeling behavior for free at document granularity. Multi-document transactions exist (more later), but leaning on them constantly is a sign the model is fighting you.

The one-question modeler: "how does the app read this data?" Read as one thing → one document. Recombined in ways you can't predict at design time → that's a relational workload wearing a hoodie; normalize it in Postgres instead.

2 · Syntax & core API

Nobody writes queries in a system-design round, but naming the operators fluently signals you've actually operated this thing. The API is JavaScript-flavored: filters, updates and pipelines are all documents themselves.

CRUD & queries

insertOne / insertManyWrite documents. _id is auto-filled with an ObjectId — 12 bytes whose first 4 bytes are a timestamp (remember this; it decides a shard's fate later).
find(filter, projection)Rich declarative filters: $eq/$gt/$in/$regex, nested fields via dot-paths, array operators ($elemMatch, $all). Projection trims fields — the seed of a covered query.
updateOne + operators$set, $inc, $push, $pull, $addToSet mutate in place, atomically — one document's worth of changes is all-or-nothing, always.
findOneAndUpdateAtomic read-modify-write returning the doc — the idiom for claiming a job, bumping a counter, optimistic version checks.
aggregate([stages])The aggregation pipeline: documents flow through $match → $group → $sort…. This is Mongo's GROUP BY, its reporting engine, and its join ($lookup).
createIndexSecondary B-tree indexes, same mental model as Postgres: compound, unique, TTL, partial, multikey, text, geo.
explain("executionStats")The truth serum: IXSCAN vs COLLSCAN, totalDocsExamined vs nReturned. A healthy query examines about what it returns.
// find: filter + projection + sort — all declarative documents
db.products.find(
  { category: "laptop", "specs.ram_gb": { $gte: 16 }, price: { $lte: 2000 } },
  { name: 1, price: 1, _id: 0 }            // projection: only what you need
).sort({ price: -1 }).limit(20)

// one document, several changes — atomic WITHOUT a transaction
db.orders.updateOne(
  { _id: orderId, status: "cart" },         // filter doubles as a guard (CAS-style)
  { $set: { status: "paid" }, $inc: { version: 1 },
    $push: { events: { t: new Date(), e: "paid" } } }
)

Aggregation pipeline

// revenue by SKU for July — stages run top to bottom like a unix pipe
db.orders.aggregate([
  { $match: { status: "paid", ts: { $gte: ISODate("2026-07-01") } } },  // filter FIRST — this stage can use indexes
  { $unwind: "$items" },                                    // explode the embedded array
  { $group: { _id: "$items.sku", revenue: { $sum: "$items.price" }, n: { $sum: 1 } } },
  { $sort: { revenue: -1 } }, { $limit: 10 }
])
// $lookup = left outer join — exists, works, and is a smell if it's on every hot path

Indexes worth naming

compound{ userId: 1, ts: -1 } — order matters: equality fields first, then sort, then range (the ESR rule of thumb). Serves any prefix of itself.
multikeyIndex a field that's an array → one index entry per element. Powerful (tags, SKUs), and the reason unbounded arrays bloat storage.
TTL{ createdAt: 1 }, { expireAfterSeconds: 86400 } — a background sweep (runs on the order of once a minute) deletes expired docs. Sessions, tokens, caches.
partialIndex only documents matching a predicate — e.g. only status: "active". Small index, cheap writes for everything else.
text / 2dsphereBasic full-text and geo queries. Real relevance search wants a Lucene engine (Atlas Search / Elasticsearch) — say so before the interviewer does.
hashedIndex on a hash of the field — exists almost entirely to serve hashed shard keys.
covered queryFilter + projection satisfied entirely from the index — zero document fetches. The projection { name: 1, _id: 0 } above is fishing for exactly this.

3 · Use cases — when documents are the answer

🛍 Product catalog — the canonical win
A laptop has CPU, RAM, screen size; a t-shirt has fabric, fit, color; a book has ISBN and page count. In a relational schema this ends as a 300-column table of NULLs or an EAV monstrosity. As documents, each product simply carries its own attributes, with indexes on the fields people actually filter by.
Why Mongo: per-document shape variance is the native case, not a workaround; new product types ship without migrations.
👤 User profiles & preferences
Profile, settings, notification preferences, feature flags, saved addresses — read as one blob on every login, written field-by-field. One document per user, fetched by _id in one round-trip, updated atomically with $set.
Why Mongo: the read is always "the whole aggregate for one user" — the exact shape a document is.
📰 CMS / content platform
Articles with nested blocks (paragraphs, embeds, galleries), drafts whose structure keeps evolving, per-tenant custom fields. Content is a tree; documents store trees natively, and the editor round-trips the same JSON it renders.
Why Mongo: the document is the article; versioned shapes coexist in one collection while the product iterates.
📡 Device events with the bucket pattern
IoT readings or user activity where payload shape varies by device type. Instead of one doc per reading (billions of tiny docs) or one array per device (unbounded!), store one document per device per hour with a bounded array of readings inside.
Why the bucket: bounded growth, ~60× fewer index entries, and reads fetch an hour of data in one document. (Dedicated time-series collections exist too — name them, then move on.)

When it's the WRONG answer

many-to-many + ad-hoc joinsStudents↔courses↔teachers, queried from every direction. Documents force you to pick one nesting; every other access path becomes app-side joins or $lookup chains. Normalize it — Postgres.
cross-entity invariantsDebit one account, credit another, never half — double-entry ledgers, inventory reservations. Multi-doc transactions exist but cost latency and complexity; a relational DB gives this for free.
analytics / OLAP"Scan 2 B rows, group by 3 dimensions" — the aggregation pipeline is not a columnar warehouse. Ship to ClickHouse / BigQuery / Snowflake.
search with relevance$text is a checkbox feature. Typo tolerance, ranking, facets → Lucene (Elasticsearch / OpenSearch / Atlas Search).

4 · Distributed setup — replica sets, then shards

Unit one is the replica set: one primary takes all writes and appends them to the oplog (a capped collection of idempotent operations); secondaries tail the oplog and replay it. If the primary dies, the survivors hold an election (a Raft-style majority vote; with default timeouts failover completes on the order of ~10 seconds) and a secondary promotes itself. Clients use the same connection string throughout — drivers discover the new primary and retry. A production set is 3 or 5 data-bearing members; you can stretch to 50 members but only 7 vote.

What makes MongoDB interesting in interviews is that consistency is tunable per operation, on three dials. writeConcern: w:1 acks after the primary alone; w:"majority" waits for a majority to persist it (the default since 5.0). readConcern: local reads whatever the node has; majority reads only majority-committed data; linearizable gives real-time guarantees on the primary at real latency cost. readPreference: primary by default, or offload to secondaries — accepting that secondaries are asynchronous replicas, so a user who just wrote may read the past. The senior move is naming which dial each endpoint gets: money path w:majority / readConcern:majority, activity feed w:1 + secondary reads.

The classic gotcha — w:1 + failover = silent un-write. Primary acks your write at w:1, dies before any secondary copied it; a new primary is elected without it. When the old primary rejoins, it rolls back the acked write (dumped to a rollback file for a human to contemplate). The user saw "saved"; the database disagrees. If losing it hurts, the answer is w:"majority" — you pay one replication round-trip to make the write election-proof.

When one primary can't carry the write rate or the working set, you shard: data splits across multiple replica sets (shards) by ranges of a shard key. Stateless mongos routers front the cluster, and a config server replica set stores the routing table: which chunk (contiguous key range, default max size on the order of 64–128 MB depending on version) lives on which shard. A balancer splits and migrates chunks to keep shards even. Every query now has two possible shapes: filter contains the shard key → targeted to one shard; otherwise → scatter-gather to all shards, where p99 latency is the slowest shard's answer.

And so the whole game compresses into one decision: the shard key. A ranged key preserves order (great for range scans, targeted lookups) but a monotonically increasing key — like the default ObjectId with its timestamp prefix — aims every insert at the single chunk at the end of the range: one shard burns while the rest idle. A hashed key spreads writes evenly but destroys order: every range query becomes scatter-gather. Resharding online exists (5.0+) but rewrites the whole collection in the background over hours — which is why the folklore says you marry your shard key.   2. Run the queries: _id lookup is targeted, the 1-hour range is targeted onto the hot shard, the user query scatters to all four.

  5. Notice every key switch wiped the cluster: in production that's reshardCollection — hours of background rewriting. Choose once, choose well.
🔍 Deep dive · Embed vs reference — the patterns that make or break the model click to expand

The honest rules are boring: embed for one-to-few, data owned by the parent, read together, bounded (addresses in a user, line items in an order). Reference for one-to-many and anything queried on its own (orders out of users, comments out of posts). The interesting part is the middle, where the named patterns live.

Extended reference — reference, but copy the 2–3 fields you always need into the referrer: the order stores customerId plus customerName and city, so the orders list renders with zero joins. You've bought read speed with a denormalization debt: names change rarely, and when they do, a background pass fixes history. Saying "I denormalize deliberately and I know the update cost" is precisely the signal interviewers fish for.

Bucket — for streams: one document per device per hour, readings appended into a bounded array with a count field; when full, roll to a new bucket. Turns billions of tiny docs into millions of medium ones, collapses index size, matches the read ("last hour of sensor 7"). Outlier — model for the 99%: a post document embeds its likes, but the 0.1% of viral posts set an overflow: true flag and spill extra likes to a side collection, so Bieber doesn't dictate schema for everyone. Subset — embed the newest 10 comments for first paint, keep the full history referenced.

The anti-patterns are the same list inverted: unbounded arrays (any embedded list without a cap), 16 MB roulette (documents that only usually fit), and relational cosplay — fully normalized collections joined by $lookup on every read, which combines the guarantees of neither world. And for blobs: GridFS chunks files into ~255 KB pieces inside the DB, but in a system-design answer, large binaries go to object storage and the document keeps the URL.

🔍 Deep dive · WiredTiger — what actually happens on disk click to expand

Since 3.2 the default storage engine is WiredTiger, and it quietly retired MongoDB's worst old reputation. Writers get document-level concurrency: two updates to different documents in one collection proceed in parallel (the ancient engine locked whole databases, then collections). Under the hood it's MVCC — readers see a consistent snapshot while writers work on copies; conflicting writers to the same document retry via optimistic concurrency rather than blocking.

Durability is the standard WAL recipe wearing different names. Writes land in cache and in the journal (write-ahead log), which is fsynced on the order of every 100 ms — and immediately for writes demanding j: true, which w:"majority" implies by default. Every ~60 s WiredTiger writes a checkpoint, a clean consistent image of the data files; crash recovery is "load the last checkpoint, replay the journal since". Lose power at the wrong instant with w:1 and no journal ack, and the last handful of milliseconds is gone — one more reason the money path runs at majority.

Two numbers worth carrying: the WiredTiger cache defaults to roughly half of (RAM − 1 GB), and it's a document cache — your working set (hot docs + indexes) should fit inside or latency falls off a cliff; and data is compressed on disk — Snappy by default (fast, ~2–4× on JSON-ish data), zstd/zlib if you trade CPU for space, prefix compression for indexes. BSON's repeated field names compress beautifully, which is why "documents waste space on keys" is mostly an obsolete complaint.

🔍 Deep dive · Replication mechanics — oplog, elections & the rollback story click to expand

The oplog is a capped collection (local.oplog.rs) of idempotent operations — not the statements you ran, but their effects (an $inc is logged as the resulting $set-like op), so replaying twice is safe. Secondaries tail it like a change feed and apply ops in batches. Its size (default ~5% of free disk, within roughly a 1–50 GB window) is your replication headroom: a secondary that falls behind by more than the oplog window can't catch up incrementally and must do a full resync. Watch replication lag like a fire alarm — it's also your staleness bound for secondary reads and the data-loss bound for w:1.

Elections are Raft-flavored: terms, majority votes, and a same-term guarantee that only one primary exists. The subtlety interviewers love: during a network partition the old primary may not yet know it's deposed — for a few seconds a stale primary can still serve readConcern: local reads on the minority side. That's why linearizable reads exist (they confirm a majority round-trip before answering) and why "MongoDB is CP-ish, but only if you pay at both the write and read counter" is the accurate summary.

After a failover, the deposed primary compares its oplog with the new primary's, finds the divergence point, and rolls back everything after it — those ops are written to rollback files, not merged. Reads at readConcern: majority never see such doomed writes, which is the practical reason that dial exists. One architecture note that scores points: a PSA set (primary–secondary–arbiter, where the arbiter votes but stores nothing) keeps electing fine when a data node dies, but w:"majority" writes need a majority of data-bearing members — with one of two data nodes down they stall. Arbiters are a budget trick with a sharp edge.

🔍 Deep dive · Transactions — real ACID, and why reaching for it is a smell click to expand

Since 4.0 MongoDB has genuine multi-document ACID transactions on replica sets, and since 4.2 across shards (a two-phase commit coordinated over the participating shards). Snapshot isolation, all-or-nothing, the works. So why does every experienced Mongo engineer wince slightly when they appear?

Because of what they cost. A transaction holds a snapshot open and buffers its writes; long-running ones pin WiredTiger cache and abort at a default lifetime of 60 seconds. Documents touched inside are locked against concurrent writers, so hot documents turn transactions into a conflict-retry carnival. Cross-shard transactions add coordinator round-trips to commit latency. None of this is broken — it's the honest price of coordination, the same price Spanner or Postgres pays — but MongoDB's whole value proposition was designing so you don't need it: the document is supposed to BE the transaction boundary.

So the interview framing: single-document atomicity covers the vast majority of well-modeled operations, including read-modify-write via findOneAndUpdate. Multi-document transactions are the escape hatch for the genuinely cross-entity cases — and if your design invokes them on every request, that's not a Mongo feature you're using, that's a relational database you're missing. Mention retryable writes too (modern drivers retry a failed write exactly once, safely, across failovers — on by default): it kills a whole class of "but what if the ack was lost" follow-ups.

⚖️ At scale — numbers, limits & failure modes

Order-of-magnitude figures for capacity talk — round aggressively out loud.

document≤ 16 MB BSON, ≤ 100 nesting levels. Blobs → object storage + URL (GridFS exists; rarely the right answer).
replica set≤ 50 members, ≤ 7 voting; failover on the order of ~10 s with default timeouts. Oplog default ~5% of free disk — its window is your lag/resync budget.
WiredTigerCache ≈ 50% of (RAM − 1 GB); journal fsync ~100 ms cadence; checkpoints ~60 s; Snappy compression by default. Working set must fit cache.
throughput feelA well-tuned replica set on solid hardware sustains on the order of tens of thousands of simple writes/s; shard when the write rate, working set, or storage outgrows one primary — not before.
shardingChunk max size on the order of 64–128 MB (version-dependent). Balancer migrations cost I/O — schedule windows. Clusters run from 2 to hundreds of shards; config servers are their own replica set.
aggregation~100 MB per pipeline stage in RAM before it needs allowDiskUse; $lookup fan-out multiplies reads. Heavy analytics belongs elsewhere.
transactionsDefault lifetime 60 s; cross-shard = 2PC latency. Keep them rare, short, and away from hot documents.
Failure modes that only appear at scale: the hot shard (monotonic shard key → every insert on one shard; the cluster scales reads while writes melt one box); jumbo chunks (one over-popular key value makes a chunk unsplittable and unmovable — the balancer routes around it forever); scatter-gather amplification (p99 of a fan-out query = the slowest shard; one degraded shard poisons every non-targeted query); unbounded array growth (documents creep toward 16 MB, updates rewrite megabytes, multikey indexes bloat); secondary reads sold as fresh (replication lag turns "read your own write" into a support ticket); PSA majority stall (arbiter + one dead data node = w:majority writes hang).

8 · In the interview

How it comes up: you propose it — "product catalog with wildly varying attributes; I'll store each product as a document" — or you defend against it: "why MongoDB and not Postgres?" The follow-up script is stable: how do you model X? (embed vs reference, out loud), what's atomic? (the document; transactions exist but are the exception), what happens on failover? (election ~10 s, w:1 rollback risk, majority as the fix), how does it scale? (shard key choice, targeted vs scatter-gather, hot-shard story).

The choice script: default to Postgres; move to MongoDB when the data is naturally aggregate-shaped and heterogeneous (catalog, profiles, CMS, flexible events), reads fetch whole entities by key, and you want built-in replica-set failover and sharding rather than bolting them onto an RDBMS. Handle the JSONB retort honestly: for one node's worth of data with occasional JSON fields, Postgres JSONB wins — relational spine plus document flexibility. MongoDB earns its keep when the document model is the primary model and horizontal scale-out is on the roadmap, not a hypothetical. Then close the loop: "if requirements drift toward many-to-many joins and cross-entity invariants everywhere, that's my signal the document bet was wrong."

Alternatives — one line each

Postgres JSONBThe retort to have an answer for: relational + documents in one engine, real FKs and transactions. Single-node ceiling and manual sharding are what MongoDB answers back with.
DynamoDBManaged KV/document with hard access-pattern discipline and per-request pricing; no rich ad-hoc queries or server-side aggregation. Choose for known access patterns at AWS scale; Mongo for query flexibility.
CassandraWrite-optimized wide-column, leaderless, linear scale; query-first modeling with no joins and limited filtering. Telemetry firehoses, not flexible catalogs.
CouchbaseDocument store with memcached DNA and SQL-ish N1QL; strong in caching-adjacent and mobile-sync niches.

Phrases that score: "the document is my transaction boundary — that's a modeling decision, not luck" · "w:majority and readConcern majority on the money path; w:1 and secondary reads where staleness is priced in" · "shard key is userId because every query is per-user; ObjectId would funnel all inserts to one shard — its first four bytes are a timestamp" · "that array is unbounded, so I'll bucket it" · "if we're reaching for $lookup and transactions on every request, the model is fighting us and Postgres is the honest answer".

9 · Check yourself

You embedded comments in each post document. Six months later the app is slow. What happened, and what's the fix?

Popular posts grew unbounded arrays: documents creep toward the 16 MB cap, every new comment rewrites an ever-larger document, the multikey index on comments bloats, and hot documents evict everything else from cache. Fix: move comments to their own collection keyed by postId (reference), optionally keep the newest handful embedded for first paint (subset pattern), or bucket comments in pages. Rule: if it grows without limit, it doesn't live inside the document.

A write was acknowledged with w:1; the primary crashed a moment later. The user's data is gone. Walk through why — and the two-dial fix.

With w:1 the primary acked before any secondary replicated the write. The election picked a secondary that never saw it; when the old primary rejoined it rolled back the divergent write into a rollback file. Fix dial one: w:"majority" — the ack now means a majority has it, so any electable node includes it. Dial two: readConcern:"majority" so readers can never observe data that might roll back. Cost: one replication round-trip of latency — cheap insurance on anything that matters.

You sharded on the default _id (ObjectId) and one shard is on fire while three idle. Why, and what are your options?

ObjectId's first 4 bytes are a timestamp, so keys are monotonically increasing — every insert targets the single chunk at the top of the range, i.e. one shard. The balancer migrates old chunks away but the write frontier stays put. Options: hashed(_id) for even writes (losing range targeting), a compound key like {tenantId, ts} that distributes by a high-cardinality prefix while keeping per-tenant locality, or reshardCollection (5.0+) to migrate online — at the cost of hours of background rewriting. This is why you interview the shard key harder than you interview candidates.

The interviewer asks: "MongoDB has transactions now, so it's just as good as Postgres for our banking ledger, right?"

Push back gently: yes, 4.0+/4.2+ transactions are real ACID (snapshot isolation, cross-shard 2PC). But they carry a 60 s default lifetime, cache pressure, conflict-retry loops on hot documents, and extra commit latency across shards — and if every operation needs multi-document invariants, you're paying Mongo's costs to emulate what a relational database does natively, with none of the document model's payoff. Documents win when the aggregate is the transaction boundary; ledgers are the textbook case where it isn't.

Your read replicas (secondaries) serve the profile page. Users complain they edit their name and still see the old one. What do you tune?

Secondary reads are asynchronous — replication lag makes read-your-own-write fail. Options in increasing strength: route the post-write read to the primary (readPreference), use a causally consistent session so reads wait for their own writes, or drop secondary reads for this endpoint entirely. Keep secondary reads for content where staleness is explicitly acceptable (feeds, counts), and monitor replication lag as the staleness bound you're selling.
