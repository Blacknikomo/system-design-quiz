⚡ DynamoDB — partitions, consistency & GSIs

Managed key-value / wide-column store. Watch how the partition key hashes to a partition, how a leader replica acks writes before followers catch up, and why an eventually-consistent read can be stale while a strongly-consistent one isn't.
partition key → hash leader + followers eventual by default RCU / WCU billing

Every item's partition key is hashed to choose a partition. A write goes to that partition's leader replica, which acks the client, then asynchronously replicates to followers. A strongly-consistent read hits the leader (always latest, 1 RCU); an eventually-consistent read may hit a lagging follower (possibly stale, 0.5 RCU).

Signature gotchas: item size limit is 400KB. ProjectionExpression only saves network bandwidth, not RCU — you still pay for the full item size read. A hot partition key throttles (per-partition ceiling ≈ 3000 RCU / 1000 WCU). And strongly-consistent reads are NOT available on GSIs.

Run an operation

Run PutItem: the partition key is hashed to pick a partition, the write lands on that partition's leader, then replicates to followers. Pause replication, PutItem a new value, then GetItem to compare eventual vs strong reads.

What you're seeing

  1. PutItem: hash(pk) % 3 picks the partition; the write hits its leader (blue).
  2. Leader acks the client (1 WCU/1KB), then async-replicates to followers (purple).
  3. GetItem · eventual may read a stale follower (0.5 RCU).
  4. GetItem · strong reads the leader → always latest (1 RCU = 2×).
  5. A GSI re-keys items by email on its own partitions — eventual only, no strong reads.
LSI vs GSI: a Local Secondary Index shares the table's partition key (co-located on the same partitions), supports strong reads, and must be defined at table creation. A Global Secondary Index uses a different key, lives on separate partitions, is eventually consistent only, and can be added anytime.
