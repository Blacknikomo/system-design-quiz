📦 Compute & Orchestration — VMs, containers, K8s, serverless

From hypervisors to Firecracker: the isolation ladder, the Kubernetes control loop, rollouts that don't page you — and when the right answer is "just use Cloud Run". ☕ ~18 min read
isolation ladder namespaces + cgroups control loop readiness gates traffic canary Firecracker

Every compute platform is an answer to one question: how thick a wall do you build around a process? Thick walls (VMs) are safe but slow and expensive; thin walls (containers) start in milliseconds and pack densely but share a kernel; Firecracker microVMs split the difference, and serverless rents you the wall by the millisecond. Kubernetes then adds the second big idea: don't place workloads by hand — declare what should exist, and let a control loop make it true, forever.

In the interview: compute choice is usually one sentence — "stateless services in containers behind a load balancer, autoscaled on CPU/QPS" — and the minutes belong to your data tier. But the follow-ups are predictable and this page arms exactly them: how do you deploy without downtime? (readiness gates + canary), what happens when a machine dies? (the control loop), Lambda or containers? (duty cycle), and the EM special: what does adopting Kubernetes cost in headcount?

1 · The mental model: the isolation ladder

Think of compute as housing. Bare metal is a detached house: everything on the lot is yours, nobody shares your plumbing, and moving in takes weeks. A virtual machine is an apartment: the hypervisor is the concrete between units, and every apartment has its own plumbing — its own kernel. A container is a room in a shared flat: your door (namespaces) makes the flat look like it's all yours, the house rules (cgroups) cap how much hot water you may use, but there is one plumbing stack for everyone — the host kernel. You can move in with a suitcase in minutes, and you can hear the neighbors through the walls. A microVM is a capsule hotel: real walls, but the capsule is prefabricated and craned into place in a tenth of a second. And a serverless function isn't housing at all — it's renting a desk by the minute in a building someone else owns, patches, and guards.

Now the same ladder in engineering terms. A VM runs on a hypervisor that presents virtual hardware; the guest boots its own operating system kernel. That's why the isolation is strong — the attack surface between tenants is the narrow hypervisor interface, not the sprawling syscall API — and why the unit is heavy: gigabytes of memory per guest, boot times measured in tens of seconds, density measured in tens per host.

A container is not a small VM. It is a normal Linux process wearing blinkers. Namespaces limit what it can see — pid, net, mnt, uts, ipc, user (plus cgroup and time namespaces in modern kernels), each one hiding a global resource so the process believes it owns the machine. Cgroups limit what it can use — CPU shares and quotas, a memory ceiling enforced by the OOM killer. There is no guest kernel to boot, so starting a container is just starting a process: milliseconds, hundreds per host. The price of that density is the boundary itself: every container on a host is one kernel vulnerability away from every other. The wall is a syscall filter, not silicon.

For years the industry treated this as a binary choice — safe-but-slow or fast-but-shared. Firecracker broke the dichotomy: a minimal KVM-based virtual machine monitor that throws away legacy device emulation (no BIOS-era baggage, just virtio net/block and a serial console) and boots a real VM with its own kernel in ~100–125 ms, with memory overhead small enough to run thousands of microVMs on one host. This is what AWS Lambda and Fargate actually run your code in — VM-grade walls at nearly container prices. Google plays the same game differently with gVisor, a user-space kernel that intercepts syscalls (stronger boundary than raw containers, some performance tax).

The ladder, with order-of-magnitude numbers worth remembering:

bare metalProvisioning in minutes-to-days · one tenant per host · isolation: physical. Still real: latency-critical trading, licensing, GPUs you own.
VMBoot ~tens of seconds · ~tens per host · own kernel per guest, hypervisor boundary. The unit clouds actually sell you (EC2, GCE).
containerStart in ~ms (plus image pull!) · hundreds per host · shared host kernel, namespaces + cgroups boundary.
microVMBoot ~100 ms-order (Firecracker) · thousands per host · own kernel, minimal device model — VM isolation at near-container speed.
functionWarm invoke ~instant, cold start ~100 ms → seconds · density is the platform's problem · one microVM per concurrent execution (Lambda).
The one-sentence rule: pick the weakest isolation you can defend. Your own trusted code sharing a kernel is fine — use containers and enjoy the density. The moment you run other people's code (multi-tenant platforms, CI runners, plugin sandboxes, AI-generated code execution), containers alone are not a security boundary you should bet the company on — you want VM-grade walls, which today means microVMs.

2 · The vocabulary — Docker & the Kubernetes objects

Nobody will ask you to write YAML in a design round, but fluent vocabulary signals operational scar tissue. Two dialects matter: Docker (packaging) and Kubernetes (orchestration).

Docker in six ideas

An image is a stack of immutable, content-addressed layers merged by an overlay filesystem; at runtime the container gets one thin writable layer on top. Because layers are addressed by digest, registries store and ship each layer once — twenty services on the same base image share it. And because the image carries the entire filesystem, "works on my machine" dies: the machine ships with the code.

docker build -t web:9f2c .Assemble an image from a Dockerfile. Every instruction produces a cacheable layer — order determines rebuild speed (see below).
docker run -p 8080:80 --memory 512m --cpus 1Start a container — note the flags are just cgroup knobs exposed politely.
docker push / pull registry/web@sha256:9f2c…Move images via a registry. Tags move, digests don't — production should pin digests.
docker history web:9f2cShow the layer stack and per-layer sizes — the first thing to run when an image is mysteriously 2 GB.
docker exec -it api bashShell into a running container — debugging 101.
docker compose upA local multi-service topology from one YAML file — the standard dev-environment answer.
# Layer order IS cache strategy: stable lines first, volatile lines last.
FROM node:20-slim AS build          # base layers — shared by every service on this base
WORKDIR /app
COPY package*.json ./               # the dependency manifest only — changes rarely
RUN npm ci --omit=dev               # fat layer, rebuilt ONLY when the manifest changes
COPY . .                            # your code — changes every commit, busts cache from HERE down

FROM gcr.io/distroless/nodejs20     # multi-stage: runtime image with no shell, no toolchain
COPY --from=build /app /app
CMD ["/app/server.js"]              # small image = fast pulls = fast scale-out and rollbacks

If you invert the order — COPY . . before npm ci — every one-line code change re-downloads every dependency. Teams lose minutes per build for months before someone notices. That's the whole game: deps before code.

Kubernetes — the objects that matter in design talk

PodThe smallest schedulable unit: one or more containers co-scheduled on the same node, sharing a network namespace (they talk over localhost) and volumes. Sidecars live here.
Deployment → ReplicaSet"Keep N replicas of this pod template running, and roll updates safely." The workhorse — 90% of what you'll ever declare.
ServiceA stable virtual IP + DNS name load-balancing across an ever-churning set of pods. Solves service discovery: pods are cattle, the Service is the address.
Ingress / GatewayL7 HTTP entry into the cluster: host/path routing, TLS termination. Backed by a real proxy (nginx, Envoy, cloud LB).
ConfigMap / SecretConfig injected as env vars or files. Gotcha worth saying out loud: a Secret is base64-encoded, not encrypted, unless you turn on encryption at rest / external KMS.
StatefulSetPods with stable identities (db-0, db-1, db-2) and per-pod persistent volumes — for quorum systems that care who's who. Necessary but not sufficient for databases (see the deep dives for why you still usually buy the managed one).
DaemonSetExactly one pod per node: log shippers, metrics agents, CNI plumbing.
Job / CronJobRun-to-completion batch work, optionally on a schedule, with retries tracked by the cluster.
HPAHorizontalPodAutoscaler — scale a Deployment's replica count on CPU or custom metrics (details in the autoscaling deep dive).
apiVersion: apps/v1
kind: Deployment                              # a statement of DESIRED state, not instructions
spec:
  replicas: 6
  strategy:
    rollingUpdate: { maxSurge: 1, maxUnavailable: 0 }   # never dip below 6 Ready pods
  template:
    spec:
      containers:
      - name: web
        image: registry/web@sha256:9f2c…      # digest pin, not :latest
        resources:
          requests: { cpu: 250m, memory: 256Mi }  # what the SCHEDULER reserves — bin-packing input
          limits:   { memory: 512Mi }             # cross this → OOMKill (exit 137). CPU limit omitted:
                                                  # CPU over-use is throttled, memory over-use is fatal
        readinessProbe:                       # gates TRAFFIC — the deploy-safety mechanism
          httpGet: { path: /healthz, port: 8080 }
        livenessProbe:                        # gates RESTARTS — use sparingly
          httpGet: { path: /livez, port: 8080 }

Two distinctions here carry disproportionate interview weight. First, requests vs limits: requests are the scheduler's booking (a pod only lands where its requests fit), limits are the cgroup ceiling at runtime. Blow past a memory limit and the kernel kills the container — exit code 137, the infamous OOMKill; blow past a CPU limit and you're merely throttled. Second, readiness vs liveness: a failing readiness probe removes the pod from the Service's endpoints — no traffic, pod keeps running; a failing liveness probe restarts the container. Readiness is what makes deploys safe. Liveness, misconfigured, turns a slow downstream dependency into a cluster-wide restart storm — the classic self-inflicted outage.

3 · Use cases — matching workloads to rungs

🌐 The default: a fleet of stateless services
Deployment + Service + HPA on a managed cluster (GKE/EKS) or on ECS/Cloud Run. The image is the deploy artifact, nodes are cattle, self-healing and bin-packing come free.
Why containers: one packaging standard for every language, high density on shared nodes, and the rollout machinery (probes, surge, canary) is where deploy safety lives.
⚡ Event glue & spiky async work — Lambda
S3 upload → thumbnail; webhook receivers; a cron that runs 40 seconds a day; queue consumers with wildly variable depth. Zero idle cost, per-ms billing, scaling handled by the platform.
Why serverless: when duty cycle is low or spikes are unpredictable, paying only for execution beats keeping warm capacity — and there's nothing to patch on Sunday.
🚀 A product team with no platform team — Cloud Run / Fargate / App Runner
Ship a container, get HTTPS, autoscaling (including to zero on Cloud Run), and rollbacks — with no cluster to feed. The compute choice that doesn't require hiring.
Why managed containers: you keep the container abstraction (portable, testable locally) and delete the orchestration ops. For most companies this is the honest default.
🧪 Batch, cron & ML jobs — the scheduler earns its keep
Nightly reports as CronJobs, training runs as Jobs on GPU node pools, opportunistic work on spot/preemptible instances. Kubernetes shines exactly when workloads are many, lumpy and heterogeneous — bin-packing them onto shared metal is real money.
Why K8s here: requests/limits give the scheduler honest sizes to pack; priorities and preemption let batch soak up whatever the online services aren't using.

When it's the WRONG answer

K8s for 3 services, 5 engineersThe cluster becomes your most complex component: version upgrades ~3×/year, CNI/ingress/secrets choices, node patching. The ops tax exceeds the product. Cloud Run/Fargate first; earn your way into K8s.
Lambda for steady 24/7 loadPer-ms pricing is renting by the minute what you use all day — at high, flat utilization always-on containers are typically far cheaper, and cold starts poison strict p99.
Lambda for long-lived connections15-minute execution cap and a one-request-per-instance model don't fit WebSockets/streaming. You end up storing connection state elsewhere and fighting the platform.
your database on K8s, hand-rolledA StatefulSet gives stable names and volumes — it does not give you failover judgment, backup discipline, or upgrade orchestration. That's a DBA team encoded in software; buy it managed (RDS, Cloud SQL, Spanner) unless you have strong reasons.
a service mesh at 4 servicesmTLS and retries for four services is a shared HTTP client library, not a fleet of Envoy sidecars. Meshes are platform-team technology (see §4).

4 · Distributed setup — Kubernetes is a while-loop around a database

Strip away the YAML and Kubernetes is one idea, repeated: desired state lives in a database; controllers run an infinite loop making reality match it. You never tell the cluster to do something; you tell it what should exist. replicas: 6 is not a command to start six pods — it's a fact about the world, and somewhere a loop wakes up, counts five, and files the difference as work.

The anatomy follows from that. The API server is the only door — every component, human or robot, talks through it. Behind it sits etcd, a key-value store replicated with Raft consensus, holding the entire desired and observed state of the cluster (for the consensus intuition — leaders, quorums, why odd numbers — see the ZooKeeper/coordination page; same family of ideas). The controller manager runs the loops: the Deployment controller reconciles ReplicaSets, the ReplicaSet controller reconciles pod counts, the node controller watches heartbeats. The scheduler is just another loop, one that watches for pods with no node assigned, filters nodes where the pod's requests fit (plus affinity/taint rules), scores the survivors, and writes the placement back. On every node a kubelet reads what it's supposed to run, makes it real through the container runtime, and reports status back up. Nobody talks to anybody directly; everyone reads and writes the shared database. It's blackboard architecture, and it's why the system is so composable.

Watch it self-heal, because this is the signature demo. A node loses power. Its kubelet stops heartbeating; after a grace period (order of ~40 s with defaults) the node controller marks it NotReady, and after a toleration window (~5 min by default) the pods on it are marked for eviction. The ReplicaSet controller — which knows nothing about nodes or power supplies — simply observes five pods where the desired state says six and creates one. The scheduler — which knows nothing about the outage — sees an unassigned pod and places it. A kubelet elsewhere starts it. Nobody handled the failure. There is no failure-handling code path; recovery and normal operation are the same loop. That's what "declarative" buys, and it's the honest one-line answer to "why Kubernetes over a fleet of VMs with scripts".

Two consequences are worth saying in an interview. First, the loop fights you too: hand-edit a pod that a ReplicaSet owns and the controller will calmly revert or replace it — which is exactly why GitOps works (a repo holds desired state; Argo CD is just one more reconciler, syncing cluster ← git). Second, a control-plane outage is not a data-plane outage: if the API server or etcd goes down, running pods keep serving traffic — what you lose is the ability to change anything: deploy, scale, or heal. That asymmetry is why managed control planes (EKS/GKE/AKS) are such an easy buy: the cloud runs the API server and etcd with an SLA, and you still own everything above.

Once you have dozens of services on the cluster, the next temptation is a service mesh: an Envoy proxy next to every pod (sidecar — or node-level in ambient designs), giving you mTLS everywhere, retries/timeouts/circuit-breaking as config, percentage traffic-splits, and uniform telemetry without touching application code. The costs are just as real: a fleet of proxies to version and debug, a control plane to run, and ~ms-order added latency per hop that a p99 budget will notice. The honest adoption bar: many services, real zero-trust or traffic-shaping requirements, and a platform team to own it. Below that bar, a good shared client library does 80% of it.

Gotchas that separate readers from operators: (1) etcd is the blast radius — it's a config store, not a database; keep objects small and counts sane (practical ceiling ~single-digit GB), because everything watches it and it Raft-replicates every write. (2) Mass rescheduling is a storm: a zone outage doesn't just move pods — hundreds of them land on surviving nodes at once and all pull images simultaneously; the registry becomes your hidden single point of failure. (3) One giant cluster is an anti-pattern: well before the documented limits, orgs shard clusters by region/environment/team — for blast radius and because upgrading a cluster everyone shares is terrifying.

5 · 🎛 Play with it — the rollout safety simulator

A Deployment of 6 pods on v1 serves a steady 120 req/s. You ship v2 — which may be broken. Four strategies: rolling surge (add 1 new, then remove 1 old — never below 6 Ready), rolling maxUnavailable=2 (remove first, cheaper, dips capacity), blue-green (build a full parallel stack, flip all traffic atomically), and canary (1 pod + an automated metrics gate). The readiness probe toggle decides whether broken pods ever receive traffic. Each ▶ Step is one reconcile-loop iteration plus one interval of live traffic — watch the error meter and the failed-request counter.

Event log

What to try

  1. Leave the defaults (surge rolling · v2 broken · probe ON) and press Step repeatedly: the v2 pod never becomes Ready, so no v1 pod is ever removed. The rollout stalls at 0% errors — the stall is the safety.
  2. Reset, uncheck readiness probe, same rollout: every step ships another broken pod straight into the load balancer — watch the error meter climb to 100%. Gradual ≠ safe without gates.
  3. Try maxUnavailable=2 with the probe ON: still 0% errors, but only 4/6 pods serving — the probe saved correctness, the strategy still cost capacity (p99 pays).
  4. Blue-green, probe OFF, v2 broken: the cutover step is an instant 100% outage — then one more step flips back. Total blast radius: one full interval of everything. With the probe ON, the cutover is simply refused.
  5. Canary, probe OFF, v2 broken: one pod takes ~14% of traffic for one interval, then the metrics gate auto-halts and rolls back. Compare the failed requests counter across runs 2, 4 and 5.
  6. Uncheck "v2 is broken" and run any strategy end-to-end. Note what blue-green cost you: 12 pods live during the window.
🔍 Deep dive · Rollouts & release safety — surge, blue-green, canary, flags click to expand

A rolling update has exactly two knobs. maxSurge: how many extra pods may exist during the roll (cost of the roll); maxUnavailable: how many pods below the desired count you tolerate (capacity risk of the roll). Kubernetes defaults both to 25%; the conservative production shape for latency-sensitive services is maxSurge: 1, maxUnavailable: 0 — pay for one extra pod, never dip below full strength. Progress is gated by readiness: a new pod that never turns Ready blocks the roll, and after progressDeadlineSeconds (default 600) the Deployment is marked stalled. Note what that means culturally: with no readiness probe there is no gate — the platform will cheerfully replace a working fleet with a broken one, one pod at a time.

Blue-green refuses gradualism: build the complete v2 stack next to v1, verify it (smoke tests against the idle stack — a luxury rolling can't offer), then flip the router atomically. Rollback is the same flip in reverse, near-instant, because the old stack is still warm — that's what the 2× capacity bill during the window buys. The catch people miss: the cutover is atomic per request, not per system — background jobs, queue consumers and in-flight sessions straddle the flip, so v1 and v2 still coexist in time.

Canary is the grown-up compromise: send a small slice — one pod, or 1–5% via traffic-split — watch real metrics (error rate, p99, business KPIs) against the baseline, and only then promote in stages. The senior move is making the judgment automated: tools like Argo Rollouts or Flagger run the metric queries and halt/rollback without a human — a tired human staring at Grafana at 2am is not a gate. Canary's blind spot: a 1% canary needs meaningful traffic to produce statistically meaningful errors, so low-QPS services canary slowly or not at all.

Feature flags complete the picture by decoupling deploy from release: ship the code dark, then turn it on per-cohort at runtime — and off again in seconds when it misbehaves, with no rollout at all. Flags are canaries at the code-path level, and the fastest rollback mechanism you will ever own. Their tax is real too: flag debt, combinatorial testing, and the day someone deletes the wrong one.

Finally, the constraint every strategy shares: during any rollout, old and new run simultaneously — so schema and API changes must be backward-compatible across one version. The discipline is expand → migrate → contract: add the new column/endpoint, ship code that works with both, backfill, then remove the old — each phase a separate, individually-rollback-able deploy. Most "the rollback made it worse" stories are a violation of exactly this.

🔍 Deep dive · What a container actually is — namespaces, cgroups, overlayfs click to expand

You can build a crude container with two shell commands, and doing it once permanently demystifies Docker:

# a "container" without Docker (root shell, cgroup v2)
unshare --fork --pid --mount-proc bash   # new pid namespace…
ps aux                                   # …your shell is PID 1; the host's processes are gone

# and a resource wall: cap this shell's subtree at 256 MiB
mkdir /sys/fs/cgroup/demo
echo $$ > /sys/fs/cgroup/demo/cgroup.procs
echo 268435456 > /sys/fs/cgroup/demo/memory.max   # exceed it → the OOM killer visits (exit 137)

That's the whole trick. Namespaces partition visibility: pid (process tree), net (interfaces, ports — why two containers can both bind :8080), mnt (filesystem tree), user (root inside ≠ root outside — the single best hardening step), uts (hostname), ipc. Cgroups partition consumption: cpu weight and quota, memory ceilings. Docker's contribution was never the isolation — the kernel had it — but the packaging: the image format, layer caching and registries that made environments shippable.

Layers, concretely: each Dockerfile instruction produces a read-only filesystem diff; overlayfs stacks them into one apparent tree, plus a per-container writable layer on top (copy-on-write on first modification). Layers are content-addressed by digest, so hosts and registries deduplicate ruthlessly — which is why the second pull of a service on a node is fast, and why mass-rescheduling onto fresh nodes (nothing cached) is an image-pull storm.

Now the security honesty. Every container shares the host kernel, and the syscall interface is enormous — one exploitable kernel bug is a container escape. Defaults help (seccomp filters the syscall list, AppArmor/SELinux confine, capabilities are dropped), and user namespaces help more, but the industry's actual answer for hostile multi-tenancy is to change the boundary: gVisor interposes a user-space kernel between the app and the host (huge attack-surface reduction, noticeable syscall overhead), Kata Containers and Firecracker put each workload in a real (micro)VM with its own kernel. Firecracker's numbers — ~100–125 ms boot, memory overhead per microVM in the single-digit-MB order, thousands per host — are what let AWS run Lambda as one microVM per concurrent request stream and still make money. The ladder from §1, closed into a loop: serverless is VMs again, just very small and very fast.

🔍 Deep dive · Autoscaling — HPA, VPA, cluster autoscaler, and why headroom survives them all click to expand

There are three autoscalers and they answer different questions. HPA (horizontal): more pods? It watches a metric — CPU utilization against requests, or custom ones like QPS/queue depth — and sets the replica count to roughly ceil(current × actual/target) on a ~15 s control loop, with a stabilization window (~5 min by default on scale-down) so it doesn't flap. VPA (vertical): bigger pods? It observes real usage and adjusts requests — classically by restarting pods, which is why many teams run it in recommendation mode and apply the numbers by hand. Cluster Autoscaler / Karpenter: more nodes? When pods sit unschedulable because no node fits their requests, it provisions machines — and machines take minutes: boot, join, warm the image cache.

Stack the lags and you get the number that matters: from "traffic doubled" to "capacity doubled" is ~30–60 s if pods fit on existing nodes, and 3–5+ minutes if nodes must be added. A flash-crowd spike is faster than that. So the uncomfortable truth: autoscaling tracks daily curves; headroom absorbs spikes. Serious services run at 50–70% of capacity on purpose, and/or put a queue in front so the spike becomes backlog instead of 5xx. Saying "HPA on CPU at 60% target, plus ~40% headroom because node scale-up is minutes" is a complete, senior answer.

Two refinements worth naming. Scale on the right metric: CPU is a proxy; queue depth or in-flight requests per pod track user pain directly — KEDA packages event-driven scaling (including to zero) for queue consumers. And scale-to-zero is not a Kubernetes-native trick for HTTP services — vanilla HPA won't go below 1 because someone must answer the next request; that's Knative's/Cloud Run's party trick: hold the request at the edge, cold-start a container, forward. The cold-start latency is the price, which returns us to serverless economics (next dive).

🔍 Deep dive · Serverless under the hood — Firecracker, cold starts, and the bill click to expand

What actually happens on a Lambda invoke: the request hits the service, which either reuses a warm execution environment — a Firecracker microVM with your runtime and code already initialized — or performs a cold start: provision the microVM (~100 ms-order), boot the runtime, run your init code. That last part is why cold starts range from ~100–300 ms for a lean Node/Python function to seconds for a JVM with a fat framework and a 100 MB bundle. Mitigations, in escalating order of cost: trim the bundle, move work out of init, SnapStart (snapshot the initialized VM and restore instead of re-initializing — built for the JVM case), and provisioned concurrency — pre-warmed environments you pay for around the clock, which quietly deletes the scale-to-zero economics you came for.

The model's sharp edges are all consequences of one fact: an execution environment handles one request at a time. Concurrency = number of live microVMs, so 1,000 concurrent requests = 1,000 environments — and your account has a concurrency quota (historically ~1,000 by default, raisable). Each environment opening its own database connection is how Lambda fleets famously exhaust Postgres — hence RDS Proxy and a general preference for HTTP-based data APIs (like DynamoDB's) behind functions. Add the 15-minute execution cap, ephemeral local disk, and per-ms × memory-size billing, and the fit rule writes itself.

Where serverless wins: spiky or low-duty-cycle work (the 40-seconds-a-day cron, the webhook burst), event glue between managed services, unpredictable scale where paying per-use beats capacity planning. Where it loses: steady high-utilization load (always-on containers are typically much cheaper once you'd keep them busy most of the day), strict p99 latency (cold starts), long-lived connections, heavy local state. The middle groundlings: Fargate — serverless containers: you size the task, AWS finds the machine; startup in the tens of seconds, no duration cap, no cluster to run. Cloud Run — request-scaled containers with scale-to-zero and, crucially, many concurrent requests per instance (configurable, default ~80), which fixes the connection-explosion problem while keeping the pay-per-use shape.

One more honesty checkpoint: serverless is not "no ops". It's different ops — quotas and concurrency limits, cold-start budgets, distributed tracing across dozens of tiny functions, IAM sprawl. You've traded patching machines for managing a constellation. Usually a good trade; never a free one.

⚖️ At scale — numbers, limits & failure modes

Ballparks and design targets, not SLAs — quote the order of magnitude, and say so.

K8s cluster envelopeUpstream-tested design targets: ~5,000 nodes, ~150k pods per cluster, ~110 pods per node default. In practice orgs run many smaller clusters long before hitting these — blast radius and upgrade risk bite first.
etcdHappy at single-digit GB (default quota ~2 GiB, ~8 GiB recommended ceiling). It's a Raft-replicated config store — every write hits a quorum, everything watches it. Big blobs and high-churn objects don't belong there.
pod cold startScheduling is sub-second; image pull dominates — ~100 MB images pull in seconds, multi-GB images in minutes. Small images + digest pinning + pre-pulled base layers are a scale-out latency feature, not hygiene.
autoscaling lagHPA reacts in ~15–60 s end-to-end; adding nodes takes ~1–5 min (provision, join, warm cache). Hence 30–50% headroom on anything spike-prone.
rollout defaultsRollingUpdate defaults 25% surge / 25% unavailable; progress deadline 600 s. Probes are opt-in — no readiness probe, no safety gate.
LambdaCold start ~100–300 ms (lean Node/Python) → 1 s+ (JVM/heavy bundles); 15 min max execution; memory to 10 GB; per-ms billing; account concurrency quota (order of ~1k default, raisable); one request per environment at a time.
density ladderVMs: GBs each, tens per host · containers: MBs overhead, hundreds per host · Firecracker microVMs: ~MB-order overhead, thousands per host, ~100–125 ms boot.
mesh taxSidecar proxies add ~ms-order latency per hop and a fleet of Envoys to operate. Fine at p50; audible in a tight p99 budget across 5 hops.
Failure modes that only appear at scale: OOMKill loops (no/undersized memory limits → exit 137 → CrashLoopBackOff — the most common "K8s is flaky" root cause); probe-less deploys (every release is a brief outage nobody can explain); autoscaler thrash (scale-up/down flapping without stabilization windows — pods churn, caches stay cold); image-pull storms (zone failure → mass reschedule → every node pulls simultaneously → registry rate-limits become the outage); noisy neighbors (pods without requests make the scheduler fly blind — one greedy pod starves a node); retry amplification (client + mesh + gateway each retrying 3× = 27× load on a struggling service — retries need budgets, ideally in one layer only).

8 · In the interview

How it comes up: almost never as "design Kubernetes" — it's the substrate under every design. Say your one sentence early ("stateless services in containers on managed infra, HPA on QPS, rolling deploys with readiness gates") and spend your minutes on data. Then expect probes: how do you deploy without downtime? (readiness gating + canary + auto-rollback), a node dies mid-traffic — walk me through recovery (the control loop, with the ~40 s / ~5 min defaults hedged as defaults), how do you handle a 10× spike? (headroom + queue, because node autoscaling is minutes), why not Lambda? (duty cycle, p99, connection model).

The EM-specific question — "your team wants to migrate to K8s, what do you do?" — is an organizational buy-vs-build question wearing a YAML costume. The strong answer names the real cost: Kubernetes is free the way a puppy is free; running it well is a platform team (upgrades ~3×/year, node images, ingress/CNI/secret choices, on-call for the cluster itself). So: what problem are we solving that Cloud Run/Fargate can't? Who feeds the cluster — do we have (or budget) platform engineers? What's the migration cost and the exit criteria? Adopting K8s to run three CRUD services is a résumé-driven decision; adopting it because you run hundreds of heterogeneous workloads and already staff a platform group is an obvious one.

The choice script: default to managed containers (Cloud Run / Fargate / App Runner) for product services — one sentence, zero cluster ops. Reach for Kubernetes when you need its ecosystem (operators, jobs, sidecars, GPU scheduling, multi-cloud portability) and have platform engineers to own it. Use Lambda for event glue and spiky, short, stateless work. Keep plain VMs + autoscaling group in your pocket — still a legitimate, boring answer for monoliths and special hardware. And never hand-roll stateful stores on K8s without a strong reason; buy the managed database and spend your innovation tokens elsewhere.

9 · Check yourself

What actually separates a container from a VM — and when is container isolation not enough?

A VM virtualizes hardware and boots its own kernel; the tenant boundary is the narrow hypervisor interface. A container is a normal process on the shared host kernel, with namespaces limiting what it sees and cgroups limiting what it uses — start in ms, pack hundreds per host, but every tenant is one kernel exploit from its neighbors. That's fine for your own trusted code; for other people's code (multi-tenant SaaS runners, CI, plugin/AI-code sandboxes) you want VM-grade walls at container-grade speed — microVMs (Firecracker, what Lambda/Fargate use) or a syscall-interposition layer like gVisor.

Every deploy causes ~30 seconds of 5xx, yet the new pods "start fine". What's missing?

A readiness probe. Without one, Kubernetes counts a pod Ready the moment its process starts, adds it to the Service endpoints, and routes traffic while the app is still warming up (or broken) — and simultaneously terminates old pods. With a readiness probe, new pods receive traffic only after passing it, and a pod that never passes blocks the rollout instead of blacking out users. (Check the shutdown side too: a preStop delay/graceful drain stops the reverse problem — killing pods with requests in flight.)

A node powers off at 3am. Trace what Kubernetes does — who "handles" the failure?

Nobody, and that's the answer's point. The kubelet's heartbeats stop → node controller marks the node NotReady (order of ~40 s with defaults) → after the eviction toleration (~5 min default) its pods are marked dead → the ReplicaSet controller merely observes 5 running vs 6 desired and creates a pod → the scheduler places it on a healthy node → that kubelet starts it, readiness gates traffic. Recovery is not a special code path — it's the same reconcile loop that runs when nothing is wrong. That's the practical meaning of "declarative".

Interviewer: "Why not run Postgres on the cluster with a StatefulSet? It gives you stable identity and volumes."

It does — and that's the easy 10%. The hard 90% is what managed databases sell: failover judgment (when is the primary really dead? split-brain safety), backups tested by restore, minor-version upgrades without downtime, storage performance tuning, and someone on call who's seen your failure before. Operators (e.g. for Postgres) encode much of this, but you're still operating a database platform. Defensible exceptions: very large scale where managed pricing breaks, data-sovereignty/on-prem constraints, or a team whose actual job is databases. Default: buy it.

Steady 2,000 req/s API — Lambda or containers? Walk the reasoning.

Containers. Steady high duty-cycle is the worst case for per-ms pricing: you'd pay serverless premium for capacity you use continuously, while a small always-on fleet stays busy and cheap. Cold starts poison strict p99, and one-request-per-environment means ~thousands of concurrent environments each wanting a DB connection (hello, connection exhaustion — RDS Proxy exists precisely for this). Lambda stays in the design for the spiky edges — webhooks, async fan-out, cron — where scale-to-zero and per-use billing actually pay. The boundary to say out loud: duty cycle and latency tolerance decide, not fashion.
