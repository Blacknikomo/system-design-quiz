📐 Algorithms in JS — canonical forms & how they get broken

The reference implementations you should know cold, each followed by the ways it's typically mis-implemented. Built for the Code Comprehension round — most planted bugs are one-line deviations from these templates.

How to use this page: read each canonical form until you could write it from memory, then study its bug table. In the interview you won't implement these — you'll diff unfamiliar code against the canonical shape in your head. 🔎 Searching & sorting binary search · merge · partition

Binary search — exact match O(log n)

The classic inclusive-bounds template. Its three load-bearing choices — lo <= hi, mid ± 1, inclusive hi — must agree with each other; mixing templates is the #1 planted bug.

function binarySearch(arr, target) {          // arr must be sorted ascending
  let lo = 0, hi = arr.length - 1;            // INCLUSIVE bounds
  while (lo <= hi) {                          // <= : a 1-element range is still searchable
    const mid = (lo + hi) >> 1;               // floor — plain /2 gives a fractional index in JS
    if (arr[mid] === target) return mid;
    if (arr[mid] < target) lo = mid + 1;      // ±1: mid is already ruled out
    else hi = mid - 1;
  }
  return -1;
}

The lower-bound variant (first index with arr[i] >= target — i.e. the insertion point) uses the other template. Know both; never mix them:

function lowerBound(arr, target) {
  let lo = 0, hi = arr.length;                // hi is EXCLUSIVE here
  while (lo < hi) {                           // strict <
    const mid = (lo + hi) >> 1;
    if (arr[mid] < target) lo = mid + 1;
    else hi = mid;                            // keep mid — it may BE the answer
  }
  return lo;                                  // may equal arr.length (insert at end)
}
Common bugWhat breaksSev
while (lo < hi) with inclusive hi = length - 1The final 1-element range is never tested — misses the target when it sits at the last narrowed position.P0
hi = mid (or lo = mid) in the exact-match templateRange stops shrinking when lo === hi (or on 2 elements) → infinite loop. In the lower-bound template hi = mid is correct — that's why mixing templates is deadly.P0
const mid = (lo + hi) / 2Fractional index → arr[1.5] is undefined, all comparisons false → loop runs to a wrong answer or forever.P0
No check that input is sorted (or sorted by the same comparator)Silently wrong results on unsorted data. In review: ask "who guarantees the ordering — and is it the same key we search by?"P1
Searching objects with ===Reference equality never matches a structurally-equal object; needs a key extractor / comparator.P0

Note: the famous (lo + hi) integer overflow from Java/C doesn't bite in JS until 2⁵³ — but mentioning "in Java this would need lo + (hi - lo) / 2" is a free signal.

Merge two sorted arrays O(n + m)

The heart of merge sort and of every "combine two sorted streams" task (log merging, sorted pagination). Two cursors, one output.

function merge(a, b) {
  const out = [];
  let i = 0, j = 0;
  while (i < a.length && j < b.length) {
    if (a[i] <= b[j]) out.push(a[i++]);       // <= keeps the merge STABLE (a wins ties)
    else out.push(b[j++]);
  }
  while (i < a.length) out.push(a[i++]);      // drain the tails —
  while (j < b.length) out.push(b[j++]);      // exactly one array has leftovers
  return out;
}
Common bugWhat breaksSev
Missing tail drainsEverything after one array is exhausted is silently dropped. Test that catches it: arrays of different lengths.P0
Reading a[i] / b[j] without bounds check in a single combined loopundefined enters the comparison; undefined <= x is always false, so one side gets drained in the wrong order.P0
< instead of <=Merge still sorts, but becomes unstable — equal elements from b jump ahead. Matters when elements carry payloads (sort by ts, keep insertion order).P1
Comparator via subtraction on non-numbersa - b on strings/Dates coerces or yields NaN; NaN makes the comparison false every time → arbitrary order.P0

Quicksort — Lomuto partition O(n log n) avg · O(n²) worst

What matters in review: the base case, excluding the pivot from recursion, and knowing when the O(n²) worst case is reachable.

function quicksort(arr, lo = 0, hi = arr.length - 1) {
  if (lo >= hi) return arr;                   // base case: 0 or 1 elements
  const p = partition(arr, lo, hi);
  quicksort(arr, lo, p - 1);                  // EXCLUDE the pivot: it's final.
  quicksort(arr, p + 1, hi);                  // include it → infinite recursion on duplicates
  return arr;                                 // sorts IN PLACE — callers must know
}

function partition(arr, lo, hi) {
  const pivot = arr[hi];                      // last element as pivot
  let i = lo;                                 // boundary of the "< pivot" zone
  for (let j = lo; j < hi; j++) {             // j < hi: don't touch the pivot itself
    if (arr[j] < pivot) {
      [arr[i], arr[j]] = [arr[j], arr[i]];
      i++;
    }
  }
  [arr[i], arr[hi]] = [arr[hi], arr[i]];      // drop pivot into its final slot
  return i;
}
Common bugWhat breaksSev
Recursing on (lo, p) and (p, hi) — pivot includedOn arrays with duplicates (or even without), a partition that doesn't shrink → stack overflow.P0
Missing / wrong base case (lo > hi only, not >=)Usually still terminates, but 1-element calls do useless partitions; combined with the bug above it hangs.P1
Last-element pivot on already-sorted inputEvery partition splits n → (n−1, 0): O(n²) time and O(n) recursion depth → stack overflow around n ≈ 10⁴. Fix: random pivot or median-of-three. Ask: "can prod data arrive pre-sorted?" — it usually can.P1
In-place mutation not surfaced in the APIsortUsers(users) silently reorders the caller's array — action at a distance. Name it (sortInPlace) or copy first.P1
Falling back to arr.sort() with no comparatorDefault sort is lexicographic: [10, 9, 1].sort() → [1, 10, 9]. Needs (a, b) => a - b for numbers.P0

Also know: quicksort is not stable; JS's built-in Array.prototype.sort is required to be stable (since ES2019) and is O(n log n) — in real code, "why not just sort() with a comparator?" is a legitimate review question.

🕸️ Graphs BFS · DFS · topological sort · Dijkstra

BFS — breadth-first search O(V + E)

THE algorithm for shortest path in an unweighted graph. Two invariants carry it: mark visited when enqueuing, and use a real O(1) queue.

function bfs(adj, start) {                    // adj: Map node → [neighbors]
  const dist = new Map([[start, 0]]);         // doubles as the "visited" set
  const queue = [start];
  let head = 0;                               // index-based queue: shift() is O(n)
  while (head < queue.length) {
    const node = queue[head++];
    for (const next of adj.get(node) ?? []) {
      if (!dist.has(next)) {                  // mark when ENQUEUED, not when dequeued
        dist.set(next, dist.get(node) + 1);
        queue.push(next);
      }
    }
  }
  return dist;                                // unreachable nodes are simply absent
}
Common bugWhat breaksSev
Marking visited on dequeue instead of enqueueThe same node gets enqueued via multiple neighbors before it's ever processed — queue blows up exponentially on dense graphs. Answers stay correct, runtime doesn't.P0
queue.shift()O(n) per dequeue → O(n²) total. Fine at n=100, a hang at n=10⁶. The head-index trick (or a deque) fixes it.P1
Using a stack (pop()) but calling it BFSIt's DFS now — traversal still visits everything, but "shortest path" results are wrong. Name/behavior mismatch: check push/pop ends match the claimed order.P0
Grid BFS: bounds check after reading the cellgrid[r][c] with r out of range throws (or yields undefined and poisons logic). Check bounds first, then visited, then value.P0
Assuming one BFS covers the whole graphDisconnected components are never visited. If the task is "process every node", loop over all nodes and BFS from each unvisited one.P1

DFS — recursive & iterative O(V + E)

function dfs(adj, node, visited = new Set()) {
  visited.add(node);                          // mark BEFORE recursing — or cycles recurse forever
  for (const next of adj.get(node) ?? []) {
    if (!visited.has(next)) dfs(adj, next, visited);
  }
  return visited;
}

// Iterative — for deep graphs (JS call stack dies around ~10⁴ frames)
function dfsIterative(adj, start) {
  const visited = new Set();
  const stack = [start];
  while (stack.length) {
    const node = stack.pop();
    if (visited.has(node)) continue;          // a node can sit in the stack twice — dedupe on pop
    visited.add(node);
    for (const next of adj.get(node) ?? []) {
      if (!visited.has(next)) stack.push(next);
    }
  }
  return visited;
}
Common bugWhat breaksSev
Marking visited after the recursive callsOn any cycle the recursion never bottoms out → stack overflow. The mark must precede exploration.P0
Recursive DFS on user-sized input~10⁴+ deep chain (linked-list-shaped graph, deep file tree) → RangeError: Maximum call stack size exceeded. Review question: "how deep can this get in prod?"P1
Cycle detection in a directed graph with a single visited setA visited node reached again isn't necessarily a cycle — it may be a diamond (A→B→D, A→C→D). Directed cycle detection needs three states: unvisited / in current path / done. One boolean gives false positives.P0
Iterative version without the dedupe-on-pop checkSame node processed multiple times; with side effects per node (writes, sends) that's duplicated work or corrupted state.P1

Topological sort — Kahn's algorithm O(V + E)

Ordering with dependencies: build systems, task schedulers, migration runners. The one check everyone forgets is the cycle check at the end.

function topoSort(n, edges) {                 // edges: [from, to] = "from must precede to"
  const adj = Array.from({ length: n }, () => []);
  const indeg = new Array(n).fill(0);
  for (const [u, v] of edges) {
    adj[u].push(v);
    indeg[v]++;                               // count incoming edges
  }
  const queue = [];
  for (let v = 0; v < n; v++) if (indeg[v] === 0) queue.push(v);
  const order = [];
  let head = 0;
  while (head < queue.length) {
    const u = queue[head++];
    order.push(u);
    for (const v of adj[u]) {
      if (--indeg[v] === 0) queue.push(v);    // v becomes free only when ALL prereqs done
    }
  }
  return order.length === n ? order : null;   // < n ⇒ cycle. NEVER skip this check
}
Common bugWhat breaksSev
No order.length === n checkOn a cyclic input the function returns a silent partial order — the scheduler just never runs some tasks. Worst kind of bug: no error, missing work.P0
Edge direction flipped (adj[v].push(u) / indeg[u]++)Produces the reverse topological order — tests with symmetric fixtures won't catch it. Trace one concrete edge: "install deps before app".P0
Pushing a node when indeg > 0 (e.g. if (indeg[v] <= 1))Nodes run before all their prerequisites complete. In a migration runner that's data corruption.P0
Nodes that appear only in edges, with n derived from edgesIsolated nodes (no edges at all) vanish from the order. Ask where n comes from.P1

Dijkstra — shortest paths with weights O((V + E) log V)

BFS's weighted sibling. The canonical JS version uses a min-heap with lazy deletion: push duplicates, skip stale ones on pop.

function dijkstra(adj, src) {                 // adj: Map node → [[next, weight], ...]
  const dist = new Map([[src, 0]]);
  const pq = new MinHeap((a, b) => a[0] - b[0]);  // entries: [dist, node] — comparator, see Heap below
  pq.push([0, src]);
  while (pq.size) {
    const [d, u] = pq.pop();
    if (d > (dist.get(u) ?? Infinity)) continue;  // STALE entry — a shorter path already won. Skip.
    for (const [v, w] of adj.get(u) ?? []) {
      const nd = d + w;
      if (nd < (dist.get(v) ?? Infinity)) {
        dist.set(v, nd);
        pq.push([nd, v]);                     // lazy deletion: just push again, don't decrease-key
      }
    }
  }
  return dist;
}
Common bugWhat breaksSev
Plain BFS / FIFO queue on a weighted graphFewest hops ≠ shortest distance. Classic when someone "extends" existing BFS code with weights.P0
No stale-entry skip, node finalized on first pushMarking a node "done" when first discovered (BFS habit) finalizes a non-optimal distance. The skip-on-pop line is what makes lazy deletion correct.P0
Negative edge weightsDijkstra's greedy invariant breaks silently — wrong distances, no error. Review question: "can this weight be negative (refunds, penalties, time offsets)?" If yes → Bellman-Ford.P0
if (!dist.get(v)) instead of ndist < (dist.get(v) ?? Infinity)Truthiness treats a legitimate distance of 0 (the source!) as "unset" and re-relaxes it. The ?? Infinity pattern dodges both this and undefined comparisons.P0
Sorted array instead of a heap, re-sorted every pushO(n log n) per operation → O(n² log n) total. Works in the demo, dies at scale — a P1 with a "how big is V?" question attached.P1
🪟 Windows & pointers sliding window · two pointers

Sliding window — longest substring without repeats O(n)

The variable-size window pattern: right expands every iteration, left jumps forward on violation, answer recorded after the window is valid again.

function longestUnique(s) {
  const last = new Map();                     // char → last index seen
  let best = 0, left = 0;
  for (let right = 0; right < s.length; right++) {
    const c = s[right];
    if (last.has(c) && last.get(c) >= left) { // duplicate INSIDE the current window?
      left = last.get(c) + 1;                 // jump PAST it — not left++
    }
    last.set(c, right);
    best = Math.max(best, right - left + 1);  // +1: window [left..right] is inclusive
  }
  return best;
}
Common bugWhat breaksSev
right - left without +1Every answer off by one. The single most-planted bug in window code — check it first, takes 2 seconds.P0
Missing the >= left guardA duplicate behind the window drags left backwards — window grows illegally, answers too large. Trace "abba": at the final a, the stale a@0 must be ignored.P0
left++ instead of jumping past the duplicate (in this Map variant)Window still contains the duplicate → invariant broken, wrong answers. (In the while-shrink variant left++ in a loop is correct — know which variant you're reading.)P0
Recording best before restoring validity (in while-shrink variants: if instead of while)An invalid window gets counted as a candidate answer. Rule: expand → restore invariant → record.P0
Iterating with s[right] on unicode beyond BMPEmoji / rare CJK are surrogate pairs: indexing splits them. Usually fine for interview strings — but noticing is a P2 that scores.P2

Two pointers — pair sum in a sorted array O(n)

function pairSum(arr, target) {               // arr sorted ascending
  let i = 0, j = arr.length - 1;
  while (i < j) {                             // STRICT < : an element can't pair with itself
    const sum = arr[i] + arr[j];
    if (sum === target) return [i, j];
    if (sum < target) i++;                    // too small → only the left move can help
    else j--;
  }
  return null;
}
Common bugWhat breaksSev
i <= jWhen pointers meet, arr[i] + arr[i] pairs an element with itself — [3] "finds" 6. Contract question: is reusing an element allowed?P0
Running it on an unsorted arrayThe move-the-pointer logic relies entirely on sortedness; unsorted input gives arbitrary misses. If the code sorts first, indexes returned now point into the sorted copy — a second, subtler bug.P0
Moving both pointers on a match while collecting all pairs with duplicates presentEither misses duplicate pairs or double-counts them; needs explicit duplicate-skipping loops (while (arr[i] === arr[i+1]) i++) — which themselves need bounds checks.P1
Overflow habits from other languagesa + b can't overflow to negative in JS, but precision silently degrades past 2⁵³ (IDs!). Number.isSafeInteger is the check.P2
🏗️ Structures in code binary heap · LRU cache

Binary min-heap push/pop O(log n) · peek O(1)

Array-backed complete tree. All the bugs live in the index arithmetic and in pop()'s edge cases. 0-based math: parent (i-1)>>1, children 2i+1, 2i+2.

class MinHeap {
  constructor(cmp = (a, b) => a - b) {        // comparator: NEVER rely on < for objects/arrays
    this.a = [];
    this.cmp = cmp;
  }
  get size() { return this.a.length; }
  peek() { return this.a[0]; }

  push(x) {
    const a = this.a;
    a.push(x);
    let i = a.length - 1;
    while (i > 0) {                           // sift UP
      const p = (i - 1) >> 1;                 // 0-based parent
      if (this.cmp(a[p], a[i]) <= 0) break;
      [a[p], a[i]] = [a[i], a[p]];
      i = p;
    }
  }

  pop() {
    const a = this.a;
    if (a.length === 0) return undefined;
    const top = a[0];
    const last = a.pop();                     // also handles the 1-element heap
    if (a.length > 0) {
      a[0] = last;                            // move LAST to root, then sink it
      let i = 0;
      while (true) {                          // sift DOWN
        const l = 2 * i + 1, r = 2 * i + 2;
        let m = i;
        if (l < a.length && this.cmp(a[l], a[m]) < 0) m = l;
        if (r < a.length && this.cmp(a[r], a[m]) < 0) m = r;  // compare BOTH children
        if (m === i) break;
        [a[m], a[i]] = [a[i], a[m]];
        i = m;
      }
    }
    return top;
  }
}
Common bugWhat breaksSev
1-based index math (2i, 2i+1, parent i>>1) on a 0-based arrayHeap property silently violated near the root; pop returns non-minimums sometimes — flaky, data-dependent, awful to debug. Verify the math against the storage convention first thing.P0
Sift-down comparing only the left childWhen the right child is the smaller one, it never bubbles up → wrong pop order. Test: push [2, 3, 1].P0
Comparing with < instead of a comparatorFor numbers fine; for arrays/objects JS coerces to strings: [10] < [2] is true. A heap of [dist, node] pairs sorts lexicographically — Dijkstra quietly wrong.P0
pop() ignoring the empty / 1-element casesa[0] = a.pop() on a 1-element heap re-inserts the element it just removed — heap never empties, consumer loops forever.P0
Top-K via full sort instead of a size-K heapO(n log n) + O(n) memory vs O(n log k). Legit at small n — ask about n before flagging. Streaming input makes the heap mandatory.P1

LRU cache — Map-based get/put O(1)

In JS the doubly-linked-list + hashmap classic collapses into a Map, because Map preserves insertion order and re-inserting moves a key to the back. The recency bookkeeping is where all bugs live.

class LRUCache {
  constructor(capacity) {
    this.cap = capacity;
    this.m = new Map();                       // oldest = first key in iteration order
  }
  get(key) {
    if (!this.m.has(key)) return undefined;
    const v = this.m.get(key);
    this.m.delete(key);
    this.m.set(key, v);                       // refresh recency ON READ — this is the "U" in LRU
    return v;
  }
  put(key, value) {
    if (this.m.has(key)) this.m.delete(key);  // set() on an existing key KEEPS its old position
    this.m.set(key, value);
    if (this.m.size > this.cap) {
      const oldest = this.m.keys().next().value;
      this.m.delete(oldest);                  // evict least-recently-used
    }
  }
}
Common bugWhat breaksSev
get() doesn't refresh recencyThe cache is silently FIFO, not LRU — hot keys get evicted while cold recent writes survive. Functionally "works", hit rate quietly tanks. Name/behavior mismatch.P0
put() on existing key without delete-firstMap.set updates the value but keeps the original position — the key's recency never updates on overwrite.P1
Evicting before checking whether the key already existsUpdating an existing key on a full cache wrongly evicts someone — capacity shrinks by one victim per update.P1
if (!this.m.get(key)) for the existence checkStored 0, "", false, null all read as "miss" → permanent re-fetch of legitimately-falsy values. Use has().P0
Plain object {} as the storeNo insertion-order guarantee for numeric-like keys (integer keys iterate first, sorted!), keys coerced to strings (1 vs "1" collide), prototype pollution via "__proto__". Map exists for this.P0
No TTL / no size bound in a "cache" that only ever growsNot this class — but in review, any hand-rolled cache = {} without eviction is a slow memory leak. Standard P1 finding.P1
🧠 Recursion & memoization memo caches · DP shape

Memoization O(n) from O(2ⁿ)

The pattern matters more than fib itself: where the cache lives, how keys are built, and what counts as a hit are the three bug farms.

function makeFib() {
  const memo = new Map([[0, 0], [1, 1]]);     // cache OUTSIDE the recursive fn, INSIDE the closure
  return function fib(n) {
    if (memo.has(n)) return memo.get(n);      // has(), not truthiness — fib(0) is 0, a falsy hit
    const val = fib(n - 1) + fib(n - 2);
    memo.set(n, val);                         // write BEFORE returning
    return val;
  };
}
const fib = makeFib();

// Generic memoize for a single-primitive-arg pure function
function memoize(fn) {
  const cache = new Map();
  return (x) => {
    if (!cache.has(x)) cache.set(x, fn(x));
    return cache.get(x);
  };
}
Common bugWhat breaksSev
if (memo[n]) return memo[n]Cached falsy results (0, "", false, null) are recomputed forever — correctness survives, the exponential blowup returns. With fib the cached 0 makes it O(2ⁿ) again.P0
Cache declared inside the recursive function's own body / recreated per top-level callEach outer call starts cold; cross-call reuse — the entire point — never happens. Look at where const memo sits relative to the returned function.P1
Multi-arg key via args.join(",")Collisions: f("a,b") vs f("a", "b") share a key; objects join as "[object Object]" — every object hits the same entry. Use JSON.stringify deliberately, or nested Maps.P0
Plain object as cache with hostile keysmemo["constructor"] / memo["__proto__"] exist before you set them — truthy garbage hits. Map again.P1
Memoizing an impure function (reads config, Date.now, DB)First call's answer is frozen forever; staleness masquerades as caching. Review question: "is this referentially transparent?"P1
Async memoization caching the value, not the promiseConcurrent first calls all miss (cache fills only after await) → thundering herd of N identical requests. Cache the promise itself; delete the entry on rejection or failures get cached forever.P0
Unbounded memo on user-driven keysA cache keyed by request input grows monotonically — memory leak in a long-lived process. Bound it (LRU above) or scope its lifetime.P1

Bottom-up DP variant of the same bugs: array sized n instead of n + 1 (read past the end → undefined → NaN cascades), and base cases at the wrong indices. NaN anywhere in a DP table means a base-case or sizing bug — say so out loud.

⚡ JS-specific debounce/throttle · promise pool · deep clone · event loop

Debounce & throttle

Debounce = act after the burst goes quiet (search-as-you-type). Throttle = act at most once per interval during the burst (scroll handlers). Confusing them is itself the classic bug.

function debounce(fn, ms) {
  let timer;
  return function (...args) {                 // function, not arrow: preserve caller's `this`
    clearTimeout(timer);                      // forget this line = no debounce at all
    timer = setTimeout(() => fn.apply(this, args), ms);
  };
}

function throttle(fn, ms) {                   // leading + trailing edges
  let last = 0, timer = null, lastArgs;
  return function (...args) {
    lastArgs = args;                          // always remember the LATEST args
    const now = Date.now();
    if (now - last >= ms) {
      last = now;
      fn.apply(this, args);                   // leading call: fire immediately
    } else if (!timer) {
      timer = setTimeout(() => {              // trailing call: don't lose the final event
        timer = null;
        last = Date.now();
        fn.apply(this, lastArgs);
      }, ms - (now - last));
    }
  };
}
Common bugWhat breaksSev
Missing clearTimeout in debounceEvery keystroke schedules its own call — you get N delayed calls instead of 1. The name says debounce; the behavior is "delay everything".P0
Throttle without a trailing callThe last event of a burst is dropped — final scroll position never rendered, last keystroke never searched. Ask which edges the caller needs: leading, trailing, or both.P1
Arrow function as the returned wrapperthis is captured from the definition site, not the call site — methods debounced on a class lose their instance. (Arrow inside setTimeout is correct — that one should inherit.)P1
Capturing the first burst's args for the trailing callTrailing invocation replays a stale event — UI updates to an old scroll position. lastArgs must be overwritten on every call.P1
Debounced async fn — responses racingDebounce limits starts, not completions: an older slow response can overwrite a newer fast one. Needs a request token / AbortController — the bug lives one level above the debounce.P0
Never cancelled on teardownThe pending timer fires after unmount/dispose and touches dead state. Expose a .cancel(); call it in cleanup.P1

Promise pool — run N tasks, at most K in flight concurrency limiter

The pattern behind every "fetch 10 000 URLs politely". Key insight: tasks must be factories (() => Promise) — a Promise starts running the moment it's created.

async function promisePool(tasks, limit) {    // tasks: Array<() => Promise>
  const results = new Array(tasks.length);
  let next = 0;
  async function worker() {
    while (next < tasks.length) {
      const i = next++;                       // claim an index BEFORE the await —
      results[i] = await tasks[i]();          // sync code between awaits is atomic in JS
    }
  }
  const workers = Array.from(
    { length: Math.min(limit, tasks.length) },
    () => worker()
  );
  await Promise.all(workers);                 // one rejection rejects the pool — wrap tasks
  return results;                             // results in ORIGINAL order, not completion order
}
Common bugWhat breaksSev
tasks.map(t => t()) anywhere before poolingAll N promises are created — and therefore already running — at once. The "pool" then politely awaits an avalanche it already unleashed. This is THE bug to scan for.P0
await inside forEachforEach ignores the returned promises: nothing is awaited, errors are unhandled rejections, "done" fires before any work completes. Use for..of or collect promises for Promise.all.P0
Claiming the index after an await (results[next] = await …; next++)Two workers read the same next across the await gap — duplicated work, skipped tasks. Single-threaded JS still has interleaving races at every await.P0
Sequential for..of await sold as "parallel"Correct but K=1: 10 000 sequential fetches. Performance finding — pair it with "what's the acceptable wall time and the downstream rate limit?"P1
No per-task error handlingFirst rejection rejects Promise.all; in-flight work keeps running with no observer. Decide: fail-fast or Promise.allSettled semantics — and make the choice visible in the API.P1
Assuming results arrive in completion orderAnything keyed by "push as they finish" reorders under load. Index-addressed writes (as above) keep input order.P1

Deep clone

First ask why the code clones at all — the best finding is often "this clone hides a design problem". When a real deep clone is needed, the platform now provides one.

// The right default since 2022 — handles cycles, Dates, Maps, Sets, typed arrays:
const copy = structuredClone(value);
// Limits: throws on functions and DOM nodes; drops property getters; loses class prototypes.

// Hand-rolled (what you'll actually review) — the checklist version:
function deepClone(value, seen = new WeakMap()) {
  if (value === null || typeof value !== "object") return value;  // primitives (and functions) pass through
  if (seen.has(value)) return seen.get(value);                    // CYCLES — without this: infinite recursion
  if (value instanceof Date) return new Date(value);
  if (value instanceof Map)
    return new Map([...value].map(([k, v]) => [k, deepClone(v, seen)]));
  if (value instanceof Set)
    return new Set([...value].map((v) => deepClone(v, seen)));
  const out = Array.isArray(value) ? [] : {};
  seen.set(value, out);                       // register BEFORE recursing (cycles again)
  for (const k of Object.keys(value)) out[k] = deepClone(value[k], seen);
  return out;
}
Common bugWhat breaksSev
{...obj} / Object.assign / arr.slice() as a "deep" copyOne level only — nested objects are still shared; a later mutation reaches through the "copy". The classic shows up as "sometimes the original changes too".P0
JSON.parse(JSON.stringify(x))Drops undefined, functions and symbol keys; Date → string; Map/Set → {}; NaN/Infinity → null; throws on cycles and BigInt. Acceptable only for known-plain JSON data — say that precisely.P1
No cycle handling in a hand-rolled cloneAny back-reference (child.parent) → stack overflow. The WeakMap must be populated before recursing into children.P0
Cloning class instances as plain objectsMethods and instanceof silently vanish; the "copy" walks and quacks differently. Ask whether identity/prototype matters here.P1
Cloning defensively on every call in hot pathsO(size) per call where a reader could share immutably. The deeper finding: mutation-happy design forcing copies everywhere.P1

Event-loop ordering — sync → microtasks → macrotasks

Not an algorithm, but the mental model every JS async bug reduces to. One queue drains completely before the other gets a turn.

console.log("1: sync");
setTimeout(() => console.log("4: macrotask (timer)"), 0);
Promise.resolve().then(() => console.log("3: microtask"));
console.log("2: sync");
// Order: 1, 2, 3, 4 — ALL microtasks drain before the next macrotask.

async function f() {
  console.log("A: runs synchronously");       // code before the first await is SYNC
  await something;
  console.log("B: a microtask continuation"); // everything after await is deferred
}

✅ Check yourself

Binary search: the code has hi = arr.length - 1 and while (lo < hi). What's the bug and what input exposes it?
Inclusive upper bound needs lo <= hi — with strict < the loop exits when the range narrows to one element without testing it. Exposed by any target that sits where the range converges — simplest: a 1-element array [42], target 42 → returns -1. Severity P0.
A BFS marks nodes visited when they're dequeued. Is the answer wrong, or something else?
Distances/reachability stay correct — the first dequeue of a node still happens at the right layer. What breaks is performance: the same node is enqueued once per incoming edge before its first dequeue, so the queue can grow toward O(E) duplicates and revisit work explodes on dense graphs. It's a P0-at-scale masquerading as "works on my tests".
Why must promise-pool tasks be functions returning promises, not promises?
A Promise represents work already started — by the time you have an array of promises, all N requests are in flight and no pool can un-start them. Factories let the pool control the start moment. Scan for tasks.map(t => t()) or an array built with fetch(...) calls directly.
An LRU cache's get() is just return this.m.get(key). What's the finding and how do you phrase it?
Recency never updates on reads, so eviction order is insertion order — it's a FIFO cache wearing an LRU name. Phrase as observation + trace + question: "If a key is read constantly but written once, it's still evicted first — walking capacity=2 with get-heavy traffic: … Is FIFO acceptable here, or should get() refresh recency?" That's a P0 on hit-rate-sensitive paths.
What does if (memo[n]) return memo[n] get wrong, and why is it worse than it looks?
Falsy cached values (0, "", false, null) never hit — they're recomputed on every call. For fib it reintroduces exponential time because fib(0)=0 sits on every recursion path. It looks like a style nit but it's a complexity-class regression: P0. The fix is has() on a Map (or key in obj).
Single-threaded JS can't have races — true?
False in the way that matters: every await is a yield point. Check-then-act across an await (cache fill, balance check, "is there already a request in flight?") interleaves exactly like a race. The fixes are single-flight patterns: cache the promise itself, claim state synchronously before awaiting, or queue mutations.
