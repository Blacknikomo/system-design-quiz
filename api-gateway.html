🚪 API Gateway — auth, rate limiting & routing

A single entry point in front of your services. Watch a request flow through the pipeline: authenticate → rate-limit → route to the right backend — and see exactly where 401s, 429s and 404s get returned.
single entry point auth rate limiting SSL termination routing AWS API Gateway · Kong · nginx

An API gateway sits in front of your backend services and handles cross-cutting concerns in one place — authentication, rate limiting, request logging, TLS/SSL termination — so individual services don't each reimplement them. After those checks pass, it routes the request to the correct backend based on the path.

The pipeline: every request runs the gauntlet — ① Auth (valid API key?) → ② Rate limit (token bucket per client) → ③ Route (path prefix → service). A failure at any stage short-circuits the rest and returns an error (401 / 429 / 404). Only a fully-passing request reaches a backend and gets a 200.
Token bucket: each client gets a bucket of capacity 5 that refills 1 token every 2s. Each allowed request spends a token; an empty bucket returns 429 Too Many Requests. Spam Send to drain it and trip the limit.

What you're seeing

  1. Client sends an HTTP request → it hits the gateway, the single entry point.
  2. ① Auth: gateway checks the API key. Invalid/missing → 401 and stop.
  3. ② Rate limit: spend one token from the client's bucket. Empty → 429 and stop.
  4. ③ Route: match the path prefix against the routing table → forward to that backend.
  5. Backend responds → gateway returns 200. No matching route → 404.
