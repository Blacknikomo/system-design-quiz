📬 Message Queues — RabbitMQ & SQS

The classic work queue: competing consumers, acks, visibility timeouts, dead letters — and why it is not a Kafka replacement (nor the other way around). ☕ ~15 min read
at-least-once destructive read competing consumers visibility timeout DLQ

A message queue is a to-do list shared by workers. Producers drop tasks in; each task is handed to exactly one worker; when the worker confirms it's done, the task is gone forever. That single design decision — consumed means deleted — is what separates queues (RabbitMQ, SQS) from logs (Kafka), and it drives every property on this page.

In the interview: reach for a queue whenever you say the words "process this asynchronously" — sending emails, resizing images, running payments, crawling URLs. The interviewer then expects you to handle, unprompted: duplicate deliveries (at-least-once), poison messages (DLQ), backpressure (queue depth), and ordering (or the lack of it).

1 · The mental model: a shared to-do list

Imagine a busy kitchen. Orders come in from waiters (producers) faster than any single cook can handle. So the orders go onto a spike — a queue — and each of five cooks (consumers) grabs the next ticket when free. Two things make this work: the ticket leaves the spike when grabbed (no two cooks fry the same steak), and if a cook faints mid-dish, the ticket must somehow get back on the spike (no order is silently lost).

Every queue system is an implementation of those two guarantees, and the second one is where all the interesting engineering lives. The broker can't see inside your worker. All it knows is: "I handed message #42 to consumer B, and B never confirmed." Did B crash before doing the work? After doing the work but before confirming? The broker cannot distinguish these cases — so it redelivers, and your system must tolerate a task being executed twice. That is the famous at-least-once delivery, and it isn't a bug or a cut corner: it is a mathematical consequence of unreliable networks (the Two Generals problem wearing a work apron).

The two systems worth naming in an interview sit at opposite ends of the ops spectrum:

RabbitMQ — a self-hosted, feature-rich broker (Erlang, speaks AMQP). Its superpower is routing: producers publish to an exchange, and the exchange copies the message into queues based on binding rules — direct match, wildcard topics, or broadcast. The broker pushes messages to connected consumers and tracks per-message acknowledgements.

Amazon SQS — a fully managed queue behind a plain HTTPS API. No servers, no partitions to size, effectively no throughput ceiling on standard queues. Consumers poll for messages; each received message becomes invisible for a visibility timeout, and is deleted only when the consumer explicitly says so. If the delete never comes — the message reappears. That one mechanism replaces acks, redelivery and crash-detection in one move.

Queue vs log — the one-sentence test. If every consumer competes to grab the next unit of work → queue (RabbitMQ/SQS). If several independent readers each need the whole stream, possibly re-read from the past → log (Kafka). A queue forgets a message the moment it's processed; a log remembers it for the retention window regardless of who read it.

2 · Syntax & core API

You won't write code in a system-design round, but fluently naming the operations signals hands-on experience. The vocabulary differs; the concepts map 1:1.

RabbitMQ (AMQP model)

exchangeWhere producers publish. Types: direct (exact routing-key match), topic (wildcard match, order.*.eu), fanout (copy to every bound queue), headers (rare).
queue ← bindingA queue subscribes to an exchange with a binding key. One message can be copied into many queues (that's the fan-out) but within one queue it goes to exactly one consumer.
basic.publishSend to an exchange with a routing key. persistent: true to survive broker restart (with a durable queue).
basic.consume / basic.ackBroker pushes messages; consumer explicitly acks each one. Unacked + dropped connection ⇒ redelivery to someone else.
basic.nack / rejectRefuse a message: requeue=true (try again) or false (drop / dead-letter it).
basic.qos (prefetch)"Give me at most N unacked messages at a time." The backpressure knob — prevents one consumer from hoarding the queue.
x-dead-letter-exchangeQueue argument: rejected/expired messages get republished to this exchange → your DLQ.
publisher confirmsBroker-side ack to the producer that the message is safely enqueued — the publish-side half of at-least-once.
# declare topology (idempotent), then consume — amqplib-style pseudocode
ch.assertExchange('orders', 'topic')
ch.assertQueue('email-worker', { durable: true, arguments: { 'x-queue-type': 'quorum' } })
ch.bindQueue('email-worker', 'orders', 'order.created.*')
ch.prefetch(10)                              // ≤10 unacked in flight
ch.consume('email-worker', msg => {
  sendEmail(msg)                             // do the work FIRST…
  ch.ack(msg)                                // …confirm LAST
})
ch.publish('orders', 'order.created.eu', body, { persistent: true })

SQS (HTTP API)

SendMessageEnqueue (batch up to 10). DelaySeconds ≤ 15 min for scheduled-ish delivery.
ReceiveMessagePoll up to 10 messages. WaitTimeSeconds: 20 = long polling — do this always; short polling burns money and misses messages on sparse queues.
(visibility timeout)Received message is hidden, not deleted — default 30 s, max 12 h. Nobody else sees it while you work.
DeleteMessageThe "ack": pass the ReceiptHandle from this receive. No delete before timeout ⇒ message reappears.
ChangeMessageVisibilityHeartbeat for long jobs: extend the timeout while still working.
RedrivePolicymaxReceiveCount per queue: after N failed receives the message moves to the configured DLQ automatically.
# the whole consumer lifecycle in three calls
aws sqs receive-message --queue-url $Q --wait-time-seconds 20 --max-number-of-messages 10
# …process each message; then, per message:
aws sqs delete-message --queue-url $Q --receipt-handle $H
# job takes longer than expected? extend the lock:
aws sqs change-message-visibility --queue-url $Q --receipt-handle $H --visibility-timeout 600

3 · Use cases — when a queue is the answer

🖼 Offload slow work from the request path
The canonical use. User uploads a video → API stores the blob, enqueues {videoId}, returns 202 in 50 ms. A fleet of transcoder workers chews through the queue at its own pace. The user-facing latency no longer depends on the heaviest job in the system.
Why a queue: retries, crash-safety and worker autoscaling come for free; the API tier stays stateless.
🌊 Buffer spiky writes (load leveling)
Flash sale: 50k orders/min for 10 minutes, but payment provider handles 5k/min. The queue absorbs the spike; workers drain it at the sustainable rate. Queue depth becomes your honest, real-time overload signal.
Why a queue: it converts "we fall over at peak" into "p99 completion time grows at peak" — a much better failure mode.
💀 Retry + dead-letter pipeline
Payment webhook processing fails on 0.1% of messages (malformed payloads, downstream 500s). With maxReceiveCount: 3 + DLQ, poison messages quarantine themselves instead of blocking the queue, and an alarm on DLQ depth pages a human.
Why a queue: the retry loop is broker-side — your worker code stays a straight line.
📣 Fan-out to many independent workers — SNS + SQS
One order.created event must trigger email, analytics, and inventory — three services with different speeds and failure modes. SQS alone can't fan out (one queue = one delivery); the AWS idiom is SNS topic → one SQS queue per subscriber. In RabbitMQ, a fanout/topic exchange bound to several queues does the same in-broker.
Why this shape: each subscriber gets its own retry/DLQ/backlog, so a slow consumer never starves a fast one.

When a queue is the WRONG answer

replay / audit"Re-run yesterday's events through the new fraud model" — impossible; consumed messages are gone. You want Kafka's retained log.
many readers of one streamFive teams each want every event. A queue delivers each message once total; you'd need SNS fan-out duct tape. Kafka consumer groups solve exactly this.
strict global orderingQueues + retries + parallel consumers reorder freely. FIFO modes exist but cap throughput (see deep dive). Per-key ordering at scale → Kafka partitions.
request/responseIf the caller waits for the result, that's RPC (gRPC/HTTP) — putting a queue in the middle just adds latency and failure modes.

4 · Distributed setup — what "running it for real" means

RabbitMQ: you run a cluster of 3 or 5 nodes (odd, because modern replicated queues use Raft). Exchange/binding metadata lives on every node. The queues themselves are the interesting part: a quorum queue is replicated across nodes with a Raft leader — a write is confirmed once a majority has it, so losing a minority of nodes loses nothing and failover is automatic. The older classic mirrored queues had ugly split-brain semantics and are deprecated/removed in RabbitMQ 4.x — mentioning "quorum queues, because mirrored queues were unsafe on partition" is a nice senior-level flex. For WAN links you don't stretch a cluster; you use federation/shovel to relay between clusters.

SQS: there is no cluster to design — that's the point. Messages are stored redundantly across multiple AZs, and the service shards internally to give standard queues practically unlimited throughput. The costs of that elasticity are the two properties people forget: standard queues are best-effort ordering (messages can arrive out of order) and at-least-once (occasionally a duplicate even without a consumer crash — the redundant storage layer itself may double-deliver on an internal failover).

The universal gotcha — at-least-once is contagious. Whatever the broker promises, your system is at-least-once end-to-end: producers retry on ambiguous timeouts (duplicate enqueue), brokers redeliver on missing acks (duplicate delivery). So the consumer must be idempotent — dedupe on a business key (orderId), use conditional writes, or make the operation naturally idempotent ("set status=paid" not "increment balance"). Say this before the interviewer asks.

5 · 🎛 Play with it — visibility timeout, redelivery & DLQ

Crash a worker mid-job and watch the broker redeliver. A poison message fails forever — until maxReceiveCount: 3 exiles it to the DLQ. Toggle FIFO to see head-of-line blocking.

What to try

  1. Send 3 → Tick → both workers grab one; the messages turn ⏳ in-flight, not deleted.
  2. Tick twice more → work done → DeleteMessage → gone for good.
  3. Send 3 → Tick → 💥 Crash worker-2 → keep ticking: its message's timeout expires and it is redelivered (receive count grows).
  4. ☠️ Poison → it fails, redelivers, fails… third receive sends it to the DLQ.
  5. FIFO on → send 3 + poison first: nothing behind the poison moves until it dead-letters — head-of-line blocking.
🔍 Deep dive · At-least-once, duplicates, and the idempotency tax click to expand

Why can't the broker just deliver exactly once? Trace one crash: the worker receives message #42, charges the customer, and dies one instruction before the ack. The broker sees only "no ack". Redeliver → double charge. Don't redeliver → maybe the crash was before the charge, and the order is silently lost. With acknowledgement-based delivery over a lossy network, you must choose: at-least-once (redeliver) or at-most-once (don't). Queues choose at-least-once, because losing work is usually worse than repeating it.

"Exactly-once" in marketing copy means exactly-once processing, built from at-least-once delivery + deduplication. SQS FIFO dedupes on MessageDeduplicationId — but only within a 5-minute window, and only against duplicate sends. A consumer crash after processing still causes a reprocess. The honest stack is: broker gives at-least-once; your consumer supplies idempotency — a dedup table keyed by business ID (insert-if-absent in the same transaction as the work), conditional writes, or naturally idempotent operations.

One more trap: visibility timeout shorter than processing time. Job takes 90 s, timeout is 30 s → at t+30 the broker assumes death and redelivers → two workers now run the same job concurrently, both eventually "succeed", and you've built a duplicate generator. Fix: timeout ≈ several × p99 processing time, plus a ChangeMessageVisibility heartbeat for long jobs.

🔍 Deep dive · Ordering — why FIFO costs you your throughput click to expand

A standard queue with parallel consumers cannot preserve order, and it's not sloppiness: worker A gets msg 1, worker B gets msg 2, B finishes first — order inverted. Add a redelivery (msg 1 crashes and returns 30 s later) and messages time-travel. Any system with retries + parallelism reorders; the only question is whether you care.

When you do care, you serialize — and serialization is the enemy of throughput. SQS FIFO queues make the trade explicit with MessageGroupId: strict order within a group, parallelism only across groups. One group = one worker at a time = one stuck message stalls everything behind it (head-of-line blocking — try it in the simulator). The design skill is choosing a group key with high cardinality but meaningful order: userId, orderId, chatRoomId — "per-user order" is almost always what the product actually needs, not global order.

RabbitMQ gives FIFO within a queue only for the happy path — one consumer, no requeues. A nack(requeue=true) puts the message back at the head-ish, redeliveries interleave, and with prefetch > 1 across consumers, ordering is gone. Kafka's answer (order per partition via key hashing) is the same idea as message groups, scaled up — which is why "I need per-key ordering at 100k msg/s" is usually the moment to switch technologies.

🔍 Deep dive · RabbitMQ internals — queues are processes, Raft replaced mirrors click to expand

RabbitMQ is an Erlang system, and it shows: every queue is a single Erlang process — an actor with a mailbox. That's the source of both its elegance and its scaling ceiling: one queue executes on roughly one CPU core, so a single queue tops out in the low tens of thousands of msg/s no matter how big the machine is. The idiomatic scale-out is more queues (per tenant, per shard, per priority), not a bigger queue — RabbitMQ happily runs hundreds of thousands of queues.

Replication history is an interview story in itself. Classic mirrored queues replicated by shipping operations to mirrors, and on a network partition could lose acked messages or split-brain; the workarounds (pause_minority) traded availability to duck the problem. Quorum queues (3.8+) rebuilt replication on Raft: a publish is confirmed only after a majority of replicas persist it, leader failover is safe by construction, and the cost is honest — higher latency per message and memory/disk for the log. Mirrored queues are gone in 4.x; if you say "RabbitMQ replication" in 2026, you mean quorum queues.

Two more mechanisms worth naming: credit-based flow control — a fast publisher is throttled (its connection literally blocks) when queues or memory fall behind, which surprises teams expecting "fire and forget"; and memory alarms — past a watermark the broker blocks all publishers until consumers drain the backlog. Both are RabbitMQ choosing to slow producers rather than die — but it means a huge backlog degrades the whole broker, not just one queue. Deep backlogs are Kafka's home turf, not RabbitMQ's.

🔍 Deep dive · Backpressure, DLQ hygiene & retry design click to expand

Queue depth is the most honest metric in your system. Latency dashboards lie under load; the integral of (arrival rate − service rate) doesn't. Alert on depth and on oldest-message age (SQS: ApproximateAgeOfOldestMessage) — depth alone misses a stuck FIFO group, age catches it. Autoscale workers on depth-per-worker; scale in slowly to avoid flapping.

Retries need a budget and spacing. Immediate requeue on failure = a hot loop hammering the same broken dependency. The pattern: retry with exponential backoff by routing failures through delay — SQS: per-message DelaySeconds on re-send, or tiered retry queues (retry-1m, retry-10m); RabbitMQ: a TTL'd retry queue whose DLX points back at the work queue (messages "cook" in the retry queue, then reappear). After N attempts → DLQ.

A DLQ without a process is a landfill. The senior answer covers: an alarm on DLQ > 0; a triage runbook (is it a poison payload, a bug, or a downed dependency?); a redrive path to send fixed messages back to the source queue (SQS has one-click redrive); and retention long enough to survive a long weekend (14 days, not 4). Interviewers reward "and DLQ depth pages us, with a redrive runbook" far more than the mere word "DLQ".

⚖️ At scale — numbers, limits & failure modes

Figures to quote in capacity talk. Round aggressively — order of magnitude is the point.

SQS standardEffectively unlimited throughput (horizontally sharded by AWS); at-least-once; best-effort ordering. Payload ≤ 256 KB (bigger → pointer to S3). Batch ≤ 10 msgs/call.
SQS FIFO300 msg/s bare, ~3,000 msg/s with batching; high-throughput mode raises it to tens of thousands msg/s (region-dependent — quote the order, not the digit). Order per MessageGroupId; dedup window 5 min.
SQS timersVisibility timeout: default 30 s, max 12 h. Retention: default 4 days, max 14 days. Long poll ≤ 20 s. DelaySeconds ≤ 15 min.
RabbitMQ / queueOne queue ≈ one Erlang process ≈ one core: plan ~10–40k msg/s per quorum queue (small persistent messages). Scale = more queues, not bigger ones. A cluster of 3–5 nodes with many queues reaches hundreds of thousands msg/s.
RabbitMQ limitsMessage size: keep ≤ 1 MB (default hard cap 16 MiB in modern versions) — blobs go to object storage, queue carries the pointer. Quorum queues: odd replica count (3 or 5), majority must be up to accept publishes.
latencyRabbitMQ in-DC: ~1–5 ms publish→deliver (quorum confirm adds fsync+majority). SQS: ~10–100 ms per API call — it's an HTTPS service; batch and long-poll or pay per-call latency and per-request pricing.
Failure modes that only appear at scale: backlog cliff (RabbitMQ past its memory watermark blocks ALL publishers — a deep backlog degrades the broker, not just one queue); redelivery storm (mass worker crash/deploy → thousands of visibility timeouts expire together → thundering-herd reprocessing); FIFO group starvation (one hot MessageGroupId serializes; one poison message halts its whole group); visibility timeout < p99 processing time (silent duplicate factory); empty-receive bill (short polling a sparse queue = paying for millions of empty responses — always long-poll).

8 · In the interview

How it comes up: almost never as "design a queue" — always as a component you introduce yourself: "uploads are processed async through a queue". The follow-ups are predictable: what if the worker dies? (visibility timeout / redelivery), what if it dies repeatedly? (DLQ + alarm), can work happen twice? (yes — idempotent consumer), is order preserved? (no — and here's why that's fine / here's the FIFO trade).

The choice script: default to SQS in an AWS-flavored design ("managed, infinite scale, nothing to operate"). Name RabbitMQ when you need its routing (topic exchanges, per-tenant queues), on-prem/self-hosted constraints, or sub-5 ms broker latency. Switch to Kafka the moment the requirements say replay, multiple independent readers, event sourcing, or per-key ordering at high throughput. Saying the boundary out loud is the senior signal — the technologies are easy, the cutover criteria are the skill.

Alternatives — one line each

KafkaNot a queue — a retained, partitioned log. Replay, many reader groups, per-key order. Worse at: per-message ack/DLQ ergonomics, work-stealing across consumers.

Phrases that score: "consumers are idempotent, keyed on orderId, so at-least-once is safe" · "visibility timeout is 3× p99 processing time, with heartbeat extension" · "DLQ after 3 receives, alarmed, with a redrive runbook" · "standard queue — we don't need ordering, so we keep unlimited throughput" · "if we later need replay, this boundary is where Kafka replaces SQS".

9 · Check yourself

A worker crashes after charging the customer but before deleting the message. What happens, and whose job is it to fix?

The visibility timeout expires and the message is redelivered — the broker cannot know the work was done. A second worker charges again unless the consumer is idempotent: e.g. insert into a processed-payments table keyed by orderId in the same transaction as the charge, and skip if present. Fixing this is always the consumer's job — no broker setting eliminates it.

Why can't SQS standard queues guarantee ordering, and what's the real cost of switching to FIFO?

Standard queues are sharded across many storage hosts and serve parallel consumers with retries — reordering is inherent. FIFO restores order per MessageGroupId by serializing each group: throughput per group collapses to one-in-flight-at-a-time, and a poison message causes head-of-line blocking for its whole group. Cross-group parallelism is the only scaling dimension, so group-key cardinality becomes a design decision.

When would you pick RabbitMQ over SQS, concretely?

(1) Routing logic in the broker — topic exchanges fanning one event into per-team/per-tenant queues; (2) self-hosted / non-AWS / on-prem constraints; (3) latency — single-digit ms broker vs tens of ms per HTTPS call; (4) protocol needs (AMQP/MQTT/STOMP). Otherwise SQS wins on ops: nothing to patch, no capacity planning, per-request pricing.

What breaks first when a RabbitMQ backlog grows into the tens of millions of messages?

Memory. The broker hits its memory watermark and blocks all publishers (flow control), effectively an outage for producers; paging messages to disk saves memory but slows the queue further. Deep, long-lived backlogs are the signature workload of a log (Kafka: sequential disk, backlog size irrelevant to broker health) — quote this as the migration trigger.

Interviewer: "So we get exactly-once with SQS FIFO, right?"

No — FIFO gives exactly-once enqueue within a 5-minute dedup window (dedupes retried sends) plus ordered delivery. A consumer crash after processing but before delete still reprocesses. End-to-end exactly-once = at-least-once delivery + idempotent processing; there is no configuration that replaces the dedup table.
