🗃️ B-Trees & B+Trees — how DB indexes stay shallow

A balanced, high-fan-out tree built to minimize disk I/O. Watch keys route down, nodes split on overflow (median pushed up), and range scans walk the linked leaf chain.
B+tree high fan-out always balanced data only in leaves linked leaves

A B+tree is the data structure behind almost every relational database index (Postgres, MySQL/InnoDB). Each node holds many sorted keys (here: order 4 → up to 3 keys per node, 4 children). Internal nodes are pure signposts; the actual data/row-pointers live only in leaves, and the leaves are linked left→right so a range scan is a single sequential walk. High fan-out keeps the tree just a few levels deep, so a lookup is only a handful of block reads.

The gotcha: a binary tree on disk is too tall — depth ≈ log₂(N) means many random I/Os, and each hop is a separate disk seek. B-trees trade cheap in-RAM key comparisons for far fewer block reads (depth ≈ log_fanout(N)) — that's the whole point: minimize disk I/O. Most things called "B-tree indexes" are really B+trees (data in leaves + linked leaves). Contrast with an LSM-tree (Cassandra/RocksDB): write-optimized via append + background compaction, vs the B+tree's read-friendly, in-place-update layout.

What you're seeing

  1. Insert: route from the root down to a leaf, comparing keys at each level.
  2. If a leaf overflows (>3 keys), it splits; the median is copied/pushed up to the parent.
  3. Splits cascade upward; if the root splits, the tree grows one level taller — rarely.
  4. Search follows the same route to a single leaf — visited nodes = disk reads ≈ tree height.
  5. Range scan: find lo, then walk the linked leaf chain →→ until hi. No re-descending the tree.
