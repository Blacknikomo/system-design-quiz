🗂️ Hash Tables, Maps & Sets

A string hash maps each key to a bucket index. Insert, look up, and delete; watch two keys collide into one bucket's chain, and the table resize + rehash when the load factor crosses 0.75.
O(1) average separate chaining load factor → resize unordered

A hash table stores entries in an array of buckets. A hash function turns a key into an integer; bucket = hash(key) % numBuckets. Two keys can land in the same bucket — a collision — so each bucket holds a small list (a chain). As entries pile up, the load factor = entries ÷ buckets rises; when it crosses ~0.75 we resize (double the buckets) and rehash every entry into the bigger array.

The gotcha: O(1) is average / amortized, not guaranteed. Worst case is O(n) — all keys collide into one chain, or a resize rehashes every entry (that single insert is O(n), but spread out it amortizes to O(1)). Hash maps are unordered — if you need sorted iteration or range queries, use a balanced tree / sorted set instead. And adversaries can force collisions on purpose (HashDoS) — production hashes are seeded/randomized to defend against it.
Beyond chaining: open addressing (linear/quadratic probing, Robin Hood hashing) stores collisions in the table itself instead of chaining; and Java 8+ converts a long collision chain into a red-black tree, capping a bad bucket at O(log n) instead of O(n).

What you're seeing

  1. hash(key) → integer → % numBuckets → bucket index.
  2. Insert appends {key,val} to that bucket's chain (update if key exists).
  3. Two keys in the same bucket = a collision; the chain grows, lookups walk it.
  4. Load factor = entries ÷ buckets; crossing 0.75 triggers a resize.
  5. Resize doubles buckets and rehashes every entry — O(n), but amortized O(1) per insert.
