⚖️ Scaling — Consistent Hashing & Rebalancing

The mechanic behind every horizontally-sharded system. Add or remove a node and watch how many keys have to move — and why consistent hashing (Cassandra ring, DynamoDB partitions, Redis Cluster slots) beats naive hash % N. Second tab: Kafka consumer-group rebalancing.
consistent hashing virtual nodes minimal key movement consumer rebalance

Naive sharding (hash(key) % N) breaks the moment N changes: almost every key remaps to a new node, so adding one machine triggers a near-total reshuffle. Consistent hashing places keys and nodes on a ring; a key belongs to the next node clockwise, so adding/removing a node only moves the ~1/N of keys adjacent to the change. Virtual nodes give each physical node many points on the ring so load stays even and a lost node's keys spread across all survivors — not onto one neighbor.

The gotcha: without virtual nodes, ranges are uneven (one node owns a huge arc) and removing a node dumps all its keys onto a single successor — an instant hotspot. With vnodes, both problems vanish. This is exactly why Cassandra and Dynamo use vnodes.

What you're seeing

  1. Keys and nodes hash onto a ring (0…1). A key is owned by the next node clockwise.
  2. Add a node → it only steals the keys in the arc just before it (~1/N). Everything else stays put.
  3. Remove a node → only its keys move, to their next clockwise successor.
  4. Virtual nodes spread each node across many ring points → even load + even redistribution.
  5. Compare with naive hash % N: the counter shows how many keys it would have moved (almost all).
