📊 Arrays & Dynamic Arrays — size vs. capacity

Contiguous, index-addressable memory. Watch the backing buffer double on overflow, see why append is amortized O(1), and feel the O(n) cost of shifting on insert/delete.
contiguous memory O(1) random access amortized O(1) append cache-friendly

A dynamic array (Go slice, C++ vector, Java ArrayList, Python list) stores elements in one contiguous block. It tracks two numbers: size (logical count) and capacity (slots in the backing buffer). Indexing is O(1) pointer arithmetic. When size == capacity, append allocates a larger buffer (commonly 1.5–2×, e.g. Java/C++ ~2×, Python ~1.125×) and copies everything over.

The gotcha: append is amortized O(1) — most appends are O(1), but the occasional resize is O(n); doubling makes the copies sum to a geometric series, so n appends cost O(n) total. Insert/delete at the front or middle is always O(n) because every following element must shift. The win is cache locality: contiguous layout means a linear scan streams through cache lines — usually beating a linked list in practice.

What you're seeing

  1. Access [i] jumps straight to a cell via address math — O(1), no scan.
  2. Append drops into the next free slot — O(1)… until the buffer is full.
  3. On overflow, a 2× buffer is allocated and all elements are copied — that one append is O(n).
  4. Insert @ i shifts elements i..end one slot right — O(n).
  5. Delete @ i shifts elements after i one slot left — O(n).
