🔢 Count-Min Sketch

A probabilistic frequency table: "how many times have I seen x?" in sublinear memory. It can overestimate a count, but never undercounts — the mirror image of a Bloom filter's "no false negatives".
probabilistic frequency d rows × w counters overestimate only O(d) add / query

A Count-Min Sketch is a 2-D array of counters with d rows and w columns (all 0), plus d independent hash functions — one per row, each mapping an item to a column in [0, w). Add(x, c): for every row i, increment count[i][hi(x)] += c — one counter per row. Estimate(x): read those same d counters and return the minimum. Collisions can only add to a counter (other items hashing to the same cell), so every row is an over-count; the smallest of them is the tightest upper bound — hence "count-min".

The gotcha: CMS only over-counts, never under-counts — flip that in an interview and it's an instant tell (it's the exact mirror of Bloom's "no false negatives"). Error is absolute: est(x) ≤ true(x) + ε·N with probability ≥ 1−δ, where N is the total of all counts. So heavy hitters stay accurate but rare items can be swamped by collisions from frequent ones. Pick w = ⌈e/ε⌉ (wider ⇒ less error per estimate) and d = ⌈ln(1/δ)⌉ (deeper ⇒ lower chance of a bad estimate). You can't enumerate keys (like Bloom) — pair it with a min-heap to track top-K / heavy hitters. Deletes are only safe on a strictly non-negative stream (decrements can violate the never-undercount guarantee → use Count-Mean-Min or a signed variant). Cousins: Bloom filter (membership), HyperLogLog (distinct cardinality).
Add(x, c)O(d) — increment one counter per row
Estimate(x)O(d) — read d counters, return the min
SpaceO(d·w) counters — independent of stream length & distinct-key count
Errorest ≤ true + ε·N w.p. ≥ 1−δ, with w=⌈e/ε⌉, d=⌈ln 1/δ⌉

What you're seeing

  1. Add(x, c): each row's hash picks one column; that single counter is bumped by c. d rows → d counters touched.
  2. Estimate(x): read the same d counters. The minimum (green) is the answer — the least-inflated row.
  3. Every counter is ≥ the true count, because other items may have hashed into the same cell. So the estimate is never too low.
  4. The error = estimate − true count. It comes entirely from collisions; the min picks whichever row got the fewest.
  5. Shrink w (more crowding) to see error grow; add rows (d) to lower the odds that every row collided.
