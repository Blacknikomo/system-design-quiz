⏳ Temporal & Durable Workflows — code that survives crashes

Durable execution: event-sourced workflow histories, replay, deterministic code, sagas as try/catch — and where Step Functions and Airflow end and Temporal begins. ☕ ~15 min read
durable execution event sourcing replay determinism saga durable timers

Temporal runs multi-step business logic as durable execution: your workflow code is deterministic orchestration, every side effect goes through an activity, and every step is event-sourced into a server-side history — so when the process dies mid-flow, another worker replays the history, lands in exactly the same state, and keeps going. The program outlives the process.

In the interview: the moment your design has 3+ sequential side effects with retries, waits, or human steps — charge → reserve → ship → email — name durable workflow orchestration instead of hand-rolling queues + cron sweepers + a state-machine table. The interviewer then expects you to explain, unprompted: replay (activities are NOT re-executed), the determinism contract, at-least-once activities → idempotency, and the boundary vs Step Functions / Airflow.

1 · The mental model: a chess game with a written move log

Start with the problem. An order comes in: charge the card → reserve inventory → ship → email the customer. Four calls to four services, any of which can fail, timeout, or half-succeed. The charge goes through and the process crashes before reserving — now you have money and no order. So you build the usual scaffolding: a state-machine table (status = 'CHARGED'), a queue per step, a cron sweeper that finds rows stuck in limbo, retry counters, a dedupe table, and compensation scripts for support to run by hand. Congratulations: you have written a workflow engine. Badly. Every team in the company has written this engine, badly, and each copy has its own novel bugs. That is the pitch for durable execution: extract the engine once, make it correct, let application code go back to being a plain function.

The mental model is a chess game with a written move log. The rules of chess are deterministic (your workflow code); the moves played so far are written in a ledger (the event history, kept on the Temporal server); a player (a worker — your ordinary stateless process) sits at the board. If the player leaves — crashes, gets deployed over, loses power — the game is not lost. Any other player can pick up the ledger, replay the moves on a fresh board, arrive at exactly the same position, and play on. Nothing about the game lives in the player's head; the ledger is the game.

One rule makes this work. Some "moves" are dice rolls — calls into the messy outside world: charging a card, calling an API, reading the clock. Those are activities, and their results get written into the ledger. When a new player replays the game, they do not re-roll the dice — they read the recorded roll and move on. Concretely: on replay, the workflow function runs again from the top, but every await chargeCard(...) whose result already exists in history returns instantly, with the recorded result. The card is not charged twice. Replay re-executes decisions, never side effects. Execution resumes for real only at the first step with no recorded result.

This is why the determinism contract exists: replay only reconstructs the right state if the code takes the same path given the same history. So workflow code must not read the wall clock, generate randomness, do I/O, or iterate a hash map in nondeterministic order. The SDK gives you deterministic substitutes — workflow time, seeded random, sideEffect — and everything else goes in an activity. Break the rule and replay diverges from the ledger: the SDK detects the mismatch and fails the workflow task with a non-determinism error rather than corrupting state.

Misconception to kill early: the Temporal server never runs your code. It stores histories, timers and task queues. Your workers — plain processes you deploy, scale and secure like any service — long-poll task queues, execute workflow and activity code, and report results back. The server is the ledger and the scheduler; the workers are the players.

2 · Syntax & core API

The vocabulary below is Temporal's (TypeScript SDK flavor), but Cadence, Azure Durable Functions and the newer engines share the concepts almost 1:1.

activityThe unit of side effect — any I/O lives here. Auto-retried per policy (initial interval, backoff coefficient, max attempts); each invocation's result is recorded in history.
retry policyPer-activity: { initialInterval: '1s', backoffCoefficient: 2, maximumAttempts: 5 }. Default is retry-forever with capped backoff — set limits or a downstream outage becomes an infinite retry storm.
sleep('30 days')Durable timer — a history event plus a server-side timer, not a thread. Survives worker restarts, deploys, and the server's own failover. Waiting a month costs ~nothing.
signalAsync input into a running workflow — "manager approved", "payment webhook arrived", "cancel". Delivered through history, so it's never lost even if no worker is up.
queryRead live workflow state (e.g. "which step is this order on?") without perturbing it. (Newer SDKs also have update: a synchronous, validated mutate-and-respond.)
child workflowA workflow started by a workflow — fan-out, per-item lifecycles, separate histories and retries.
continueAsNewClose this execution and atomically start a fresh one with carried-over args — the escape hatch for the history size cap (loops, entity workflows).
task queueNamed queue workers poll. Routing (GPU work → GPU workers), rate control, and deploy isolation happen here.
heartbeatLong activities report liveness + progress; a missed heartbeatTimeout fails the attempt fast instead of waiting out a huge timeout, and the recorded progress lets the retry resume mid-list.
// order fulfillment as a SAGA — compensations are just a stack + try/catch
const { chargeCard, reserveInventory, shipOrder, sendEmail,
        refundCard, releaseInventory } = proxyActivities({
  startToCloseTimeout: '30s',
  retry: { initialInterval: '1s', backoffCoefficient: 2, maximumAttempts: 5 },
});

export async function orderWorkflow(order) {
  const compensations = [];
  try {
    await chargeCard(order);                    // activity → result recorded in history
    compensations.push(() => refundCard(order));
    await reserveInventory(order);
    compensations.push(() => releaseInventory(order));
    await sleep('2 days');                      // durable timer — survives every restart
    await shipOrder(order);
    await sendEmail(order, 'shipped');
    return 'fulfilled';
  } catch (err) {
    for (const undo of compensations.reverse()) // unwind in REVERSE order
      await undo();
    throw err;                                  // workflow ends failed-but-consistent
  }
}
// human-in-the-loop: wait for a signal OR a durable timeout — no polling anywhere
let approved = false;
setHandler(approveSignal, () => { approved = true; });

const inTime = await condition(() => approved, '7 days');  // blocks "for a week" durably
if (!inTime) {
  await escalateToManager(order);              // nobody clicked — escalate
}

3 · Use cases — when durable workflows are the answer

📦 Order fulfillment / booking sagas
Charge, reserve, ship, notify — across services that each fail independently. The workflow is the saga coordinator: forward steps as activities, compensations in the catch block, every state transition durable and auditable.
Why durable workflows: retries, recovery and compensation ordering come from the engine; the business logic reads top-to-bottom like the spec.
🧑‍⚖️ Human-in-the-loop approvals
KYC review, expense approval, content moderation: await condition(approved, '7 days'), remind at day 3 with a timer, escalate on timeout. The "process" may live for weeks; no thread, cron job, or state table exists anywhere.
Why durable workflows: signals + durable timers model "wait for a human, but not forever" in three lines — the DIY version is a cron sweeper reading a status table.
💳 Payment retries & dunning
Subscription renewal fails → retry tomorrow, then in 3 days, email a warning, downgrade after a week. A workflow per subscription cycle sleeps for days between attempts and keeps exact count of where it is.
Why durable workflows: the schedule is stateful — "attempt 3 of 4, warned once" — precisely what queues and cron are bad at expressing.
🏗 Infra provisioning & "cron with state"
Create network → database → deploy → DNS → verify, with rollback on failure; or a nightly reconciliation that must never double-run and must resume from item 40,231 after a crash (heartbeats carry the cursor).
Why durable workflows: long-running, resumable, idempotent orchestration is the whole product; this is what control planes are made of.

When it's the WRONG answer

high-throughput data pipelinesMillions of records/sec through transforms is stream processing — Flink/Spark. A workflow engine event-sources every step; per-record orchestration at that volume is cost and latency suicide.
one async job + retry"Resize the image, retry 3×" — a queue + worker + DLQ is enough. Orchestration earns its keep at multiple steps, long waits, or compensation logic.
sub-second request pathsEvery step is a durable write + task dispatch — think tens of ms of overhead per transition, not µs. Keep synchronous user-facing hops as plain RPC.
scheduled batch DAGs for data"Rebuild the warehouse tables at 2am, backfill last month" — that's Airflow/Dagster territory: schedule-centric, data-aware, ops tooling for engineers. See the boundary deep dive.

4 · Distributed setup — what "running it for real" means

A Temporal cluster is a set of stateless services over a database. Frontend terminates gRPC and routes; History is the heart — it owns workflow executions, appends events, and is sharded so millions of concurrent executions spread across nodes; Matching hosts the task queues workers poll; an internal worker service runs system logic. Underneath sits a persistence store — Cassandra or SQL (MySQL/Postgres) — holding histories and mutable state, with visibility (search/list "all failed order workflows for customer X") typically backed by Elasticsearch. One ops gotcha worth quoting: the history shard count is fixed at cluster creation and effectively sets the scaling ceiling — pick it generously. Or skip all of this: Temporal Cloud is the managed control plane, and your workers still run in your VPC — code and data plane stay yours.

Your side of the contract is workers: ordinary stateless processes (deploy on k8s, autoscale like anything else) that long-poll task queues, execute workflow/activity code, and report back. Nothing pushes to them; no inbound ports. For efficiency the SDK keeps recently-active workflows cached in worker memory (sticky execution) so most workflow tasks continue in-place without replay — a cache eviction or worker death is what triggers the full replay you'll watch in the simulator. Heritage, for the interview: this lineage runs AWS Simple Workflow → Cadence (built at Uber by Maxim Fateev & Samar Abbas) → Temporal, with Azure Durable Functions as the same idea grown at Microsoft.

The orchestration tax is real. Every step becomes durable writes to the history store plus task-queue hops — per-transition latency is milliseconds to tens of milliseconds, and cluster capacity is measured in state transitions per second, not workflows. Fine for business processes (an order doesn't care about 50 ms between steps); wrong for hot request paths and per-record data processing. Say this trade out loud before the interviewer pokes it.

🔍 Deep dive · Replay & the determinism contract click to expand

Walk one crash precisely. The workflow has completed chargeCard and reserveInventory; history holds their Scheduled/Completed events, results included. The worker dies. The server notices only an expired task; it re-dispatches the workflow task to another worker, with the history attached. The new worker runs orderWorkflow() from the first line. The first await chargeCard(order)? History says: completed, result {authId: "ch_0091"} — the await resolves immediately with that value; no charge happens. Same for the reserve. The code path arrives at shipOrder, finds no recorded result — this is where real execution resumes. Replay is milliseconds of pure computation; in steady state you rarely even pay it, because sticky execution keeps live workflows cached on a worker and replay happens only on eviction or death.

The contract that makes this sound: given the same history, the code must make the same calls in the same order. Forbidden in workflow code: Date.now(), Math.random(), UUID generation, network/disk I/O, environment reads, iteration with nondeterministic order, spawning raw threads. Allowed: the SDK's workflow.now() (the time is itself sourced from history), deterministic random/UUIDs, sideEffect (execute once, record the value), and activities for everything with a wire attached. SDKs detect divergence — "history says ActivityTaskScheduled(shipOrder), but the code just asked for sendEmail" — and fail the workflow task with a non-determinism error instead of corrupting state.

The subtle consequence: deploying new workflow code while old executions are in flight. A workflow sleeping for 30 days will replay against code deployed weeks later; if the new code takes a different path, that's non-determinism. Temporal's answers: patching/versioning APIs (branch on patched('fix-123') — old histories take the old branch, new executions the new one) and worker versioning (pin executions to worker build ids). Saying "long-lived workflows make deploys a compatibility problem, and here's the patching story" is a staff-level flex few candidates make.

🔍 Deep dive · What's exactly-once and what's at-least-once — don't oversell click to expand

The workflow's state machine advances effectively exactly-once: every transition is an event appended to one history in one database, replay is idempotent by construction, and two workers can't both own a workflow task. That's the honest strong claim. The equally honest weak one: activities are at-least-once. An activity can complete its side effect and crash before reporting; the server sees a timeout and retries; the outside world experiences the effect twice. No orchestration engine can fix this — it's the Two Generals problem wearing a lanyard — so activity code must be idempotent: pass an idempotency key (payment processors take one; workflowId + activityId is a natural choice), use conditional writes, or make the operation naturally idempotent ("set status=shipped", not "increment shipments").

Know the timeout taxonomy, because it's where reliability is actually configured: startToClose (one attempt's budget — the one you should always set), scheduleToClose (all attempts including retries), scheduleToStart (how long a task may wait unclaimed — your "no workers polling" alarm), and heartbeatTimeout (fast death detection for long activities, with recorded progress so a retry resumes at item 40,231 instead of item 0).

And the default that bites: Temporal's default retry policy retries activities indefinitely with exponentially growing, capped backoff. Wonderful for riding out a 20-minute dependency outage; dangerous if a bug makes an activity fail forever — the workflow just… waits, retrying. Production hygiene: set maximumAttempts or scheduleToClose, mark non-retryable error types (a 400 will never succeed on retry), and alert on workflows stuck in retry loops.

🔍 Deep dive · History limits, continue-as-new & entity workflows click to expand

Everything durable has a price: the history must stay loadable and replayable. Temporal caps a single execution's history at roughly tens of thousands of events / ~50 MB (warnings fire well before the hard limits). A workflow looping "process signal, sleep, repeat" forever will hit the cap — the escape hatch is continue-as-new: close this execution and atomically start a fresh one with the accumulated state passed as arguments. Same workflow id, new run, history back to a handful of events. Any workflow with an unbounded loop should continue-as-new on a schedule (every N iterations / when the SDK suggests it).

That pattern unlocks the entity workflow (actor) style: one long-lived workflow per user / order / subscription, holding its state in workflow variables, mutated by signals, read by queries, rolling over via continue-as-new for years. It's an actor model where the mailbox and state survive any crash — many teams use it as a replacement for "status columns plus sweepers" entirely.

Payloads have limits too — single arguments/results are capped around a couple of MB (and the whole gRPC message not much more). The idiom is the same as with queues: pass references, not data — the S3/GCS key travels through history; the bytes never do. Design smell to call out: an activity returning a 50 MB report into workflow state. History is a ledger, not a bus.

🔍 Deep dive · The boundary: Temporal vs Step Functions vs Airflow click to expand

Three tools get conflated because all draw boxes and arrows. Separate them by what flows and what triggers. Airflow/Dagster: data flows, a schedule triggers — batch DAGs, backfills, sensor-and-retry ops tooling for data engineering. It is not built for millions of concurrent per-order executions or waiting on a human. Temporal / Step Functions: business transactions flow, events trigger — one execution per order/user/payment, thousands starting per second, living seconds to months.

Within transactional orchestration, the split is code vs configuration. AWS Step Functions defines the state machine in JSON (Amazon States Language): fully managed, IAM-integrated, direct service integrations, per-state-transition pricing, and two flavors — Standard (durable, up to 1 year, exactly-once-style execution semantics, history capped at ~25k events) and Express (up to 5 minutes, at-least-once, high-throughput, priced by duration). It's a superb answer inside AWS for moderately-sized flows. It gets painful where JSON is a bad programming language: complex branching, collections, compensation stacks, versioning of long-lived executions, local testing.

Temporal is the inverse: the workflow is code (loops, types, unit tests, your IDE), signals/queries/timers are first-class, it's portable (self-host or Temporal Cloud) — and you own workers plus a nontrivial dependency. The interview script: "all-in on AWS with a modest state machine → Step Functions; complex, long-lived, signal-heavy business logic or multi-cloud → Temporal; scheduled data DAGs → Airflow." Bonus point: they converge — Step Functions is what you get when orchestration is configuration, Temporal when it's code.

⚖️ At scale — numbers, limits & failure modes

Order-of-magnitude figures for capacity talk — Temporal's own limits move between versions, so quote the order, not the digit.

history / execution~tens of thousands of events / ~50 MB hard ceiling per execution (warnings earlier) → loops and entities must continue-as-new.
payloadsSingle payloads capped around ~2 MB (gRPC message ~4 MB) — pass S3/blob references through history, never bytes.
timersDurable sleeps of days–months are routine; a sleeping workflow consumes no worker resources — only rows in the store.
step overheadEach transition = durable writes + task dispatch: ms to tens of ms per hop. Workflows are for business time, not request time.
throughputCapacity is state transitions/sec (one activity ≈ a handful of transitions). Scales with history shards + the persistence tier; shard count is fixed at cluster creation.
workersStateless, horizontally scaled; autoscale on task-queue backlog. Schedule-to-start latency is the metric that says "not enough workers".
retentionClosed-workflow histories retained for a configurable window (typically days–months), then deleted or archived — history doubles as an audit log while it lives.
Failure modes that only appear at scale: non-deterministic deploys (new code breaks replay of in-flight executions — patch/version instead); history blow-up (unbounded loops without continue-as-new); fat payloads (MB-sized results dragged through every replay); silent stalls (no worker polling a task queue — nothing errors, everything waits; alert on schedule-to-start); retry storms (default infinite retries × a downed dependency × thousands of workflows — cap attempts, mark non-retryables); hot entity (one workflow receiving a signal flood serializes on its single history).

8 · In the interview

How it comes up: rarely as "design Temporal" — usually you invoke it inside "design checkout / subscription billing / trip booking / KYC onboarding", the moment the design needs multi-step side effects with retries, long waits, or compensation. The follow-ups: what if the worker dies mid-flow? (replay from history; activities not re-executed), can a step run twice? (activities at-least-once → idempotency keys), how do you wait 30 days? (durable timer, not a thread), how do you change code with workflows in flight? (patching/versioning), why not just queues? (you'd rebuild state, retries, timers, recovery and audit by hand — the DIY engine).

The choice script: single async job → queue + DLQ. Modest state machine, all-in on AWS → Step Functions. Complex, long-lived, signal-heavy business logic, or portability/self-hosting → Temporal. Scheduled data DAGs → Airflow/Dagster. BPMN-shaped org where analysts co-own diagrams → Camunda. Then the senior close: whatever the engine, activities stay idempotent and payloads stay references — the engine moves the hard parts, it doesn't delete them.

9 · Check yourself

A worker dies right after chargeCard succeeds. Is the customer charged twice? Walk the two sub-cases.

Case 1 — the result was recorded: ActivityTaskCompleted is in history; the new worker replays, the await resolves from history, no second charge. Case 2 — the charge succeeded but the worker died before reporting: the server only sees a timed-out attempt and retries the activity — at-least-once. The retry re-calls the payment provider, so the activity must be idempotent (idempotency key, e.g. workflowId+activityId, which processors accept). Replay protects you from re-running recorded steps; idempotency protects you from re-running unreported ones.

Why can't workflow code call Date.now() or make an HTTP request directly? What exactly breaks?

Replay re-executes the code against the recorded history and must reproduce the same decisions in the same order. A raw clock or network call returns a different value on replay → the code branches differently → the commands it issues no longer match the history → the SDK raises a non-determinism error (better than silently corrupting state). Fix: workflow.now()/deterministic random (values sourced from history), sideEffect for one-shot recorded values, and activities for all I/O.

Interviewer: "So Temporal gives us exactly-once, right?"

Split the claim. Workflow state transitions are effectively exactly-once — event-sourced, single history, idempotent replay. Activity side effects are at-least-once: completion can happen without being reported, and the retry re-executes. So the engine gives exactly-once orchestration over at-least-once actions, and your activities still need idempotency. Overselling this is the classic way to lose the room.

What is continue-as-new, and when are you forced to use it?

It atomically closes the current execution and starts a fresh one (same workflow id, new run) with state carried in the arguments — resetting the history. Forced whenever history would grow unboundedly: polling/processing loops, entity workflows alive for months, signal-heavy actors — because a single execution's history is capped around tens of thousands of events / ~50 MB, and replay must stay cheap.

A 12-step KYC flow with human review, on AWS. Step Functions or Temporal — and what tips it?

Both can do it. Step Functions if the org is all-in on AWS and the flow stays a modest state machine: zero infra, IAM-native, Standard supports year-long waits and callbacks. Temporal tips when the logic outgrows JSON — complex branching and data shaping, rich signal/query interaction, real unit tests, or portability requirements — and when executions must evolve safely over months (patching/versioning). The tiebreaker is honest: "who maintains this, and is the logic configuration or code?"
