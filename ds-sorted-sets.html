🌲 Sorted Sets, BST & Balanced Trees

A binary search tree keeps keys ordered so you get O(log n) search plus range queries, min/max, floor/ceiling — but only if it stays balanced. Watch a naive BST degenerate into a linked list.
ordered O(log n) if balanced range queries floor / ceiling red-black in prod

A binary search tree stores keys so that for every node, the left subtree is smaller and the right subtree is larger. That ordering buys you operations a hash map can't do: range queries, min/max, floor/ceiling, and ordered iteration — each in O(log n) when the tree is balanced. Search walks down comparing one node per level.

The gotcha: a plain BST is not self-balancing. Inserting keys in sorted order (1,2,3,…) creates a one-sided chain — effectively a linked list with O(n) operations. Production "sorted sets" use a self-balancing tree (red-black / AVL) to guarantee O(log n). The key design judgment: a hash map gives O(1) but is unordered; a sorted set costs O(log n) but gives you order + ranges. Pick based on whether you need range/ordering.

What you're seeing

  1. Insert compares the new key at each node: go left if smaller, right if larger, until you find an empty spot.
  2. Search follows the same path — one comparison per level. Depth = cost.
  3. Range query does an in-order walk, pruning subtrees that can't contain keys in [lo,hi] (purple = collected).
  4. Floor = largest key ≤ x; Ceiling = smallest key ≥ x — found by tracking candidates on the walk down.
  5. Sorted inserts only ever go right → a chain of height n. That's why production uses self-balancing trees.
