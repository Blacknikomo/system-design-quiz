♻️ LRU & LFU Caches

A cache with a bound: when it's full, something has to go. LRU evicts the least-recently-used entry; LFU evicts the least-frequently-used. Watch the hash map + doubly linked list keep every op O(1).
O(1) get / put hash map + doubly linked list eviction policy bounded memory

LRU = a hash map (key → node) plus a doubly linked list ordered most-recently-used at the head, least at the tail. get/put splice the touched node to the head; when over capacity you drop the tail. LFU instead groups nodes into frequency buckets and tracks minFreq — on access a node moves to the next-higher bucket, and eviction takes the oldest node from the lowest non-empty bucket.

The gotcha: the list must be doubly linked so you can unlink a known node in O(1) (a singly linked list forces an O(n) scan to find its predecessor — and "scan to find the oldest" misses the whole point). LRU's weakness is cache pollution: one big sequential scan of cold keys evicts your hot working set. LFU resists scan pollution but is more complex and can cling to stale once-popular items (needs aging / decay to fix).

Get(k)O(1) — map lookup + splice node to head (LRU) / bump bucket (LFU)
Put(k,v)O(1) — insert at head; if full, unlink tail / min-bucket node
EvictO(1) — drop a node whose position is already known
SpaceO(capacity)

Log

What you're seeing

  1. Get(k): hash-map lookup is O(1). Hit → move that node toward MRU; miss → nothing changes.
  2. Put(k,v): existing key updates in place; new key inserts at the front (LRU) or freq-1 bucket (LFU).
  3. When size > capacity, evict the LRU tail (LRU) or the oldest node in the minFreq bucket (LFU).
  4. Because the node's position is already known, unlinking it is O(1) — that's why the list is doubly linked.
  5. Run scan pollution to see a one-shot sequential scan flush an LRU's hot keys (LFU survives it).
