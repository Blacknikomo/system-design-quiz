🧲 Vector Databases — nearest neighbors at scale

Embeddings, ANN indexes (HNSW vs IVF), recall as an SLO, filtered & hybrid search — and the memory math that decides pgvector vs a dedicated engine. ☕ ~15 min read
embeddings ANN HNSW / IVF recall@k filtered search hybrid retrieval

An embedding model turns every document, image or user into a point in a ~100–3,000-dimensional space where distance ≈ meaning; a vector database answers exactly one query shape — "find the k points nearest to this one" — at millions-to-billions scale, by being deliberately, measurably wrong a few percent of the time. That trade — approximate results in exchange for 100–1000× less work — is the whole technology.

In the interview: reach for a vector DB the moment requirements say "semantic search", "similar items", "recommendations", or "RAG". The interviewer then expects you to handle, unprompted: the memory math (n × d × 4 bytes), recall vs latency tuning (efSearch / nprobe), the filtered-search trap (tenant predicates during traversal, not after), and the pragmatic ladder (brute force → pgvector → dedicated engine).

1 · The mental model: a map where distance is meaning

An embedding model is a function that eats a piece of content — a paragraph, a product photo, a user's listening history — and returns a list of a few hundred to a few thousand floats. The magic is in the training objective: things that mean similar end up near each other. "How do I reset my password" and "can't log into my account" share almost no words, yet their vectors sit side by side. Content becomes geography: every item is a pin on a map, and similarity becomes distance.

Once you buy that, every retrieval problem collapses into one query shape: k-nearest-neighbors (kNN) — "given this query point, return the k closest pins." And here is the uncomfortable part: none of your existing indexes can answer it. A B-tree needs one dimension to sort by — it answers "ordered" questions (ranges, prefixes) brilliantly, but in 1,536 dimensions there is no single sort order that keeps neighbors adjacent. An inverted index answers "which documents contain this exact token" — vocabulary, not geometry. Classic spatial trees (KD-trees, R-trees) work in 2–3 dimensions and degrade toward a full scan somewhere past a few dozen — the curse of dimensionality. This query genuinely needs a new index family.

The brute-force answer — compare the query to every vector — is exact kNN: O(n·d) per query. At 100k vectors that's a single matrix multiplication, instant and exact; do that and ship it. At 10M × 1,536 dims it's on the order of 1010 multiply-adds per query — hundreds of milliseconds to seconds of CPU each time, dead on arrival at real QPS. The escape hatch is to give up exactness: an Approximate Nearest Neighbor (ANN) index returns k neighbors that overlap the true k almost always — the usual operating band is 95–99% recall — for a tiny fraction of the work. Approximation isn't an implementation shortcut; it's the product contract. Recall becomes an SLO you measure and tune, exactly like p99 latency.

Two index families dominate. HNSW builds a navigable graph over the points with express layers on top — the skip-list idea lifted into high dimensions: coarse jumps up high, precise steps at the bottom. IVF partitions the space into k-means cells and, at query time, searches only the few cells nearest the query. Both are pure CPU + RAM structures — no GPU needed for serving. The simulator below makes their trade-offs physical.

One more piece of vocabulary: distance metrics. Cosine compares angle only; dot product mixes angle with magnitude; L2 is straight-line distance. For normalized vectors (most text-embedding models ship them normalized) all three produce the same ranking — on the unit sphere, ‖a−b‖² = 2 − 2·(a·b). So don't agonize: use whatever the model card says, and make sure index and query use the same one.

The one-sentence test. "Things like this" → vector search. "Things containing exactly this" → inverted index / BM25. "The record with id / status = X" → B-tree. Saying out loud which of the three index families a query needs is half the design.

2 · Syntax & core API

You won't write SQL in the interview, but naming the knobs — ef_search, probes, "filter during traversal" — signals you've operated one of these, not just read about it.

pgvector (Postgres extension)

vector(1536)Column type: float32, 4 B per dimension (halfvec = fp16, 2 B/dim). Vectors live in ordinary rows next to your data.
<=> · <-> · <#>Distance operators: cosine distance / L2 / negative inner product. Must match the ops class the index was built with.
USING hnswBuild an HNSW index. m = graph links per node (default 16), ef_construction = build-time candidate list (default 64). Slower to build, best query speed/recall, handles inserts.
hnsw.ef_searchQuery-time candidate-list size (default 40) — the recall ↔ latency dial. Set per session or per query.
USING ivfflatIVF index: lists = number of k-means cells. Rule of thumb: rows/1000 up to ~1M rows, ~√rows beyond. Fast build, lower memory, needs data present at build time (it trains centroids).
ivfflat.probesCells probed per query — default 1, which is a recall trap. Raise toward √lists and measure.
iterative scansRecent pgvector versions keep scanning the index until enough rows survive your WHERE filter — the fix for filtered queries coming back short.
-- one datastore: your rows, your filters, your vectors
CREATE EXTENSION vector;
CREATE TABLE docs (
  id        bigserial PRIMARY KEY,
  tenant_id int NOT NULL,
  body      text,
  embedding vector(1536)              -- 1536 × 4 B ≈ 6 KB per row
);
CREATE INDEX ON docs USING hnsw (embedding vector_cosine_ops)
  WITH (m = 16, ef_construction = 64);    -- defaults shown

SET hnsw.ef_search = 100;                 -- recall dial (default 40)
SELECT id, body FROM docs
WHERE tenant_id = 42                      -- filter + ANN: see deep dive 3
ORDER BY embedding <=> $query             -- cosine distance
LIMIT 5;

3 · Use cases — when a vector DB is the answer

🤖 RAG — retrieval for an LLM
Chunk your docs (~200–500 tokens each), embed every chunk, store. At question time: embed the question, fetch the top 5–20 nearest chunks, stuff them into the prompt. The LLM answers from your current, private data instead of its training set.
Why a vector DB: freshness without retraining, ACL filters per user, and the retrieval quality ceiling is set here — hybrid + rerank (deep dive 4) is the highest-ROI lever in most RAG stacks.
🔎 Semantic search over tickets / docs / code
"Payment declined after card renewal" should find tickets that say "transaction failed following expiry update". Keyword search can't; embeddings can. Ship it hybrid: BM25 still wins on error codes, SKUs and function names.
Why a vector DB: paraphrase and synonym matching is exactly what embedding geometry encodes — and what inverted indexes structurally miss.
🛍 "More like this" — recommendation candidates
Embed items (content or two-tower behavioral models); "similar items" = kNN around the item; "for you" = kNN around the user's vector. ANN generates a few hundred candidates cheaply; a heavier ranking model reorders them.
Why a vector DB: candidate generation must scan the whole catalog in milliseconds — precisely the ANN contract. This is the classic recsys retrieval stage.
🪞 Near-duplicate & similarity detection
Re-uploaded banned images, plagiarized text, duplicate support articles. Not top-k but a threshold query: "anything within distance ε?" Embed, search, compare distance against a calibrated cutoff.
Why a vector DB: robust to crops, rewording and re-encodes that defeat exact hashing; one index serves both dedupe and search.

When a vector DB is the WRONG answer

exact / compliance search"Every document containing clause 7.3" — approximate retrieval may miss some, by design. Auditors don't accept 97% recall. Use BM25 / an inverted index.
tiny corpus (≤ ~100k)Brute force is a matrix multiply: exact, zero infra, no recall question. Adding an ANN index here is résumé-driven architecture.
source of truthA vector index is a derived, rebuildable structure — not where orders or balances live. Keep the system of record in a transactional store; treat embeddings like a search index.
filter does all the workWHERE user_id = ? ORDER BY created_at needs a B-tree. If the structured predicate already narrows to a handful of rows, similarity adds nothing — scan and sort.

4 · Distributed setup — what "running it for real" means

Replication is the boring half: replicas serve reads, queries are stateless, scale-out for QPS is standard. Sharding is where vector search is strange. In a key-value store you hash the key and prune to one shard. Here the "key" is meaning, and you can't hash meaning into ranges — vectors are sharded essentially at random, so every query fans out to every shard, each runs ANN locally, and a coordinator merges the per-shard top-k into a global top-k. Scatter-gather, always. That has two consequences worth saying in an interview: query cost grows with shard count (no pruning), and p99 = the slowest shard — one cold or overloaded node drags every query, the classic tail-latency amplification.

Writes look like an LSM tree wearing an ANN costume. Fresh vectors land in a small growing segment that is brute-force-searched (exact, fine because it's small); background jobs seal it into an immutable indexed segment; queries search all segments and merge. Deletes are tombstones: the graph or cell entry stays, results get filtered at query time, and the space (and graph quality) is reclaimed only on merge/rebuild. IVF has an extra ageing problem — centroids are trained once, so as your data distribution drifts, cells get lopsided and recall quietly sags until you retrain. Index builds are real maintenance events: plan them like you plan compactions.

The embedding-version gotcha. Vectors are only comparable if the same model version produced them. Upgrade your embedding model and every stored vector must be re-embedded and re-indexed — distances between model-v1 and model-v2 vectors are meaningless, and nothing errors: quality just silently craters. The production pattern is dual-index + backfill + cutover, and it costs one full pass over the corpus (often GPU-billed). Mention this before the interviewer does.

🔍 Deep dive · HNSW — a skip list over meaning click to expand

Hierarchical Navigable Small World (Malkov & Yashunin) is the index most engines default to. Picture a skip list: the bottom layer contains everything, each layer above keeps an exponentially thinner sample. HNSW does that with a graph per layer: every vector is a node connected to ~M near neighbors; sparse top layers make continent-sized hops, the dense bottom layer makes street-level steps. A query starts at the top entry point, greedily walks to whichever neighbor is closest to the query, drops a layer when it can't improve, and at layer 0 keeps a candidate list of the best efSearch nodes seen. Logarithmic-ish hops instead of a linear scan.

The three knobs to name: M (links per node — more links, better recall, more memory), efConstruction (how hard the build searches for good links — build time vs graph quality), and efSearch (query-time candidate list — the live recall↔latency dial; raise it and recall climbs as p99 grows, which is exactly the trade you demoed in the simulator with nprobe). Because inserts are just "search for neighbors, then link", HNSW absorbs inserts well — no retraining step, unlike IVF's centroids.

The costs are memory and deletes. Everything — full-precision vectors plus graph links — wants to be in RAM; links add roughly d×4 + M×8 bytes per vector in hnswlib-style layouts, and the rule of thumb from practice is that an HNSW deployment runs 2–5× more memory than lean quantized/IVF alternatives. Deletes are worse than expensive — they're structural: removing nodes punches holes in the paths other queries walk through. Engines tombstone-and-skip, then repair or rebuild in the background; after mass deletions, expect a recall cliff until that happens.

🔍 Deep dive · IVF, quantization & searching from disk click to expand

IVF (inverted file) is the simulator made real: run k-means over (a sample of) your vectors to get lists centroids; each vector is filed under its nearest centroid. A query ranks centroids, probes the nprobe nearest cells, and brute-forces only those. Cheap to build, cheap in RAM, embarrassingly parallel — and its failure mode is geometric: a true neighbor sitting just across a cell boundary in an unprobed cell is simply never seen. That's the red ring in the widget, and it's why probes=1 defaults are recall traps. IVF also assumes centroids still describe your data: after heavy drift or bulk inserts, cells go lopsided and recall sags until you retrain.

Quantization attacks the RAM bill. Scalar quantization stores int8 instead of float32 — 4× smaller, tiny recall cost. Product quantization (PQ) goes further: chop the vector into subvectors, replace each with a 1-byte codebook id — commonly 8–32× compression. Distances get computed against the compressed codes (fast, approximate), and good engines re-rank the top candidates with the original vectors (kept on disk) to claw quality back. IVF+PQ is the classic FAISS recipe for "a billion vectors on a handful of boxes".

When even that doesn't fit, go to SSD. DiskANN-style graph indexes (Microsoft's Vamana lineage) keep a compressed sketch of every vector in RAM for navigation and fetch full vectors from NVMe with a few random reads per query — latency in the milliseconds-to-low-tens range, RAM an order of magnitude smaller. The mental ladder for capacity: float32 in RAM → int8 → PQ → disk-based ANN → shard, each step trading a little recall or latency for a lot of money.

🔍 Deep dive · Filtered search — the tenant trap click to expand

Real queries are never pure similarity: it's "nearest docs that this user may read", "similar products in stock, under $50". The naive plan — run ANN for top-k, then apply the filter — is the post-filter trap: if your tenant owns 1% of the corpus, the global top-10 neighbors are ~99% other tenants' documents, and after filtering you return one result, or zero. Not slower — wrong. And over-fetching (top-1000, then filter) just moves the cliff while multiplying cost.

The strategies, by filter selectivity: (1) Pre-filter — when the predicate matches few rows (one user's 2,000 docs), fetch those ids and brute-force them exactly; no index needed. (2) In-traversal filtering — the engine evaluates the predicate while walking the graph, only admitting matching nodes into the candidate list, walking through non-matching ones so the graph stays connected. Qdrant's filterable HNSW, Weaviate, Pinecone's metadata filtering and pgvector's iterative index scans are all versions of this. (3) Partition outright — if tenants never share data, give each a namespace/collection: hard isolation, no filter cost, no cross-tenant leakage risk at the index layer.

The senior phrasing: "filter selectivity picks the strategy — highly selective filters pre-filter and brute-force; broad filters ride along the traversal; disjoint tenants get separate namespaces." And treat ACL filtering as a correctness feature, not a performance one — a post-filter design here is a security review finding, not a perf bug.

🔍 Deep dive · Hybrid retrieval & reranking — the quality stack click to expand

Dense vectors and keyword search fail in opposite directions. Embeddings nail paraphrase ("can't log in" ≈ "password reset") but smear exact tokens — error codes, SKUs, proper names, NullPointerException — into fuzzy neighborhoods. BM25 nails exact tokens and rare terms but is blind to synonyms. So production search runs both and fuses the ranked lists, most simply with Reciprocal Rank Fusion: score(doc) = Σ 1/(60 + ranklist). Rank-based, no score normalization across incomparable scales, one line of code — and commonly reported to lift recall +15–30% over either retriever alone.

Then spend your quality budget where it's cheapest: rerank the top-N with a cross-encoder. Your retriever is a bi-encoder — query and doc embedded separately, so similarity is one dot product and the corpus is indexable. A cross-encoder reads query and document together through one model and scores the pair — far more accurate, but O(model-inference) per pair, so it can never scan a corpus. The stack: ANN + BM25 pull ~100–200 candidates fast; the cross-encoder reorders them; top 5–10 go to the user or the LLM. Adds tens-to-low-hundreds of ms — usually the single highest-ROI quality lever in a RAG pipeline.

Interview framing: "retrieval is recall-oriented and cheap; reranking is precision-oriented and expensive; hybrid + RRF + cross-encoder on the top-100 is the default quality stack." That one sentence upgrades a "we'll use a vector DB" answer into a search-systems answer.

⚖️ At scale — numbers, limits & failure modes

Figures to quote in capacity talk. Do the RAM math out loud — it's the part interviewers check.

RAM ≈ n × d × 4 Bfloat32 raw: 1M × 768 ≈ 3 GB · 1M × 1536 ≈ 6 GB · 10M × 1536 ≈ ~60 GB · 100M × 1536 ≈ ~0.6 TB — before index overhead. This one line decides pgvector vs quantize vs shard.
HNSW overheadGraph links ≈ M×8 bytes/vector extra (~130 B at M=16) — small next to a 6 KB vector; the vectors themselves are the bill. Everything wants to be RAM-resident; budget 2–5× vs quantized/IVF setups.
quantizationfp16/halfvec 2× · int8 scalar ~4× · PQ 8–32× — each with a recall haircut; re-rank top candidates with full vectors to recover most of it.
recall SLOOperate at 95–99% recall@k. Measure against brute-force ground truth on a sampled query set — recall is invisible unless you build the harness.
latencyIn-RAM HNSW at millions of vectors: ~ms-scale per query. Brute force 10M × 1536: hundreds of ms–seconds per core. Disk-based ANN: low tens of ms. (Orders of magnitude, not benchmarks.)
index buildsHNSW at 100M scale: order of hours, CPU-parallel; IVF trains centroids on a sample (minutes) then assigns fast. Rebuilds are scheduled maintenance — plan them like compactions.
throughputOrder of 10²–10³ QPS per node at high recall (d- and recall-dependent); scale reads with replicas, capacity with shards — remembering every query hits every shard.
Failure modes that only appear at scale: recall cliff after mass deletes (tombstoned graph nodes degrade paths until repair/rebuild); stale IVF centroids (distribution drift → lopsided cells → recall sags with no errors anywhere); post-filter zero-results (selective tenant + global ANN = correctness bug); scatter-gather tail (p99 = slowest shard; one hot shard taxes every query); hot namespace (one big tenant dominates a node while others idle); embedding version skew (mixing model-v1 and v2 vectors — no errors, garbage similarity).

8 · In the interview

How it comes up: "design semantic search / RAG for enterprise docs / a recommendation feed" — or you introduce it yourself the moment similarity appears. The follow-ups are predictable: how does the index actually work? (HNSW layers, greedy descent), how much RAM? (do n×d×4 out loud), what about per-tenant access control? (filter during traversal, or namespaces), how do you know it's working? (recall@k vs ground truth as an SLO), why is keyword search still in your diagram? (hybrid + rerank).

The choice script — a ladder, not a brand. Under ~100k vectors: brute force, exact, no infra. To ~1M-ish: pgvector — one datastore, transactions, joins, filters, no new ops surface. Beyond that, or when you need filtered HNSW, quantization and shard/replica management as first-class features: a dedicated engine (Qdrant, Milvus, Weaviate — or managed Pinecone). Already running Elasticsearch/OpenSearch? Its kNN support is often "good enough without a new database". Naming the rungs and the cutover triggers is the senior signal; naming only a vendor is the junior one.

9 · Check yourself

Why can't a B-tree or an inverted index answer "the 10 most similar documents"?

A B-tree serves queries along one sort order (equality/range on a key); in ~1000-d space no single ordering keeps semantic neighbors adjacent, so "nearest" can't be a range scan. An inverted index maps exact tokens → documents; similarity of meaning isn't token overlap. kNN in high dimensions needs its own index family (graph- or partition-based ANN), and low-dimensional spatial trees (KD/R-trees) degrade toward full scans as dimensionality grows.

The interviewer says: "just run the vector search, then filter results by tenant". What breaks?

If the tenant owns a small slice of the corpus, the global top-k is dominated by other tenants' vectors; after filtering you return few or zero results even though good on-tenant matches exist. Over-fetching only moves the cliff. Correct designs: apply the predicate during traversal (filtered HNSW / iterative scans), pre-filter + exact scan when the tenant's set is small, or give disjoint tenants their own namespaces. For ACLs this is a correctness/security issue, not a perf tweak.

10M documents, 1536-dim float32 embeddings. What's the RAM bill, and what are your options?

10M × 1536 × 4 B ≈ 61 GB raw, before HNSW graph links and working memory — so a single fat node is possible but uncomfortable. Options: fp16 (~30 GB), int8 (~15 GB, minor recall cost), PQ (8–32×, bigger haircut, re-rank with originals), disk-based ANN (RAM holds compressed sketches), or shard — accepting scatter-gather on every query. Saying the multiplication out loud is the point.

HNSW vs IVF — pick one for a corpus with constant inserts and justify it.

HNSW: inserts are incremental (find neighbors, link in) with no retraining, recall/latency is strong, and efSearch gives a live recall dial. Cost: 2–5× memory and painful deletes. IVF would need periodic centroid retraining as the distribution drifts — its cells go stale and recall sags silently. IVF(+PQ) wins instead when memory or build cost dominates and the corpus is relatively static.

Recall@10 quietly dropped from 97% to ~85% over a quarter. Name three plausible causes.

(1) Mass deletes/churn punched tombstone holes in the HNSW graph — needs repair/rebuild. (2) IVF centroid staleness: the data distribution drifted (new product line, new language) so cells no longer match the data — retrain. (3) Embedding version skew: some vectors written by a newer model version — cross-version distances are meaningless. Honorable mention: someone "optimized latency" by lowering efSearch/probes. You only catch any of these if recall is measured continuously against a ground-truth sample.
