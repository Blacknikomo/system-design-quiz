🟥 Redis — replication & reads

In-memory key-value store. Watch a write hit the primary, replicate asynchronously to replicas, and see why a replica read can be stale — and how failover can drop acknowledged writes.
single-threaded async replication consistent, not durable noeviction default

Redis runs commands on a single thread, so each command is atomic. The primary takes writes and acknowledges the client before replicas have the data (async replication). Replicas serve reads to scale read traffic — but they may lag.

The gotcha: because the primary acks before replicating, a failover can promote a replica that never saw the last acked writes → those writes vanish. That's why single-node/replicated Redis is not a system of record.

Run a command

Run SET to write a key. It goes to the primary, which acks immediately, then streams to the replicas. Try pausing replication, writing, then failing over.

Command log

What you're seeing

  1. Client sends SET → goes to the primary only.
  2. Primary applies it and acks the client right away.
  3. Primary asynchronously ships the write to each replica (purple).
  4. GET from a replica reflects only what it has applied — possibly stale or missing.
  5. On failover, the most up-to-date replica is promoted; writes it never received are lost.

📖 Reference — infrastructure, sharding, commands & recipes

Infrastructure & deployment

  • Single-threaded event loop — one core runs commands; each command is atomic, but a heavy command (KEYS, big SORT) blocks the box. Run multiple instances to use more cores.
  • Persistence: RDB snapshots (fork + copy-on-write, point-in-time) and/or AOF append-only log (appendfsync everysec ≈ ≤1s loss; always = durable but slow). Enable both for cache+durability.
  • HA (single shard): Sentinel monitors master + replicas, does auto-failover, and tells clients the new master. Replication is async → a promoted replica may lose the last acked writes.
  • Managed: ElastiCache (cache) · MemoryDB (multi-AZ, durable, strongly-consistent primary) when Redis must be a source of truth.
  • Memory: set maxmemory + an eviction policy; default is noeviction (writes error when full) — set allkeys-lru/allkeys-lfu for a pure cache.

Sharding & scaling

  • Redis Cluster: 16384 hash slots, slot = CRC16(key) mod 16384; slots split across masters, each with replicas. (NOT consistent hashing.)
  • Clients cache the slot→node map; wrong node replies MOVED (permanent) or ASK (during slot migration).
  • Hash tags: only the part in braces is hashed — {user:123}:posts & {user:123}:likes share a slot, so multi-key ops / MULTI work (cluster multi-key ops must stay in one slot).
  • Scale reads: add replicas (READONLY on the connection). Scale writes/space: add masters and reshard (move slots).
  • Hot key: client-side cache, key copies (item:123:1..N → different slots), replica reads (read-hot only — a write-hot key still hits one master).
  • Sharded Pub/Sub since Redis 7 (SPUBLISH/SSUBSCRIBE) routes a channel to its slot's shard.

Use-case recipes

🗃️ Cache-aside
# read
GET user:123                     # miss → load from DB
SET user:123 "{...}" EX 300      # cache with 5-min TTL
# on write → invalidate
DEL user:123
Default pattern. For a stampede on a hot key, let one request rebuild under a short SET lock NX while others wait, add TTL jitter, or refresh-ahead.
🔒 Distributed lock
# acquire (token = random uuid, 30s safety TTL)
SET lock:order:42 <token> NX EX 30
# release — check-token-then-DEL atomically (Lua)
EVAL "if redis.call('GET',KEYS[1])==ARGV[1]
        then return redis.call('DEL',KEYS[1]) else return 0 end" 1 lock:order:42 <token>
A blind DEL could delete someone else's re-acquired lock. Treat it as an efficiency lock; for correctness use fencing tokens or a consensus store (ZooKeeper/etcd), or a DB row lock.
🚦 Rate limiter
# fixed window — atomic INCR+EXPIRE via Lua
EVAL "local n=redis.call('INCR',KEYS[1])
      if n==1 then redis.call('EXPIRE',KEYS[1],ARGV[1]) end
      return n" 1 rl:user:7 60          # reject if n > limit
# sliding window log: ZADD now; ZREMRANGEBYSCORE old; ZCARD
# token bucket: HSET tokens/last_refill; Lua refill+take
Set EXPIRE only when INCR returns 1 (else steady traffic never resets). Run it as one Lua script so a crash can't leave a counter without a TTL. Shard Redis by client key for >100k ops/s.
🏆 Leaderboard (sorted set)
ZADD board 4200 "alice"            # add/update score
ZINCRBY board 50 "alice"           # bump score
ZREVRANGE board 0 9 WITHSCORES     # top 10
ZREVRANK board "alice"             # 0-based rank
ZRANGEBYSCORE board 1000 2000      # range by score
O(log n) rank/range. This is exactly how Redis ZSETs (skip list + hash) power game/score leaderboards.
📣 Pub/Sub vs Streams
# Pub/Sub — fire-and-forget, at-most-once, no history
SUBSCRIBE chat:42        /        PUBLISH chat:42 "hi"
# Streams — persisted log, consumer groups, at-least-once
XADD events * type click adId 99
XREADGROUP GROUP g c1 COUNT 10 STREAMS events >
XACK events g <id>         # ack; XCLAIM re-assigns dead consumers' pending
Pub/Sub drops messages for offline subscribers. Streams persist + replay + track per-group offsets — but delivery is at-least-once, so make consumers idempotent.
