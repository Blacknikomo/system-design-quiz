🔍 Code Review — the reviewer's protocol

A repeatable multi-pass protocol for the Code Comprehension round: what to check, in what order, and how to say it. Pairs with the code_review_practice/ examples.
Code Comprehension 60 min JS / TS focus

In this round you're handed unfamiliar code (often with tests) and asked to review it as if a teammate sent you a PR. The signal is not "found N bugs" — it's how you read code: intent first, severity-ranked findings, questions instead of verdicts, and noticing design and contract issues, not just syntax.

In the interview: think aloud the whole time. Narrate which pass you're on ("first I want to understand what this is supposed to do, then I'll check whether it does") — the structure itself is a Staff-level signal.

⏱ Suggested 60-minute shape

~5 min
Pass 0 — Intent
Requirements, contract, tests-as-spec, clarifying questions.
~20 min
Pass 1 — Correctness
Does it do what it claims? Errors, boundaries, naming.
~20 min
Pass 2 — Quality
Performance, concurrency, design, tests, consistency.
~10 min
Wrap-up
Summarize by severity; state what you'd block on vs approve-with-nits.

Timebox loosely — if a P0 shows up in Pass 1, chase it. But always leave time to summarize: a review with no verdict is unfinished.

Pass 0Understand before you judge

Never comment on line 1 before you know what the code is for. Cheapest sources of intent, in order:

ReadWhat it tells you
Function / class / file namesThe claimed purpose. Mismatch between name and behavior is itself a finding.
Docstrings & type signaturesThe contract: inputs, outputs, errors. Note what's unspecified (null? empty? negative?).
TestsThe executable spec. What cases exist tells you the author's mental model; what's missing tells you where bugs hide.
Callers / usage (if shown)Real expectations — e.g. is the return value checked? Are errors caught?

Then ask the author (interviewer) contract questions before asserting bugs:

Ask
"What should happen for an empty input — return 0, or is that a caller error?" · "Is this called concurrently?" · "How large can this collection get in production?" · "Who consumes this error — a user or another service?"
Gotcha: skipping Pass 0 is the #1 way to fail this round — you'll "fix" intentional behavior and miss the real spec violations. Understanding the requirements is part of the review.

Pass 1Correctness sweep — does it do what it claims?

CheckWhat to look for
Correctness vs intentWalk the happy path line-by-line against the contract from Pass 0. Trace one concrete input through by hand. Watch for name/behavior mismatches (getUser that mutates), inverted conditions, wrong operator (< vs <=, && vs ||).
Boundary conditionsEmpty / single-element / max-size collections; 0, -1, negative numbers; first and last iteration of every loop (off-by-one); integer overflow & float precision (0.1+0.2); very long strings; unicode; null/undefined/NaN at every entry point.
Error handlingAre errors swallowed (catch {}), logged-and-ignored, or propagated? Does a failure leave state half-mutated (no rollback / no atomicity)? Are error messages actionable and free of sensitive data? Is the error type part of the contract or a stringly-typed surprise?
Input validation & trustWhere does the data come from? User input needs validation at the boundary: injection into queries/HTML, path traversal, prototype pollution (JS), unbounded sizes. Internal calls can trust more — say which boundary this is.
Resource handlingFiles / connections / timers / listeners closed on all paths, including the error path (finally). Unbounded growth: caches without eviction, arrays that only ever push, recursion without a base-case guard.
Naming & readabilityDo names tell the truth? Booleans read as predicates (isEmpty, not flag)? Magic numbers named? Dead code, misleading comments, functions doing more than the name promises.
Technique: for any loop or index arithmetic, run the smallest interesting example (n=0, n=1, n=2) in your head out loud. Most planted bugs in this round are boundary bugs and are found exactly this way.

Pass 2Quality sweep — is it good code?

CheckWhat to look for
PerformanceAccidental O(n²): includes/indexOf inside a loop, repeated string concat, nested scans that a Set/Map would fix. Work done per-call that could be hoisted or cached. N+1 queries / IO in a loop. But: ask about real data size first — don't optimize a 10-element list.
Better approachIs there a simpler or standard solution — a library call, a well-known data structure, removing a whole layer? Frame it as a question of trade-offs, not a rewrite demand; the author may have context you don't.
Concurrency / asyncJS-specific: un-awaited promises (floating rejections), await inside forEach, check-then-act races on shared state across an await gap, missing idempotency for retried operations. Multithreaded langs: shared mutable state, lock ordering, atomicity of compound ops.
Class / API designSingle responsibility; leaking internals (returning a mutable private array); constructor doing IO; inheritance where composition fits; parameters that always travel together (introduce a type). Is the public surface minimal? Would this API be hard to change later — is it backward-compatible to evolve?
Unit testsReview tests as code too. Do they cover the boundaries from Pass 1, the error paths, not just the happy path? Do they assert behavior or mirror the implementation (change-detector tests)? Any test interdependence, hidden shared state, time/randomness flakiness? What test would you add?
Comments & docsComments should say why, not narrate what. Stale comments that contradict code are P1 — worse than none. Public API without docs on error behavior and edge-case semantics.
ConsistencyDoes the change follow the patterns of the surrounding codebase (error style, naming scheme, layering)? A locally-fine pattern that fights the codebase's conventions raises everyone's maintenance cost. Style-guide nits (Google TS style) are P2 — mention, don't dwell.
ObservabilityWould you be able to debug this in prod at 3am? Key decisions logged with context, failures distinguishable, metrics for the interesting counters. No noisy log-per-iteration, no secrets in logs.

🎯 Severity — rank everything you say

Interviewers listen for whether you weight findings. A review that leads with a semicolon nit while a data-loss bug sits on line 12 fails on judgment, not knowledge.

LevelMeaningExamples
P0 — blockerWrong result, crash, data loss, security hole. Must fix before merge.Off-by-one drops last element; swallowed error corrupts state; SQL injection; race on shared counter.
P1 — importantWorks today but will hurt: perf cliff, missing error path, fragile design, missing key test.O(n²) on a growing dataset; no timeout on network call; API leaks mutable internals; stale comment.
P2 — nitStyle, naming, polish. Batch them, prefix with "nit:", never block on them.Naming convention, magic number, comment phrasing, import order.
Gotcha: severity depends on context you may not have — that's why you asked the scale/concurrency questions in Pass 0. "This is O(n²) — is n bounded here?" turns a guess into a calibrated finding.

💬 How to say it — questions over verdicts

Google's own review guidance: be kind, explain why, comment on the code not the author, and distinguish "must fix" from "consider". In the interview this doubles as a collaboration signal.

Weak
"This is wrong." · "Why would you do it this way?" · "You should use a Map."
Strong
"I think this drops the last element when the array length is even — walking through n=2: … Am I reading that right?" · "This lookup is O(n) inside the loop; if n can reach ~10⁵, a Set would take it to O(n) total. How large does this get?" · "nit: data2 → maybe sortedUsers, so the invariant is visible at the call site."

Pattern: observation → evidence (concrete trace) → severity → question or suggestion. Praise good parts too — noticing a well-chosen structure is also reading comprehension.

🧪 Reviewing the tests — a checklist of missing cases click to expand

"What test is missing?" is a guaranteed question. Generate candidates mechanically from the contract:

AxisCases to propose
Sizeempty, one element, two, many, at-the-limit, over-the-limit
Values0, negative, min/max, duplicates, already-sorted / reverse-sorted, unicode & empty string
Nullnessnull, undefined, missing property, NaN
Errorseach failure path actually asserted (thrown type/message), not just "doesn't crash"
Statecall twice (idempotency), interleave operations, state after a failed operation
Asyncconcurrent calls, rejection path, timeout, ordering guarantees

Also flag test smells: asserting internals instead of behavior, shared mutable fixtures, sleeps instead of awaits, one giant test instead of one-behavior-per-test.

🧾 Wrap-up script

Close the review the way a senior reviewer closes a PR:

  1. "Summary: the code intends to X. I found one P0 — …, two P1s — …, and a few nits I'd batch."
  2. "I'd block on the P0; the P1s I'd want fixed or ticketed with the author's input; nits are take-or-leave."
  3. "Missing tests I'd ask for: … (from the axes above)."
  4. "Things I liked: …" — one genuine positive, it's a real signal, not politeness.
  5. Open questions you'd take to the author rather than assume.

✅ Check yourself

Why read the tests before the implementation?
Tests are the executable spec: they reveal the author's intended contract and mental model faster than the code does, and the gaps in them predict where bugs live. Reviewing code against inferred intent (Pass 0) is what separates "found a typo" from "found a spec violation".
The code has an O(n²) loop. Is that a P0, P1, or P2?
Depends on n — which is why it's a question first: "how large does this collection get?" Bounded small n → P2/not-a-finding; user-growable or hot-path n → P1; if it can hang the service at production scale → effectively P0. Calibrating severity with a context question is the Staff-level move.
Name three places JS async code goes wrong that a sync-minded reviewer misses.
(1) await inside forEach — promises fire-and-forget; (2) floating promises with no .catch — silent unhandled rejections; (3) check-then-act across an await: state read before the await may be stale after it, a single-threaded race.
What belongs in the wrap-up besides the list of bugs?
A severity-ranked summary, an explicit merge verdict (block vs approve-with-comments), the missing test cases you'd require, at least one genuine positive, and the questions you'd take to the author instead of assuming answers.
The function is named getActiveUsers but also deactivates expired ones. Severity, and how do you phrase it?
At least P1 — a name/behavior mismatch is a contract bug: callers will invoke it "read-only" and mutate state by surprise. Phrase as observation + question: "The name suggests a pure read, but line 14 mutates status — is the side effect intended? If so I'd split it or rename to make the write visible."
