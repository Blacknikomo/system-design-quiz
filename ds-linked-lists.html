🔗 Linked Lists — nodes & pointers

Nodes scattered in memory, chained by next pointers. Insert/delete at a known node is O(1); finding it is O(n). Toggle doubly-linked, reverse the list, and run Floyd's cycle detection.
pointer-chained O(1) splice at a known node O(n) search no random access

Each node holds a value and a pointer to the next node (a doubly-linked node also keeps a prev pointer). Nodes live anywhere in memory — there's no contiguity, so you cannot jump to index i; you must walk from the head. The payoff: inserting or deleting at a node you already hold is O(1) pointer rewiring, no shifting.

The gotcha: O(1) insert/delete only holds when you already have a pointer to the node (or its predecessor). If you must first find the node, that's an O(n) search and the operation is O(n) overall. And despite the better Big-O on splices, arrays usually win in practice: pointer-chasing scatters cache misses, while contiguous arrays stream through cache. Reach for a linked list when you need stable O(1) splicing with held node references (LRU cache, intrusive lists), not for general iteration.
Floyd follow-up: to find the cycle's entry node — after the fast/slow pointers meet, reset one pointer to head and advance both one step at a time — they meet at the cycle entry.

What you're seeing

  1. Insert head: new node's next = old head, head = new — O(1).
  2. Insert tail: with a tail pointer it's O(1); otherwise you walk to the end (O(n)).
  3. Delete: find the node (O(n) search), then rewire its neighbours — the splice itself is O(1).
  4. Search / Reverse: touch every node once — O(n).
  5. Floyd: slow moves 1, fast moves 2; if they ever land on the same node there's a cycle — O(n) time, O(1) space.
