🧭 Graph Databases — when JOINs become hops

Index-free adjacency, Cypher, supernodes, why graphs shard badly — and why Facebook famously does not run its social graph on one. ☕ ~15 min read
index-free adjacency property graph Cypher / GQL single-leader writes supernodes

A graph database stores relationships as first-class data: every node physically points at its neighbors, so "friend of a friend of a friend" is three pointer-chases — not three JOINs that each get slower as the table grows. That one storage decision buys spectacular deep-traversal speed, and it costs you horizontal write scaling, because a web of pointers is exactly the thing that refuses to be sharded.

In the interview: graph DBs come up in fraud detection, recommendations, and permission systems. The interviewer expects you to know when NOT to use one (fixed 1–2 hops = a JOIN problem), to name the honest weakness (writes don't scale out; sharding a graph is a min-cut problem), and — for senior candidates — to know that Facebook's TAO and Google's Zanzibar serve graph-shaped data without a graph database.

1 · The mental model: hyperlinks for your data

Think about the difference between a paper library and the web. To follow a citation in a book, you walk back to the central catalog, look up the referenced title, find the shelf number, and walk there — and the bigger the library, the longer the lookup. To follow a hyperlink, you just click: the link itself stores where it goes, and it doesn't matter whether the web has a million pages or a trillion. Relational databases are the library catalog. Graph databases are the hyperlink.

The data model is the property graph: nodes carry labels and properties ((:User {id: 42, name: 'Ava'})), and relationships are typed, directed, and carry properties of their own (-[:FOLLOWS {since: 2021}]->). The crucial word is stored. In a relational schema, "Ava follows Ben" is a row in a join table, and the connection is reconstructed at query time by matching values through an index. In a graph database the connection was materialized at write time as a physical reference — the relationship is a record that both nodes can reach by direct offset.

That property has a name worth saying out loud: index-free adjacency. Once you're standing on a node, enumerating its neighbors is pointer arithmetic — roughly O(1) per relationship, regardless of how many nodes exist in total. Compare the relational version of one hop: an index seek costing ~O(log N) into a table whose N is the whole edge set. One hop, fine. But "friends up to 3 hops out" becomes a recursive self-JOIN whose work multiplies by the frontier at every level and pays the log-of-table-size tax on every row it touches. The graph engine's cost is proportional to the data you actually traverse; the relational cost carries a factor of the data you own.

Now the honest half, because interviewers probe for it. For a fixed, shallow query — "get this user's orders", "who follows Ava" — a JOIN with a proper index is excellent, cached, boring, and comes with the world's best tooling. A useful rule: most graph problems at ≤2 hops are join problems. Graph databases start earning their keep when depth is variable or ≥3, when the query is a pattern (cycles, diamonds) rather than a lookup, or when "how deep" is itself the question (shortest path). And one myth to preempt: index-free adjacency doesn't mean index-free database — you still use an ordinary index once, to find the starting node. It's every hop after that which skips the index.

The one-sentence test. If you can write the query with a known, small number of JOINs, use your relational database. If the query's natural language contains "…of a…" repeated an unknown number of times — friends of friends of friends, parts of parts, owner of the company that owns the company — that's a traversal, and traversals are what graph databases are for.

2 · Syntax & core API — Cypher (and its rivals)

Cypher is Neo4j's declarative language, and its party trick is that queries look like the whiteboard drawing: nodes are (), relationships are arrows. You state the pattern; the planner picks the traversal order.

(a:User {id: 42})Node pattern: variable a, label User, property filter. This is the part that uses a regular index — to find your entry point.
-[:FOLLOWS]->Typed, directed relationship. Reverse it (<-) or drop the arrow entirely to traverse both directions.
-[:FOLLOWS*1..3]->Variable-length path — the thing SQL can't say in one breath. "One to three FOLLOWS hops." Unbounded * is legal and dangerous (see deep dive 4).
MATCH … WHERE … RETURNThe read pipeline. DISTINCT and LIMIT matter more than in SQL — path patterns multiply rows fast.
MERGEMatch-or-create for a whole pattern — the graph UPSERT. The idiom for "ensure this device node exists, then link it".
shortestPath(…)Built-in bidirectional shortest-path search between two bound nodes; allShortestPaths for ties.
CREATE INDEX / CONSTRAINTIndexes exist — for node lookups by property, not for hops. A unique constraint on User.id is table stakes.
PROFILE / EXPLAINShows the plan and db hits per operator — the graph world's "rows examined". Your supernode problems show up here first.
// "people Ava reaches in ≤3 hops but doesn't already follow" — the widget below runs this
MATCH (a:User {id: 42})-[:FOLLOWS*1..3]->(b:User)
WHERE NOT (a)-[:FOLLOWS]->(b) AND b <> a
RETURN DISTINCT b.name LIMIT 20

// fraud: ensure the shared device exists and link this signup to it (MERGE = match-or-create)
MERGE (d:Device {fp: 'fp_9f3a'})
MERGE (u:User {id: 42})-[:USED]->(d)

// link analysis: how are these two accounts connected, within 6 hops?
MATCH p = shortestPath( (a:Account {id: 1})-[:TXN*..6]-(b:Account {id: 99}) )
RETURN length(p), [n IN nodes(p) | n.id]

The rest of the language landscape in three lines. Gremlin (Apache TinkerPop) is the imperative cousin — you script the walk itself: g.V().has('id',42).repeat(out('FOLLOWS')).emit().times(3).dedup().limit(20); powerful, composable, harder to read in a review. SPARQL queries RDF triple stores (subject–predicate–object; the semantic-web/knowledge-graph lineage) — you'll meet it in Neptune and enterprise ontologies. And in 2024 the ISO published GQL — the first new ISO database language since SQL — essentially standardizing Cypher-style property-graph patterns, so the vendor dialects are finally converging.

-- the same "≤3 hops" in SQL, for contrast: a recursive CTE — one more self-JOIN per level
WITH RECURSIVE reach(id, depth) AS (
  SELECT friend_id, 1 FROM follows WHERE user_id = 42
  UNION
  SELECT f.friend_id, r.depth + 1
  FROM reach r JOIN follows f ON f.user_id = r.id   -- index seek per row, log(table) each
  WHERE r.depth < 3
)
SELECT DISTINCT id FROM reach;

3 · Use cases — when the graph is the answer

🕵️ Fraud rings — pattern matching in real time
Fraudsters reuse infrastructure: the same device, drop address, or card shows up behind "unrelated" accounts. Model users, devices, addresses, cards as nodes; sharing as relationships. A new signup that closes a cycle — two accounts, one device, one payout address — is a ring, and the check is a bounded pattern match at insert time, in milliseconds. In SQL this is a pile of self-joins that grows with every new attribute type; in a graph it's one query shaped like the crime.
Why a graph: the signal is the topology (cycles, shared hubs), depth varies, and it must run synchronously on the write path.
🤝 Recommendations — collaborative traversal
"Users who bought X also bought…" is a 2–3 hop walk: from the product, to its buyers, to what else they bought, weighted by how often each item recurs. Depth 2 works in SQL; the graph version stays fast when you enrich it — through brands, categories, or social edges — because each enrichment is another hop, not another JOIN in an already-groaning query.
Why a graph: the query grows by composition (add hop types), and latency stays tied to the local neighborhood you touch.
🧠 Knowledge graphs — entities and their web
Companies, people, papers, drugs, ownership chains: heterogeneous entities with many relationship types, queried by shape ("the owner of the company that owns this subsidiary"). This is also the natural backing store for GraphRAG-style retrieval, where an LLM's context is assembled by walking typed relations rather than by similarity search alone.
Why a graph: schema flexibility (new edge types are cheap) plus variable-depth "chain of ownership" queries.
🔐 Authorization / ReBAC — "can U see D?" as reachability
Relationship-based access control: user → member-of → group → editor-of → folder → contains → doc. A permission check is "does a path exist from U to D through allowed edge types?" — graph reachability with a depth bound. This is exactly the model Google's Zanzibar formalized as relation tuples (see deep dive 3 for why Google then didn't use a graph database for it).
Why a graph: policies nest (groups in groups), so depth is unknown at design time.

When a graph DB is the WRONG answer

fixed ≤2 hops"User and their orders", "post and its comments" — JOINs with indexes are perfect, and you keep SQL's ecosystem, optimizer maturity, and your team's fluency.
whole-graph analyticsPageRank, community detection, connected components touch every node repeatedly — that's a batch scan workload, not pointer-chasing. Run it offline in Spark/Pregel-style engines and cache the result.
simple many-to-manyA tags table is not a "graph use case". If you never traverse through the relationship to a third entity, a join table is simpler and faster.
write-heavy firehoseEvent ingestion at 10⁵+ writes/s: the single-leader write path becomes the wall (see §4). Land events in Kafka/OLTP first; project a graph view if you need one.

4 · Distributed setup — one strong box, many readers

Neo4j — the reference implementation of this category — is fully ACID: real transactions, WAL, constraint enforcement. Clustering (the causal cluster) is a set of core/primary servers running Raft plus any number of read replicas. For a given database there is one leader accepting writes; a commit replicates to a majority of primaries, and replicas apply the log asynchronously. Reads scale out nicely — clients pass a bookmark so a replica can wait until it's caught up to your own writes (that's the "causal" part). Writes, however, funnel through one machine: adding servers does not add write throughput. You scale writes vertically, or you partition by hand (Neo4j's Fabric federates multiple databases — which is honest manual sharding, and cross-database traversal gives up the magic).

Why not just shard automatically, like every other database on this site? Because a graph is the pathological case for sharding. Rows shard cleanly when they're independent; graph edges exist precisely to make data dependent. Systems that do distribute the graph — JanusGraph over Cassandra/HBase, Neptune, TigerGraph — store adjacency in a partitioned backend, and pay for it: any hop that crosses a partition is a network round trip, and deep traversals become chains of RPCs. The pointer-chase that cost ~100 ns in one process costs ~0.5–1 ms across machines — four orders of magnitude, per hop.

The gotcha to volunteer before they ask: "a graph database is the opposite of Cassandra: reads that ricochet unpredictably across the dataset, writes through a single leader. I'd size it as one beefy, RAM-rich box with read replicas — and if the requirements say horizontal write scaling on graph-shaped data, that's my cue to talk about TAO/Zanzibar-style designs instead of a bigger Neo4j."

5 · 🎛 Play with it — the hops-vs-JOINs race

A 24-person social graph with one ⭐ celebrity (Cee) connected to half of it. Pick a start node and a depth, then run the friends-of-friends query. The BFS frontier lights up hop by hop, and two cost meters race: the graph engine counts adjacency entries scanned (pointer chases); the relational engine counts index page-touches for the same query written as JOINs. The cost model, honestly: the graph meter counts adjacency entries scanned — genuinely what index-free adjacency does. The SQL meter models each hop as a nested-loop index join: every matched row costs one B-tree descent ≈ log₂(90 edge-rows) ≈ 6.5 page-touches. Real planners batch better (hash/merge joins), and 90 rows fit in one page — at this size SQL is effectively identical, which is the fair-play point. What the model does capture is the shape: the relational term multiplies by log(table size) and re-plans per level, while the graph term only ever sees local degree.

🔍 Deep dive · Inside Neo4j — fixed-size records and linked-list relationships click to expand

How do you make "follow a pointer" a database primitive? Neo4j's classic store answers with a trick borrowed from arrays: every record has a fixed size, so record #N lives at byte offset N × size — no index needed to find it, just multiplication. In the classic format a node record is ~15 bytes and a relationship record ~34 bytes (small enough that hundreds of millions of them sit comfortably in the OS page cache). A node record holds a pointer to its first relationship; each relationship record holds prev/next pointers for both of its endpoints — so all relationships of a node form a doubly-linked list threaded through the relationship store, and one relationship record participates in two lists at once.

Traversal, then, is literally: multiply, read, chase, repeat. Enumerating Ava's followers never consults a B-tree; it walks her chain. This is also why the numbers on this page keep saying "proportional to local degree" — walking a chain of 40 relationships costs 40 reads whether the database holds a thousand relationships or ten billion.

The fine print worth knowing: chains mean random access. When the hot part of the graph fits in RAM, hops are ~100 ns pointer-chases and traversal throughput on the order of millions of hops/s per machine is realistic. When it doesn't fit, each hop is a potential page fault — random 8 KB reads, no sequential prefetch to save you — and performance falls off a cliff rather than a slope. Graph databases are the most RAM-hungry systems per byte of useful data on this site; capacity planning is "keep the working set cached", not "keep the disks big enough".

🔍 Deep dive · Why graphs shard badly — min-cut, supernodes, and the network tax click to expand

Sharding a table is easy because rows don't reference each other; hash the key and go. Sharding a graph means drawing a boundary through a web of edges, and every edge the boundary crosses turns a ~100 ns pointer-chase into a ~0.5–1 ms network hop. So the goal is to cut as few edges as possible while keeping shards balanced — and balanced minimum-cut graph partitioning is NP-hard. Production systems use heuristics (METIS-style offline partitioning, hash-by-vertex and hope), and real social graphs are small-world networks: well-connected everywhere, with no clean seams to cut along.

Then there's the supernode. Degree in real graphs is power-law distributed: most users have 200 edges; a celebrity has 50 million. A supernode wrecks everything at once — it won't fit in one partition's neighborhood, every second traversal passes through it, its adjacency list is a multi-megabyte read, and in the widget above you watched one 12-edge "celebrity" dominate a 24-node graph. Mitigations are all partial: cap degrees in the model, split the node by relationship type or time bucket (FOLLOWS_2024_Q1), keep vertex-centric indexes (JanusGraph's term: an index within one node's edge list, so "Cee's followers since March" doesn't scan 50 M entries), or precompute around it offline.

Distributed graph engines — JanusGraph on Cassandra, Neptune, TigerGraph — are real and useful, but be precise about what they buy: capacity (graphs bigger than one machine) and availability, not fast deep traversal. A 5-hop query whose hops each cross the network is a 5-RPC chain with fan-out; the latency math rarely beats a single big-RAM box until the graph simply can't fit. This is the honest engineering reason the biggest players changed the question instead — next fold.

🔍 Deep dive · TAO & Zanzibar — how FAANG serves graphs without graph databases click to expand

The strongest senior-signal fact on this page: Facebook's social graph does not run on a graph database. TAO (USENIX ATC '13) is a write-through, graph-shaped cache API over sharded MySQL. The API is deliberately tiny — objects (nodes) and associations (typed edges), with point reads and time-ordered one-hop range queries like "latest 50 likes of this post". No Cypher, no pattern matching — TAO doesn't even offer a two-hop query; friends-of-friends is composed by application code fanning out parallel one-hop calls. In exchange it serves on the order of a billion reads per second (against millions of writes/s), eventually consistent, across regions. The insight: at that scale the workload isn't "traverse arbitrarily", it's "render a feed" — overwhelmingly hot, shallow, cacheable one-hop reads. So they built a system for the access pattern, not for graphs in general.

Google Zanzibar (ATC '19) tells the same story for permissions. Every ACL is a relation tuple — doc:readme#viewer@user:ava or "…@group:eng#member" — stored in Spanner, sharded like any other table. A permission check is graph reachability (can I get from the doc to the user through allowed relations?), answered by recursive tuple expansion with aggressive caching, plus a dedicated flattening index (Leopard) for deeply nested group-of-group membership. Order of 10 million checks/s with tail latencies in the low tens of ms — and consistency handled via zookies (opaque timestamps that prevent the "new enemy" problem: applying old ACLs to new content). Again: graph-shaped question, specialized sharded system, no graph database.

The lesson to say verbatim in an interview: graph access patterns survive extreme scale; general graph traversal engines don't. At planetary scale you constrain the query shape (1-hop reads, permission checks) and build on boring sharded storage. Graph databases win in the broad middle — datasets up to a few billion relationships, genuinely deep or pattern-shaped queries, and teams that want the query language to do the work.

🔍 Deep dive · Query traps — unbounded expansion and other self-inflicted wounds click to expand

The signature graph-DB outage is not a crash — it's one query. MATCH (a)-[*]->(b) with no depth bound asks for every path from a, and paths (unlike nodes) multiply combinatorially: a graph with average degree 40 has ~40³ three-hop paths and ~40⁶ six-hop paths from a single start. The engine dutifully materializes them until memory or your timeout gives out. House rules that prevent it: always bound variable-length patterns (*1..4), prefer shortestPath when you mean "are they connected", LIMIT aggressively, and run PROFILE looking at db-hits before anything ships. Neo4j lets you set transaction timeouts and memory limits per query — production clusters should.

Second trap: DISTINCT-late. A friends-of-friends pattern reaches the same person along many paths; if you aggregate at the end, you paid to walk every duplicate path. Filtering and deduplicating per level (or using node-visited semantics where the language allows) keeps the frontier a set — which is exactly what the widget above simulates, and why its numbers stay sane until the supernode shows up.

Third: hot-path writes to a supernode. Every "Cee gained a follower" write touches the same node's relationship chain; at celebrity fan-out rates that's a write hotspot on top of a read hazard. Feeds and counters belong in purpose-built systems (fan-out queues, counter stores); the graph keeps the relationship of record, not the firehose.

⚖️ At scale — numbers, limits & failure modes

Ballparks for capacity talk — quote orders of magnitude, not digits.

single instanceNeo4j comfortably holds billions of nodes/relationships on one RAM-rich machine; store-format limits stopped being the practical ceiling long ago — page-cache hit rate is. Plan RAM ≈ hot subgraph + indexes.
traversal speedIn-memory: on the order of millions of hops/s per machine; each hop ~100s of ns when cached. Falling out of RAM costs a random read per hop — orders of magnitude slower, and it arrives as a cliff, not a slope.
writesSingle Raft leader per database: cluster write throughput = one machine's — order of tens of thousands of small transactions/s on strong hardware. More core servers add durability and failover, not write capacity.
readsRead replicas scale horizontally; async replication with causal bookmarks ("read your own writes") — cheap to add, the one dimension that scales linearly.
distributed hop taxCross-partition hop (JanusGraph/Neptune-style) ≈ 0.5–1 ms vs ~100 ns in-process — roughly 10³–10⁴× per hop. Deep traversals over partitions are latency chains, budget accordingly.
supernode mathDegree is power-law: median user ~10²–10³ edges, celebrity ~10⁷–10⁸. One supernode in a frontier multiplies that hop's cost by its degree — same effect you saw in the widget, times a million.
Failure modes to name: unbounded pattern ([*] with no cap materializes exponential paths — memory/timeout blowup from one bad query); supernode on the hot path (reads scan its chain, writes contend on it); cache-miss cliff (working set outgrows RAM → every hop faults → p99 detonates); write ceiling (traffic growth meets the single leader; no config flag fixes it — it's the architecture); cross-shard chatter (a "distributed graph" whose queries cross partitions turns O(hops) into O(hops) RPCs with fan-out).

8 · In the interview

How it comes up: rarely "design a graph database" — usually a domain that smells of edges: fraud detection, "people you may know", permissions with nested groups, knowledge graphs. The trap is jumping to Neo4j because the domain sounds graphy. Start relational, then earn the graph: "if our deepest query is two fixed hops, this is a JOIN problem; the moment we need variable-depth traversal — fraud rings, nested groups — recursive SQL's cost grows with table size and depth, and I'd introduce a graph store for that path."

The choice script: name the workload's depth and variability first. Fixed ≤2 hops → relational. Deep/variable/pattern-shaped, data up to a few billion edges → graph DB (one strong writer + read replicas; call out the write ceiling yourself). Whole-graph analytics → offline (Spark/Pregel), never the OLTP graph. Planet-scale but constrained shape → TAO/Zanzibar story: sharded storage + graph-shaped API, "access patterns survive, traversal engines don't." Saying the boundaries is the senior signal; any of the four answers without its boundary is a junior one.

9 · Check yourself

Why is "friends of friends of friends" expensive in SQL but cheap in a graph database?

SQL reconstructs each relationship at query time: every hop is a self-JOIN whose index lookups cost ~log(edge-table size) per row, the frontier multiplies per level, and variable depth forces a recursive CTE the optimizer handles poorly. A graph DB materialized the relationships at write time — index-free adjacency — so each hop is pointer-chasing proportional to local degree only. The graph's cost tracks data touched; SQL's carries a factor of data owned. (Fair-play caveat: at 1–2 fixed hops with good indexes, SQL is entirely fine.)

Interviewer: "Our graph is huge — we'll just shard Neo4j by user ID." What goes wrong?

Edges don't respect hash boundaries: with random sharding, most edges cross shards, and each cross-shard hop becomes an RPC (~ms) instead of a pointer-chase (~100 ns). Doing better means minimizing cut edges under balance constraints — NP-hard, and heuristics still lose to power-law supernodes whose neighborhoods span every shard. That's why Neo4j scales writes vertically (single Raft leader), why JanusGraph/Neptune accept per-hop network cost for capacity, and why FAANG-scale systems constrain the query shape instead (TAO's one-hop API).

Why does Facebook — the graph company — not run the social graph on a graph database?

Because the workload at that scale is not general traversal — it's billions of hot, shallow, cacheable reads: "latest 50 likes", "does Ava follow Ben". TAO serves exactly that: objects + associations over sharded MySQL with a massive cache tier, one-hop queries only, eventual consistency; multi-hop is composed in application code. The interview lesson: at extreme scale you keep the graph access pattern and drop the general traversal engine. Graph DBs shine at mid-scale with genuinely deep, variable queries.

When is Postgres the right "graph database"?

When traversals are bounded and moderate: WITH RECURSIVE handles org charts, category trees, 2–3-hop recommendations on millions of rows perfectly well — with zero new infrastructure, real transactions joining graph and non-graph data, and your team's existing fluency. Reach for a real graph DB when depth is variable/deep, queries are pattern-shaped (cycles), latency is tight, or the traversal is the product. Offering the Postgres baseline first is a credibility move, not a concession.

A celebrity node has 50 M edges. What breaks, and what do you do about it?

Any traversal whose frontier touches the celebrity scans a 50 M-entry adjacency chain — one node multiplies a hop's cost by ~10⁵× the median; writes contend on the same chain. Mitigations: bound and filter patterns so supernodes aren't expanded (traverse to them, not through them); split the node by edge type/time bucket; vertex-centric indexes (JanusGraph) to range-scan within an edge list; precompute celebrity-adjacent results offline; and keep firehose workloads (feed fan-out, counters) out of the graph entirely.
